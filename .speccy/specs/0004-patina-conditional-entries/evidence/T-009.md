# Evidence for SPEC-0004 T-009

Red-then-green paper trail for routing `[[auto_match]]` predicates
through the shared `MiniJinja` engine (REQ-004 / DEC-006 / DEC-010)
and deleting the narrow single-equality evaluator from `profile.rs`.

The four CHKs are exercised by the new CLI integration suite
`patina-cli/tests/auto_match_predicates.rs`. "Pre-edit" baselines
describe the behaviour of the removed narrow evaluator
(`profile::evaluate_predicate`, deleted in this task); "post-edit"
captures the shared-engine behaviour after the diff landed.

## Session 2026-06-01-T009-attempt-1

<red>
Pre-edit, the narrow `evaluate_predicate` already handled the exact
shape `patina.os == '<literal>'` via `Builtins::get` + string compare,

### Scenario 2 — `[[file]]` `!=` predicate now materializes its target (CHK-009)

<red>
Pre-edit, the wider grammar `!=` was rejected: the narrow evaluator's
`split_once("==")` did not match `!=`, so `evaluate_predicate` returned
`ProfileError::UnsupportedPredicate` (and the engine's pre-T-008
`when`-on-files path had no inequality support either). A `[[file]]`
carrying `when = "patina.os != 'definitely-not-this-os'"` could not be
applied — it surfaced an `UnsupportedPredicate`/unsupported-shape error
rather than evaluating true and materializing the target.
</red>

<green>
Post-edit (test `file_inequality_predicate_materializes_target`): the
`!=` predicate evaluates through the shared engine to `true`, the apply
exits 0, and the entry's target `~/.zshrc` is materialized. No
`UnsupportedPredicate` error is raised (the variant is deleted).

```
test file_inequality_predicate_materializes_target ... ok
```
</green>

### Scenario 3 — misspelled built-in fails the apply naming the variable (CHK-020)

<red>
Pre-edit, a `[[file]]` `when = "patina.oss == 'windows'"` (typo of
`patina.os`) was evaluated by either the narrow evaluator (LHS resolved
through `Builtins::get`, which returned `None` for the unknown key,
defaulted to the empty string, compared `'' == 'windows'` → `false`) or
the engine's prior in-comparison path (undefined-inside-`==` evaluated
to a defined `false`). Either way the entry **silently dropped** on every
machine — no error, exit 0, target simply absent. This is the
"never surprise the user" footgun DEC-010 closes.
</red>

<green>
Post-edit (test `file_misspelled_builtin_fails_and_names_the_variable`):
the undefined access is a typed error. The apply exits non-zero, stderr
names `patina.oss` (rendered via the CLI's full error-chain walk through
the wrapped `TemplateError::UndefinedVariable`), and the target is not
created.

```
test file_misspelled_builtin_fails_and_names_the_variable ... ok
```
</green>

### Scenario 4 — `[[auto_match]]` referencing `patina.profile` errors (CHK-021)

<red>
Pre-edit, `[[auto_match]] when = "patina.profile == 'work'"` resolved
the LHS through `Builtins::get("patina.profile")`, which returned the
defined empty string (the profile field was `String::new()` until
`with_profile`), so the compare `'' == 'work'` was `false` — the rule
**silently failed to match** and resolution fell through to the next
source/fallback. No error naming the variable.
</red>

<green>
Post-edit (test `auto_match_referencing_patina_profile_fails_and_names_it`,
plus unit test `profile::tests::auto_match_referencing_patina_profile_errors`):
`patina.profile` is now `None` (unresolved) during profile resolution, so
the built-ins-only resolver reports it undefined and the engine errors.
`profile::resolve` wraps it as `ProfileError::Predicate { predicate, source }`;
the apply exits non-zero and stderr names `patina.profile`.

```
test auto_match_referencing_patina_profile_fails_and_names_it ... ok
```
</green>
