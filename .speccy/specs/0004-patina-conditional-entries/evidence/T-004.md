# Evidence for SPEC-0004 T-004

Paper trail for T-004: wiring the repo-shared and active-profile variable
layers into `apply::plan` (REQ-005). T-003 parsed the root `patina.toml`
into a `RootConfig { repo_shared, per_profile }`; this task pushes those
two tables into the planning `Resolver` via `Resolver::with_repo_shared`
and `Resolver::with_per_profile`, so the resolver carried in `ResolvedPlan`
(and reused by template rendering, the hook `when` evaluator, and the diff
renderer) sees them. This closes REQ-005's two end-to-end CHKs.

## Production change

`patina-core/src/apply/engine.rs::plan`:

- Parses the root manifest with `parse_root_config(&root_manifest)` (the
  T-003 function), threading `RootConfigError` into `EngineError` via a
  new `#[from]` variant (`patina-core/src/error.rs`).
- Pushes the repo-shared `[variables]` table with
  `.with_repo_shared(table_to_layer(&root_config.repo_shared))` while
  building the resolver.
- Selects the active profile's `[profiles.<name>.variables]` table by
  `root_config.per_profile.get(&profile.name)` and, when present, pushes
  it with `.with_per_profile(...)`. The no-profile fallback (empty profile
  name) matches no key, so no per-profile table is pushed.
- A small `table_to_layer` helper projects a `toml::value::Table` into the
  resolver's `Vec<(String, String)>` layer form (string-valued entries
  only). It replaces the inline `filter_map` the per-module push already
  used, so all three pushes (repo-shared, per-profile, per-module) share
  one projection.

Precedence is unchanged: it is fixed by the resolver's layer order
(`variables/mod.rs` `get`: CLI > per-machine > per-profile > per-module >
repo-shared > built-ins). This task only populates two previously-omitted
layers.

## Session 2026-06-01-T004-attempt-1

CLI integration tests in `patina-cli/tests/variable_layers.rs` drive the
real `patina apply --yes` binary over a tempdir fixture whose root
`patina.toml` declares the variable tables, then read the materialized
`.tmpl` target.

### CHK-010 — A root `[variables]` value renders into a module template

`root_variable_renders_into_module_template`: root `[variables]` declares
`editor = "nvim"`; module `shell` has `editor.tmpl` (`editor = {{ editor }}`)
→ `~/.editor`. `patina apply --yes` exits 0 and `~/.editor` contains
`editor = nvim`.

Mutation check: without the `with_repo_shared` push, `{{ editor }}` is
undefined at every layer, so the strict-undefined template engine errors
and the apply exits non-zero — the `code == 0` assertion fails red.

### CHK-011 — An active profile value shadows the repo-shared value

`active_profile_variable_shadows_repo_shared`: root declares
`[variables] editor = "nvim"` and `[profiles.work.variables] editor =
"code"`. `PATINA_PROFILE=work patina apply --yes` exits 0 and `~/.editor`
contains `editor = code`.

Mutation check: without the `with_per_profile` push, the repo-shared
`nvim` would render instead, failing the `editor = code` assertion. The
two layer values are distinct (`nvim` vs `code`) so the assertion pins the
precedence, not merely that *some* value rendered.

### Per-module beats repo-shared

`per_module_variable_beats_repo_shared`: `editor` is set in both the root
`[variables]` (`nvim`) and the module's `[variables]` (`emacs`). Apply
renders `editor = emacs`, confirming per-module still beats repo-shared
after this task adds the repo-shared layer below it.

### No-profile fallback

`no_profile_selects_no_per_profile_table`: with `[profiles.work.variables]`
present but no active profile (the fixture `env_remove`s `PATINA_PROFILE`),
the repo-shared `nvim` renders — the profile table is inert because the
empty profile name matches no `per_profile` key.
