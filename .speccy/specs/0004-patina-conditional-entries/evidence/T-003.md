# Evidence for SPEC-0004 T-003

Paper trail for T-003: net-new deserialization of the root
`patina.toml`'s repo-shared `[variables]` table and its
`[profiles.<name>.variables]` tables, returning the raw tables for T-004
to layer into the resolver (REQ-005). This task is parse-and-return only;
it performs no layering, no precedence, and no template rendering.

## Scope of this task's proof

REQ-005's two dedicated `<scenario>` CHKs — CHK-010 (a root `[variables]`
value renders into a `.tmpl` target under `patina apply --yes`) and
CHK-011 (an active profile's `[profiles.work.variables]` value shadows it)
— are **end-to-end apply scenarios**. They require the resolver-layer
wiring (`Resolver::with_repo_shared` / `with_per_profile` in `apply::plan`)
and template rendering, both of which the decomposition explicitly assigns
to **T-004**, not T-003 (TASKS.md T-004 `<task-scenarios>` name CHK-010 /
CHK-011 as its CLI integration tests). T-003 is the parse half: it makes
the raw tables available for T-004 to push. CHK-010 / CHK-011 are therefore
out of T-003's scope and are not demonstrated here; the roll call in the
journal labels them accordingly.

T-003's own `<task-scenarios>` carry no CHK ids; they are demonstrated by
the unit tests below, which drive the real `parse_root_config` /
`parse_root_config_str` production functions in
`patina-core/src/config/root.rs`.

## Session 2026-06-01-T003-attempt-1

### Scenario 1 — A root `[variables]` table is read into the repo-shared table

`reads_root_repo_shared_variables` parses `[variables]\neditor = "nvim"`
through `parse_root_config_str` and asserts the returned
`RootConfig.repo_shared` table maps `editor` → `"nvim"` and that
`per_profile` is empty. `reads_variables_from_an_on_disk_manifest` proves
the same through the on-disk `parse_root_config` path against a written
tempdir manifest.

Mutation check: a parser that dropped the `[variables]` table (returned it
empty) would fail the `Some("nvim")` assertion.

### Scenario 2 — A `[profiles.work.variables]` table is read and keyed by `work`

`reads_per_profile_variables_keyed_by_name` parses
`[profiles.work.variables]\neditor = "code"` and asserts
`per_profile["work"]` maps `editor` → `"code"`.
`reads_both_repo_shared_and_per_profile_together` proves the two tables are
read independently in one manifest (repo-shared `nvim`, profile `work` →
`code`). `a_profile_without_a_variables_table_yields_an_empty_table` proves
a bare `[profiles.work]` still registers `work` with an empty table.

Mutation check: keying the profile table under the wrong name, or folding
it into the repo-shared table, would fail the `per_profile["work"]` lookup.

### Scenario 3 — A reserved `patina.*` key in either table is rejected

`rejects_reserved_key_in_repo_shared_table` and
`rejects_reserved_key_in_per_profile_table` each declare a `"patina.os"`
key (in `[variables]` and `[profiles.work.variables]` respectively) and
assert the parse returns `RootConfigError::Variable(VariableError::ReservedKey { key })`
with `key == "patina.os"` — reusing `variables::reject_reserved_keys`, the
same helper the module parser uses, so the error is identical to the
existing reserved-key rejection.

Mutation check: skipping the `reject_reserved_keys` call on either table
would let the reserved key through and fail the `expect_err`.

### Scenario 4 — Absent sections yield empty results, not an error

`absent_sections_yield_empty_results` parses a manifest with only a
`[patina]` table and asserts both `repo_shared` and `per_profile` are
empty with no error. `missing_manifest_file_yields_empty_results` proves
the `NotFound` fall-through on `parse_root_config` against a never-created
tempdir path (mirroring `load_auto_match_rules`).

Mutation check: treating a missing `[variables]` table as an error (e.g.
`unwrap`-ing a required field) would fail the `expect("...parses")`;
removing the `NotFound` arm would surface an IO error instead of an empty
config.

### Supplementary — Malformed TOML is a typed error naming the path

`malformed_toml_is_a_typed_error_naming_the_path` writes an unclosed
`[variables` header and asserts `parse_root_config` returns
`RootConfigError::Toml` whose `path` field is the real on-disk path (not
the `<memory>` placeholder), confirming the path re-mapping in
`parse_root_config`.
