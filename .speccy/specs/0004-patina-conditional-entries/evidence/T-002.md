# Evidence for SPEC-0004 T-002

Red-then-green paper trail for T-002: emit both the `[[file]]` and
`[[directory]]` table-arrays in one deterministic files-then-directories
order with a single monotonic `u32` entry-index space (REQ-009 ordering /
index `<done-when>` bullets, DEC-009).

T-002 satisfies the two REQ-009 `<done-when>` ordering/index bullets:

- "The plan emits all `[[file]]` operations before all `[[directory]]`
  operations, each block in declaration order."
- "Entry indices form a single monotonic space across both tables (all
  `[[file]]`, then all `[[directory]]`); no `[[file]]` and `[[directory]]`
  entry share an index."

REQ-009's only dedicated `<scenario>` CHK is CHK-022 (a `when`-false
wrong-OS directory source raising no plan error). That CHK depends on the
`when` gate (T-005) and the plan-time source kind/existence check (T-006),
neither of which exists yet, so CHK-022 is out of T-002's scope and is not
demonstrated here. The ordering/index bullets above carry no CHK id of
their own; they are demonstrated by the scenarios below.

## Session 2026-06-01-T002-attempt-1

### Scenario 1 — Files are planned before directories, each in declaration order (REQ-009 done-when bullet 3)

<red>
Pre-edit: the plan loop iterated only `config.files` (engine.rs:268, the
T-001 state) and never emitted any `[[directory]]` operation. A
declaration-order assertion over a mixed file/directory manifest could not
be written against the pre-edit code because directory entries produced no
`PlannedOperation` at all — the `directories` bucket was parsed-and-carried
(T-001) but dropped on the floor at plan time. There was no
`assemble_plan_operations` seam to assert against; the ordering property
did not exist to test.
</red>

<green>
Post-edit: `assemble_plan_operations` chains `file_entries` then
`directory_entries` and the unit test
`files_are_planned_before_directories_in_declaration_order`
(`patina-core/src/apply/engine.rs`) asserts the emitted operation sources
are exactly `["/repo/f0", "/repo/f1", "/repo/d0"]` for two `[[file]]`
entries (`f0`, `f1`) and one `[[directory]]` entry (`d0`).

```
$ cargo test -p patina-core --lib apply::engine
test apply::engine::tests::files_are_planned_before_directories_in_declaration_order ... ok
test apply::engine::tests::entry_indices_are_a_single_monotonic_space_across_both_tables ... ok
test result: ok. 5 passed; 0 failed; 0 ignored; 0 measured; 238 filtered out
```

Mutation check: reversing the chain to `directory_entries.chain(file_entries)`
would order the sources `["/repo/d0", "/repo/f0", "/repo/f1"]` and fail the
`assert_eq!` — the test catches the obvious wrong implementation.
</green>

### Scenario 2 — Entry indices form a single monotonic space across both tables (REQ-009 done-when bullet 4)

<red>
Pre-edit: there was no plan-time entry index at all. `execute` re-derived
an `entry_index` from `enumerate()` over `resolved.operations`
(engine.rs:425) — a per-operation position, not a per-declared-entry index,
and computed only at execute time. With directories never planned, a
`[[file]]`/`[[directory]]` index-collision property was unobservable: no
directory entry ever reached the index space.
</red>

<green>
Post-edit: each managed entry is assigned a monotonic `u32` `entry_index`
over the files-then-directories sequence, carried on `ResolvedOperation`,
and `execute` records that planned index instead of re-deriving one. The
unit test `entry_indices_are_a_single_monotonic_space_across_both_tables`
asserts, for two `[[file]]` entries (the first fanning out to two targets)
and two `[[directory]]` entries, that the resolved-op indices are exactly
`[0, 1, 2, 3]`, that the file entries own `[0, 1]` and the directory
entries own `[2, 3]`, and that the two ranges are disjoint.

The integration test
`directory_and_file_entries_get_distinct_indices_and_envelope_major_is_unchanged`
(`patina-cli/tests/commit_record.rs`) drives a real
`patina apply --yes` over a tempdir repo with two copy-mode `[[file]]`
entries and one copy-mode `[[directory]]` entry (two leaves), then decodes
the `<ts>.COMMIT` record and asserts: the two file entries have distinct
indices, the two directory leaves share one index, no file/directory entry
collides, every file index precedes the directory index, and the COMMIT
version-envelope major is still `2` (unchanged by this SPEC — DEC-009).

```
$ cargo test -p patina-cli --test commit_record
test directory_and_file_entries_get_distinct_indices_and_envelope_major_is_unchanged ... ok
test result: ok. 6 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

Mutation check: dropping the `entry_index = entry_index.saturating_add(1)`
increment would give every entry index `0`, failing both the
`vec![0, 1, 2, 3]` assertion (unit) and the `assert_ne!(a, b)` distinct-file
assertion (integration).
</green>

### Scenario 3 — Committed apply over both tables; COMMIT decode is collision-free and the envelope major is unchanged (REQ-009 done-when bullet 4, DEC-009 version-envelope)

<red>
Pre-edit: a `[[directory]]` entry produced no operation and so no
`ExpectedTarget` in the COMMIT record at all; a committed-apply assertion
spanning both tables had nothing to decode for the directory side.
</red>

<green>
Post-edit: covered by the same integration test in Scenario 2. The decoded
`ApplyRecord.targets` carries one `ExpectedTarget` per materialized object
(two file targets, two directory leaves), each `entry` index unique per
declared entry, and the first two bytes of the `<ts>.COMMIT` file decode to
major version `2` — the `entry: u32` wire format is unchanged, no version
bump. See the Scenario 2 green block for the run.
</green>
