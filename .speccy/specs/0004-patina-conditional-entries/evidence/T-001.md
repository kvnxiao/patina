# Evidence for SPEC-0004 T-001

Red-then-green paper trail for T-001: the kind-typed `[[file]]` /
`[[directory]]` schema split with collapsed mode names and a parsed-but-
not-yet-evaluated `when` field (REQ-001).

The behaviour under test is at the parse layer
(`patina_core::config::parse_module_config_str`); the red/green pairs are
captured by running the targeted integration test binary
`patina-core/tests/config_file_entry.rs` at the pre-edit and post-edit
states. The "red" runs were obtained by checking out the file's pre-edit
content (the single `[[file]]`-only `FileEntry` taxonomy at `HEAD`).

## Coverage rule

Each `demonstrated` CHK in the journal roll call maps to exactly one
`### Scenario N` block below. CHK-001, CHK-002, and CHK-003 are
demonstrated here.

## Session 2026-06-01-T001-attempt-1

### Scenario 1 — A `[[file]]` with no `mode` resolves to file kind + symlink (CHK-001)

<red>
Pre-edit, the resolved entry type (`FileEntry`) carried only
`source` / `targets` / `mode`; there was no `kind` field, so an
assertion on `entry.kind == EntryKind::File` did not compile (the
`EntryKind` type and the `kind` field did not exist). The pre-edit
schema could not express the file-vs-directory kind distinction at all —
every entry was implicitly a `[[file]]`.

```
$ # pre-edit file_entry.rs: `pub struct FileEntry { source, targets, mode }`
$ # no EntryKind, no `kind` field — `entry.kind` is a compile error.
error[E0609]: no field `kind` on type `&ManagedEntry`
```
</red>

<green>
Post-edit, a `[[file]]` with `source = "zshrc"`, `target = "~/.zshrc"`
and no `mode` resolves to `EntryKind::File` + `FileMode::Symlink`.

```
$ cargo test -p patina-core --test config_file_entry file_with_omitted_mode_resolves_to_file_kind_symlink
running 1 test
test file_with_omitted_mode_resolves_to_file_kind_symlink ... ok
test result: ok. 1 passed; 0 failed
```

The test asserts `config.files[0].kind == EntryKind::File` and
`config.files[0].mode == FileMode::Symlink`.
</green>

### Scenario 2 — A `[[directory]]` `symlink-tree` resolves to directory kind + per-leaf symlink (CHK-002)

<red>
Pre-edit, there was no `[[directory]]` table-array and no
`symlink-tree` mode. A `[[directory]]` table was silently ignored by the
`RawModule` deserializer (it had no `directory` field), and
`mode = "symlink-tree"` on the only-existing `[[file]]` table was rejected
as an unsupported mode. There was no `config.directories` field to assert
against.

```
$ # pre-edit: RawModule has only `file`, `hook`, `variables` — a
$ # [[directory]] table deserializes to nothing; ModuleConfig has no
$ # `directories` field. `config.directories` is a compile error.
error[E0609]: no field `directories` on type `ModuleConfig`
```
</red>

<green>
Post-edit, a `[[directory]]` with `source = "mpv"`,
`target = "~/.config/mpv"`, `mode = "symlink-tree"` resolves to
`EntryKind::Directory` + `FileMode::SymlinkTree`, surfaced on
`config.directories`.

```
$ cargo test -p patina-core --test config_file_entry directory_symlink_tree_resolves_to_directory_kind_per_leaf_symlink
running 1 test
test directory_symlink_tree_resolves_to_directory_kind_per_leaf_symlink ... ok
test result: ok. 1 passed; 0 failed
```

The test asserts `config.files.is_empty()`, `config.directories.len() == 1`,
`config.directories[0].kind == EntryKind::Directory`, and
`config.directories[0].mode == FileMode::SymlinkTree`.
</green>

### Scenario 3 — A `[[file]]` with `mode = "symlink-tree"` is rejected naming the accepted `[[file]]` modes (CHK-003)

<red>
Pre-edit, the single `[[file]]` `UnsupportedMode` error listed the old
allowlist `symlink`, `symlink-dir`, `copy`, `copy-tree`. A `[[file]]`
declaring `mode = "symlink-tree"` was rejected, but the rendered message
named neither `symlink-tree` as the offending value framed against the
collapsed `[[file]]` allowlist, nor the collapsed two-mode set — it
recited the four old mode names including the now-removed
`symlink-dir` / `copy-tree`.

```
$ # pre-edit UnsupportedMode message:
"[[file]] entry declares unsupported mode `symlink-tree`; the accepted
 values are `symlink`, `symlink-dir`, `copy`, `copy-tree`"
$ # contains the removed `symlink-dir` / `copy-tree` strings; does not
$ # frame the collapsed [[file]] two-mode set.
```
</red>

<green>
Post-edit, the `[[file]]`-specific `UnsupportedFileMode` error names the
offending `symlink-tree` value and the collapsed accepted `[[file]]`
modes `symlink`, `copy`, and never the removed spellings.

```
$ cargo test -p patina-core --test config_file_entry file_with_symlink_tree_mode_is_rejected_naming_accepted_file_modes
running 1 test
test file_with_symlink_tree_mode_is_rejected_naming_accepted_file_modes ... ok
test result: ok. 1 passed; 0 failed
```

The test asserts the rendered `Display` contains `symlink-tree`,
`symlink`, and `copy`, and matches
`FileEntryError::UnsupportedFileMode`. The sibling test
`rejects_unknown_file_mode_naming_accepted_values` additionally asserts the
removed `symlink-dir` / `copy-tree` strings never appear in a `[[file]]`
mode error.
</green>
