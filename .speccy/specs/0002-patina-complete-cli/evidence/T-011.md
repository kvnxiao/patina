# Evidence for SPEC-0002 T-011

Paper trail for T-011: the `patina doctor --fix` interactive remediation
path (REQ-006) and its exclusive-lock acquisition (REQ-009). The work
extends `patina-cli/src/cmd/doctor.rs` with a `run_fix` branch (the
read-only `run_report` branch is unchanged from T-010), threads `Tty` +
`PromptReader` through the `doctor` dispatch in `main.rs`, and adds a
`run_in` working-directory helper to the shared test fixture.

## Design: branch on `--fix`, reuse the existing finding computation

`run` now splits into `run_report` (the unchanged T-010 read-only path,
shared lock + escape hatch) and `run_fix`. `run_fix` reuses the pure
`compute_findings(&Inputs)` to decide what to remediate, so the fixable /
non-fixable partition is driven by the same finding set the read-only path
reports — no parallel finding logic. Per-finding consent flows through an
`accept` helper that mirrors the established `confirm` shape in
`remove.rs` / `promote.rs` (`--yes` accepts unconditionally; a TTY prompts;
a non-TTY without `--yes` declines). The Windows elevation remediation
reuses `patina_core::launch_elevate_helper` (T-009) behind a `#[cfg(windows)]`
gate, with a non-Windows stub for call-site parity exactly as `apply.rs`
does for its dev-mode gate.

## Coverage note: CHK-011 on a non-Windows CI

CHK-011 is a full run on a Windows host with Developer Mode OFF that drives
the UAC helper and asserts the registry value reads 1 afterward. The
macOS/Linux CI has no Windows host, no Developer Mode registry, and no way
to click a UAC dialog, so the end-to-end form ships as `#[cfg(windows)]`
`#[ignore]` (`windows_fix_enables_dev_mode_and_exits_zero` in
`patina-cli/tests/doctor_cli.rs`). The remediation *wiring* CHK-011 depends
on — `--fix` accepting the prompt and routing `DOC-WIN-DEVMODE` to the
elevation flow — is exercised on the CI host structurally: `fix_dev_mode`
shares the `accept` decision path that the unit tests
(`accept_tty_reads_the_answer`, `accept_yes_proceeds_without_reading`)
cover, and the `#[cfg(windows)]` body compiles in CI under
`cargo clippy --all-targets`.

## Session 2026-05-31-T011-attempt-1

### Scenario 1 — `--fix --yes` writes the default_repo pointer from the CWD (REQ-006, DOC-NO-DEFAULT-REPO)

Integration test `fix_yes_writes_default_repo_from_cwd_and_exits_zero` in
`patina-cli/tests/doctor_cli.rs`, driving the real binary via the new
`Fixture::run_in` (subprocess CWD = the fixture repo root).

<red>
Before this task there was no `--fix` path: `run` ignored `args.fix`
entirely and always took the read-only branch, so `doctor --fix --yes`
reported the `DOC-NO-DEFAULT-REPO` info finding and exited 0 WITHOUT ever
writing the `default_repo` pointer. The assertion
`fs_err::read_to_string(state_root/default_repo)` would fail with a
not-found error: the pointer file never existed.
</red>

<green>
After the task, `doctor --fix --yes` acquires the exclusive lock, sees the
`DOC-NO-DEFAULT-REPO` finding, auto-accepts (via `--yes`), canonicalizes the
CWD, and writes it through `write_persisted_default`. The test reads the
pointer and asserts its trimmed contents equal `canonicalize_path(&fx.root)`,
and that the process exits 0.

```
running 1 test
test fix_yes_writes_default_repo_from_cwd_and_exits_zero ... ok
```
</green>

### Scenario 2 — non-TTY `--fix` without `--yes` exits 1 naming `--yes` (REQ-006)

Integration test `fix_without_yes_in_non_tty_exits_one_naming_yes`. The
subprocess stdin is not a TTY, so this is the genuine non-interactive path.

<red>
With no `--fix` handling, `doctor --fix` (no `--yes`) took the read-only
branch and exited 0; the assertion `code == 1` and the stderr-names-`--yes`
assertion would both fail (exit 0, no `--yes` mention).
</red>

<green>
`run_fix` now refuses up front when `!yes && tty == NonInteractive`: it
emits a stderr warning naming `--yes` and returns `ExitCode::Generic` (1)
BEFORE acquiring the lock or mutating anything. The test asserts exit 1, the
`--yes` substring on stderr, and that the `default_repo` pointer was NOT
written (proving the early return precedes any mutation).

```
running 1 test
test fix_without_yes_in_non_tty_exits_one_naming_yes ... ok
```
</green>
