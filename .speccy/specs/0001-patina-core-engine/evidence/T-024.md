# Evidence for SPEC-0001 T-024

Paper trail for the docs-structure contract (REQ-027): `docs/ARCHITECTURE.md`
and `docs/USER_GUIDE.md` ship with a fixed set of `##`-level headings, and the
`## State directory` section of the user guide lists the six cloud-sync
providers the per-machine state directory must not live on. A new integration
test (`patina-cli/tests/docs_structure.rs`) parses both files as CommonMark via
`pulldown-cmark` and gates structural presence only — heading existence by
exact text and bullet membership in a named section — never substring-matching
the surrounding prose.

## Coverage rule

REQ-027 attaches three `<scenario>` blocks (CHK-056, CHK-057, CHK-058), each
demonstrated by one test below. The `<behavior>` bullets calling for a renamed
heading and a prefix-extended bullet to fail are demonstrated in the red halves
of Scenarios 2 and 3 (the mutation was applied to a working copy, the failure
observed, then reverted; no committed doc was left mutated).

## Session 2026-05-29-T024-attempt-1

### Scenario 1 — ARCHITECTURE.md carries its four H2 anchors (CHK-056)

<red>
Before this task, `docs/ARCHITECTURE.md` did not exist, so no executable gate
asserted its structural anchors. A downstream SPEC cross-referencing
`## Journal format` or `## Apply phases` had only the file's prose to rely on.
</red>

<green>
`patina-cli/tests/docs_structure.rs::architecture_has_required_h2_headings`
parses the file, extracts the set of H2 heading texts, and asserts each of
`Engine layers`, `Journal format`, `Apply phases`, `Recovery` is present by
exact text. Passes.
</green>

### Scenario 2 — USER_GUIDE.md carries its six H2 anchors (CHK-057)

<red>
With `## State directory` renamed to `## State dir` in a working copy of
`docs/USER_GUIDE.md`, the test fails naming the missing literal:

```
test user_guide_has_required_h2_headings ... FAILED
docs/USER_GUIDE.md missing required `## State directory` heading;
  found {"Apply flow", "Declaring dotfiles", "Installation", "Recovery",
         "State dir", "Troubleshooting"}
```

This confirms the test gates exact heading text, not a fuzzy match — the rename
is detected and named. The mutation was reverted afterwards.
</red>

<green>
`patina-cli/tests/docs_structure.rs::user_guide_has_required_h2_headings`
asserts each of `Installation`, `Declaring dotfiles`, `Apply flow`,
`State directory`, `Recovery`, `Troubleshooting` is present by exact text
against the unmutated file. Passes.
</green>

### Scenario 3 — State directory section lists the six providers (CHK-058)

<red>
With the `- Dropbox` bullet replaced by `- Dropbox (via Smart Sync)` in a
working copy, the test fails naming the missing literal:

```
test user_guide_state_directory_lists_cloud_sync_providers ... FAILED
docs/USER_GUIDE.md `## State directory` section missing bullet `Dropbox`;
  found {"Box", "Dropbox (via Smart Sync)", "Google Drive", "OneDrive",
         "Syncthing", "iCloud Drive"}
```

This confirms membership is by exact text, not prefix match: the
prefix-preserving `Dropbox (via Smart Sync)` does NOT satisfy the `Dropbox`
requirement (the SPEC's REQ-027 behavior bullet). The mutation was reverted
afterwards.
</red>

<green>
`patina-cli/tests/docs_structure.rs::user_guide_state_directory_lists_cloud_sync_providers`
extracts the top-level bullet items inside the `## State directory` section and
asserts each of `iCloud Drive`, `OneDrive`, `Dropbox`, `Box`, `Google Drive`,
`Syncthing` is present as a literal entry. Items outside the section are not
collected (the collector only gathers `<li>` events while the active H2 is
`State directory`), so a matching bullet under a different heading would not
satisfy the assertion. Passes.
</green>
