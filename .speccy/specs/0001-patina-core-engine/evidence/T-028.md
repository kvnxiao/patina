# Evidence for SPEC-0001 T-028

Red-then-green paper trail for the acquire-then-recover reorder: the
engine apply path (`patina_core::execute` in
`patina-core/src/apply/engine.rs`) now resolves the exclusive lock per
`LockPolicy` *before* calling `recover_orphans`, for all three policy
variants. This closes the REQ-030 zero-mutation hole where a contended
`NonBlocking` attempt would run orphan recovery (reversing backups,
deleting an orphan's plan/progress) before returning the contention
error.

The one `demonstrated` scenario drives the engine `execute` entry
point in-process (the only layer that can pass a test-controlled
`LockGuard` and plant a pending orphan while holding the lock); it
lives in the `mod tests` module in `patina-core/src/apply/engine.rs`.

## Session 2026-05-30-T028-attempt-1

### Scenario 1 — NonBlocking contention leaves a pending orphan untouched (CHK-068)

<red>
Before this task, `execute` called `recover_orphans(&journal_dir,
&backups_dir)?` at the top of the function, *before* resolving the
lock. With a test-controlled exclusive guard held AND a pending orphan
`<ts>.plan` (+ `<ts>.progress`) planted in the journal, a `NonBlocking`
apply would run recovery first — reversing/deleting the orphan — and
only then hit `try_acquire_lock` and return the contention error. A
test asserting that the orphan bytes are byte-identical after a
contended attempt fails against the pre-reorder code: recovery has
already removed the orphan plan/progress.
</red>

<green>
Post-edit, the lock is resolved before `recover_orphans`. With the
guard held and an orphan planted, the test
`non_blocking_contention_leaves_pending_orphan_untouched` drives
`execute(&resolved, &ApplyRequest::default(), LockPolicy::NonBlocking)`
and asserts:

```
running 1 test
test apply::engine::tests::non_blocking_contention_leaves_pending_orphan_untouched ... ok
```

The call returns
`Err(EngineError::Lock(LockError::Contended { kind: LockKind::Exclusive, .. }))`;
the planted orphan `<ts>.plan` and `<ts>.progress` still exist and the
plan's bytes are unchanged (recovery never ran); and no new
`<ts>.plan`, `<ts>.COMMIT`, or backups directory was written by the
contended attempt.
</green>
