# Evidence — SPEC-0001 / T-017

`patina status` classifies every managed target as CLEAN / DRIFTED /
MISSING / ORPHANED against the last committed apply (REQ-018). This task
lands:

- A persisted apply record. The `<ts>.COMMIT` sentinel was an empty marker
  (its presence is all crash recovery reads). The plan file is deleted at
  commit, so `status` had nothing to read. T-017 makes the sentinel carry
  a `postcard`-encoded `ApplyRecord` behind the same version envelope the
  plan uses: `last_apply` metadata (`at` RFC 3339, `user`, `host`) plus one
  `ExpectedTarget` per materialized object (a symlink records its canonical
  link target; a copy/render records a 64-bit `std::hash` fingerprint of
  the written bytes). Recovery is unaffected — it keys on the sentinel's
  *existence* and never decodes the body. (`patina-core/src/journal/record.rs`,
  `journal/mod.rs` `commit` + `read_latest_commit`, engine `build_apply_record`.)
- The `patina_core::status` module: reads the latest commit record, takes
  the **shared** lock with the warn-and-proceed-after-`SHARED_TIMEOUT`
  escape hatch (REQ-023), recomputes the current declared-target set, and
  classifies each recorded target (`status/mod.rs`, `status/classify.rs`).
- The `patina status [--json]` CLI surface routed through the `Reporter`
  (`patina-cli/src/cli.rs`, `cmd/status.rs`, `main.rs`).

Pre-edit baseline for every scenario: `patina status` was an unparseable
subcommand and `patina_core::status` returned
`EngineError::NotImplemented("status")`, so no classification existed.

## Scenario 1 — three clean operations, clean = 3 (CHK-031)

<red>
Pre-edit: no `status` subcommand and no `ApplyRecord` in the commit
sentinel; there was no `clean`/`drifted`/`missing`/`orphaned` counter and
no `last_apply` to emit. `patina status --json` failed at argument parsing.
</red>

<green>
Post-edit: a module with three `copy` `[[file]]` entries applied, then
`patina status --json` with no intervening change reports
`clean = 3, drifted = 0, missing = 0, orphaned = 0` and a `last_apply.at`
that is an RFC 3339 timestamp (`2026-…T…Z`). Reproduced as
`three_clean_operations_report_clean_counter_three` in
`patina-cli/tests/status_cli.rs`, which asserts each counter and the
`last_apply.at` shape.
</green>

## Scenario 2 — edited copy target reports drifted (CHK-032)

<red>
Pre-edit: no content-fingerprint comparison existed; an externally edited
target could not be detected.
</red>

<green>
Post-edit: a `copy` target `~/.gitconfig` applied, then bytes appended to
the materialized file. `patina status --json` reports `drifted = 1` and the
`files` entry for `…/.gitconfig` has `state = "drifted"` (the recorded
fingerprint no longer matches the live bytes). Reproduced as
`edited_copy_target_reports_drifted`.
</green>

## Scenario 3 — multi-target entry, one entry per target (CHK-048)

<red>
Pre-edit: no per-target reporting; the multi-target `targets = [...]`
fan-out had no status surface.
</red>

<green>
Post-edit: a `[[file]]` with
`targets = ["~/.claude/agent.toml", "~/.codex/agent.toml"]` (`copy`)
applied, then only the codex target overwritten. `patina status --json`
emits exactly two `files` entries — `…/.claude/agent.toml` `clean` and
`…/.codex/agent.toml` `drifted` — with `clean >= 1` and `drifted >= 1`.
Reproduced as `multi_target_entry_reports_one_entry_per_target`.
</green>

## Supporting scenarios (REQ-018 `<done-when>` MISSING / ORPHANED)

- `deleted_target_reports_missing`: a materialized `copy` target deleted on
  disk → `missing = 1`, entry `state = "missing"`.
- `removed_entry_with_surviving_target_reports_orphaned`: a target applied,
  then its `[[file]]` module removed from the repo while the file survives
  → `orphaned = 1`, entry `state = "orphaned"`. This exercises the
  current-plan recomputation that distinguishes a still-managed target from
  a dropped one.
- `symlink_target_reports_clean_then_drifts_when_replaced`: a `symlink`
  target is `clean` while it points at the recorded source and `drifted`
  once replaced by a regular file. This exercises the parent-only
  `manage_key` (a full canonicalization would follow the live symlink to
  the repo source and falsely report ORPHANED) and the verbatim-prefix
  normalization for the Windows `\\?\` link-target form.

## CHK roll call

- CHK-031 (three clean ops → clean = 3): demonstrated → Scenario 1 red/green.
- CHK-032 (edited copy → drifted = 1): demonstrated → Scenario 2 red/green.
- CHK-048 (multi-target → per-target entries): demonstrated → Scenario 3
  red/green.
