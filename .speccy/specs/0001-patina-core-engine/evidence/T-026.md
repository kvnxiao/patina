  shared by the plan and COMMIT envelopes.
- Write side captures `record.source` and `content_hash(&bytes)` for content
  targets (`patina-core/src/apply/engine.rs`).
- Read side compares a freshly computed `blake3` against the recorded `hash`
  (`patina-core/src/status/classify.rs`).
- `blake3` added to `patina-core/Cargo.toml` (workspace dep); `cargo deny
  check` passes without an allowlist edit.

Pre-edit baseline for every scenario below: the COMMIT record encoded a
`u64` `std::hash` fingerprint with no `source`, and the version envelope
major was `1`. There was no per-target source provenance and no `blake3`
hash to read.

The red-then-green pairs are exercised by the new integration test
`patina-cli/tests/commit_record.rs`, which drives `patina apply --yes` as a
subprocess against an isolated tempdir repo + state dir and decodes the
resulting COMMIT record via `patina_core::read_latest_commit`.

## Scenario 1 â€” copy target records canonical source + blake3 hash (CHK-062)

<red>
Pre-edit: `ExpectedTarget::Content` had no `source` field, and its hash was
the `u64` `fingerprint_bytes` (`std::hash`) digest. A test asserting the
recorded entry carries the canonical source path and a 32-byte `blake3` of
the source bytes could not compile against the old variant, and the hash
value was not `blake3`.
</red>

<green>
Post-edit: a `git` module declaring `[[file]] source = "gitconfig" target =
"~/.gitconfig" mode = "copy"` with `<repo>/git/gitconfig` = `[user]\n\tname
= Ada\n`, applied with `patina apply --yes`. The decoded COMMIT record's
entry for `~/.gitconfig` has `target` ending `/.gitconfig`, `source()` equal
to the canonical absolute path of `<repo>/git/gitconfig`, and `hash` equal
to `content_hash(contents)` (= `blake3` of the source bytes). Reproduced as
`copy_target_records_canonical_source_and_blake3_hash`.
</green>

## Scenario 2 â€” two unchanged applies record byte-identical hash (CHK-063)

<red>
Pre-edit: the recorded value was a `u64` fingerprint, not a `blake3` hash;
there was no `[u8; 32]` field to compare across runs.
</red>

<green>
Post-edit: a single content-mode `[[file]]` entry applied twice against the
unchanged source. The `hash` extracted from the first COMMIT record and from
the second are byte-identical (`first == second`), confirming the hash is a
stable function of the bytes. Reproduced as
`two_applies_record_byte_identical_hash`.
</green>

## Scenario 3 â€” COMMIT envelope major is 2 (CHK-064)

<red>
Pre-edit: `FILE_MAJOR_VERSION` was `1`, so the COMMIT file's first two bytes

failed.
</red>

<green>
Post-edit: after `patina apply --yes`, the single `<ts>.COMMIT` file's first
two bytes decode as the little-endian `u16` value `2`. Reproduced as
`commit_envelope_major_is_two`.
</green>
