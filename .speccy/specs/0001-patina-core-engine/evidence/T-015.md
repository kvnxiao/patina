# Evidence ‚Äî SPEC-0001 / T-015

Hook execution with `must_succeed` semantics and the `--force-deploy`
override (REQ-006). T-015 owns the **execution** side of the `[[hook]]`
schema: T-004 already parses each `[[hook]]` into a `HookEntry` (storing
`shell` and `when` verbatim, explicitly deferring the shell-on-PATH check
and predicate evaluation to T-015). This task lands
`patina_core::apply::hooks` with the three primitives the apply pipeline
(T-016 / T-018) consumes:

- `resolve_shells` ‚Äî resolve every hook's shell up front, defaulting an
  omitted `shell` to `bash` (macOS / Linux) or `pwsh` (Windows) and
  rejecting an explicit shell that does not resolve on `PATH` *before any
  hook runs*.
- `should_run` ‚Äî filter a hook by its optional `when` predicate through
  the shared T-008 `MiniJinja` engine.
- `run_hook` ‚Äî spawn the resolved shell against the hook command, await
  the exit status, and classify it into `HookOutcome::{Succeeded, Warned,
  Failed}` under the hook's `must_succeed` flag and the invocation-wide
  `ForceDeploy` override.

## Scope and the `patina apply --yes` phrasing

The CHK-013 / CHK-014 scenarios under REQ-006 are *parse-side* and were
demonstrated by T-004 (`patina-core/tests/config_hook_entry.rs`); they are
`hygiene` for T-015 (they run unchanged in the workspace suite). The
`<task-scenarios>` prose for T-015 names `patina apply --yes` and the CLI
exit codes 2 / 3 / 0 / 1 ‚Äî that literal surface (the `apply` subcommand,
the plan loop, the diff-and-prompt gate, the journal/backup wiring, and
the rollback machinery) does not exist yet; it lands with T-016 and the
T-018 orchestrator. T-015's unit is the classification and resolution
primitives those tasks call. The tests drive the public
`patina_core::apply::hooks` surface directly against the real host shell
and tempdir fixtures.

New files:

- `patina-core/src/apply/hooks.rs` ‚Äî `ForceDeploy`, `HookError`,
  `ResolvedHook`, `HookOutcome`, and the `resolve_shells` / `should_run` /

- `patina-core/tests/hooks_must_succeed.rs` ‚Äî `must_succeed` classification,
  `when` filtering, and unresolved-shell coverage.
- `patina-core/tests/hooks_force_deploy.rs` ‚Äî the `--force-deploy`

## Session 2026-05-28-T015-attempt-1

The "red" baseline for each behaviour is the state at HEAD before this
task: `patina_core::apply::hooks` did not exist, so none of the
classification or resolution behaviours below were callable at all. The
"green" state is the new primitive returning the REQ-006 outcome. Each
scenario is reproduced by a named integration test.

### Scenario 1 ‚Äî `pre_apply` failure under `must_succeed` classifies `Failed` (REQ-006 abort path)

<red>
Before T-015 there was no `run_hook`; a `pre_apply` hook's non-zero exit
had no classification path. The REQ-006 abort semantics (CLI exit 2, no
file op) had no engine-side trigger.
</red>

<green>
`pre_apply_failure_with_must_succeed_classifies_failed` builds a
`pre_apply` hook (`command = "exit 1"`, default `must_succeed = true`) on
the host default shell, resolves it, and runs it under `ForceDeploy::No`.
The outcome is `HookOutcome::Failed`, and the originating entry still
carries `command = "exit 1"` and `event = PreApply` so the orchestrator
can name the failing command on stderr and select the abort branch.

```
$ cargo test -p patina-core --test hooks_must_succeed \
    pre_apply_failure_with_must_succeed_classifies_failed
test pre_apply_failure_with_must_succeed_classifies_failed ... ok
```
</green>

### Scenario 3 ‚Äî `--force-deploy` downgrades a `must_succeed` failure to a warning, hook still runs (REQ-006 override)

<red>
No `ForceDeploy` override and no `run_hook`; there was no way to express
"run the hook, ignore its failure" and no proof the hook still executed.
</red>

<green>
`force_deploy_downgrades_post_apply_failure_to_warning` stages a
`post_apply` hook whose command creates a marker file in a tempdir and
then `exit 1`s, with `must_succeed = true`. Run under `ForceDeploy::Yes`,
the outcome is `HookOutcome::Warned` (no rollback would fire) and the
marker file exists on disk ‚Äî proving the hook genuinely executed rather
than being skipped. The contrast test
`same_hook_without_force_deploy_classifies_failed` runs the identical
fixture under `ForceDeploy::No` and gets `HookOutcome::Failed`, isolating
the override as the only difference.

```
test force_deploy_downgrades_post_apply_failure_to_warning ... ok
test same_hook_without_force_deploy_classifies_failed ... ok
```
</green>

### Scenario 4 ‚Äî a `when` predicate filters a non-matching host out before execution (REQ-006 / `<behavior>`)

<red>
T-004 stored `when` verbatim and deferred evaluation; no filter existed,
so every hook would have run regardless of its predicate.
</red>

<green>
`when_predicate_filters_out_non_matching_host` gates a hook on an OS the
test host is not (`patina.os == '<the-other-os>'`); `should_run` returns
`false`, so the orchestrator drops the hook before execution.
`when_predicate_runs_on_matching_env_var` gates a hook on a live
`patina.env.*` lookup matching the process environment (the
deterministic, `unsafe`-free stand-in for the `CI=true` scenario) and
`should_run` returns `true`.

```
test when_predicate_filters_out_non_matching_host ... ok
test when_predicate_runs_on_matching_env_var ... ok
```
</green>

### Scenario 5 ‚Äî an unresolved explicit shell errors before any hook runs (REQ-006 shell resolution)

<red>
T-004 stored `shell` verbatim with no PATH check; an unresolved shell
would have surfaced only as an opaque spawn failure at execution time,
after file operations had already run.
</red>

<green>
`unresolved_explicit_shell_errors_before_any_hook_runs` declares
`shell = "nonexistent-shell-xyz"` and calls `resolve_shells`. It returns
`HookError::ShellNotFound { shell: "nonexistent-shell-xyz" }` ‚Äî surfaced
during resolution, before any `run_hook` spawn or file operation. The
unit-test `unresolved_shell_aborts_whole_set_before_any_resolution`
further proves a bad shell anywhere in the set fails the whole resolve
rather than yielding a partial runnable set.

```
test unresolved_explicit_shell_errors_before_any_hook_runs ... ok
```
</green>
