# Evidence for SPEC-0001 T-021

Paper trail for deterministic `patina apply` stdout (REQ-021): two
consecutive applies against an unchanged source repository produce
byte-identical stdout in both `--json` and human modes, and no wall-clock
timestamp / PID / random ID leaks into user-facing output. The runs use
the built `target/debug/patina` binary in a tempdir fully isolated via
`PATINA_REPO` / `HOME` / `USERPROFILE` / `XDG_STATE_HOME` / `LOCALAPPDATA`
so the apply never touches the developer's real `$HOME`.

## Coverage rule

REQ-021 attaches exactly one `<scenario>` (CHK-035: clean state, two
consecutive `--yes --json` applies, `diff -u out1 out2` empty),
demonstrated in Scenario 1 below. The remaining `<done-when>` /
`<behavior>` bullets — human-mode byte-identity and the grep-clean
"no non-deterministic input on output paths" property — are additional
coverage in Scenarios 2 and 3.

This task was an audit-confirms-clean plus add-the-regression-test task:
the foundations laid by earlier tasks already satisfy REQ-021
(`discover_modules` sorts module handles alphabetically; `walk_files`
sorts copy-tree entries; the JSON envelope and diff renderer are pure
functions of the resolved plan and on-disk state with no timestamp / PID
/ random input). No production source changed; the deliverable is the
regression test that locks the property in place.

## Session 2026-05-29-T021-attempt-1

### Scenario 1 — two consecutive `--yes --json` applies are byte-identical (CHK-035)

A fixture with a `copy`-mode and an implicit-template (`.tmpl`) entry
across two modules. A priming apply converges the repo, then two measured
`--yes --json` applies run against the now-unchanged repo.

<red>
Before the regression test existed, REQ-021 was asserted only by grep +
code review — there was no executable gate that two consecutive applies
emit byte-identical stdout. A future change that reintroduced a
timestamp, PID, or unstable collection ordering into the JSON envelope or
the diff renderer would have shipped undetected. A naive first cut of the
test (comparing a from-scratch apply against an immediately-following
no-op apply) failed loudly, surfacing that the human-mode diff legitimately
differs between a from-scratch run (`+ indent = 2`, an insertion) and a
converged run (`  indent = 2`, a context line) — confirming the property
REQ-021 actually guards is stability *as a function of inputs*, i.e. two
runs that observe the same on-disk state.
</red>

<green>
`patina-cli/tests/deterministic_stdout.rs::json_apply_is_byte_identical_across_two_runs`:
primes the repo, then asserts `first.stdout == second.stdout` byte-for-byte
across the two measured `--yes --json` runs. Passes.
</green>

### Scenario 3 — multi-target rows follow input declaration order, not alphabetised

A multi-target `[[file]]` whose `targets` are declared
`["~/.codex/agent.toml", "~/.claude/agent.toml"]` (deliberately
non-alphabetical). "Deterministic" means "stable function of inputs", not
"alphabetised": the plan rows must preserve the declared order.

<green>
`patina-cli/tests/deterministic_stdout.rs::multi_target_rows_preserve_input_declaration_order`:
parses the `--json` envelope's `plan` array and asserts the `.codex`
target row precedes the `.claude` target row (declared order), not sorted.
Passes.
</green>

### Grep audit — no non-deterministic input on output paths

```
grep -rn 'Utc::now|SystemTime::now|process::id|jiff::Timestamp::now' \
  patina-cli/src patina-core/src \
  | grep -v 'patina-core/src/journal/' | grep -v '#[cfg(test)]'
```

Result: a single hit, `patina-cli/src/cmd/apply.rs:267` inside
`current_timestamp()`. That value keys the journal `<ts>` filename only —
the SPEC's explicitly-permitted timestamp location — and never reaches
stdout (the JSON envelope and diff renderer never reference it). The
function already carries a doc comment documenting this exemption. This is
the SPEC's own carve-out, not a violation.
