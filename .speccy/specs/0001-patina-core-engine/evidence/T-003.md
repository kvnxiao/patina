## REQ-003 — Repository-root resolution

### CHK-005 — `PATINA_REPO` resolves the repo root from an unrelated CWD

`env_var_resolves_repository_root` in
`patina-core/tests/repo_discovery.rs` constructs a tempdir with a root
`patina.toml` (`root = true`), sets `env_value = Some(repo)`, sets
`cwd` to an unrelated tempdir, and asserts the resolution equals the
canonicalized repo path. Source 1 wins over sources 2 and 3.

### CHK-006 — Walk-up from `T/zsh/` finds `T/patina.toml`

`walk_up_finds_root_from_subdirectory` in `repo_discovery.rs` creates
`T/patina.toml` (root) plus `T/zsh/` (no manifest), passes
`cwd = T/zsh` with `env_value = None`, and asserts the resolution
equals T. Source 2 fires when source 1 is absent.

### CHK-007 — Triple-fail error names `PATINA_REPO`, walk-up, persisted default

`all_sources_failing_names_each_source` in `repo_discovery.rs`
asserts the error matches `RepoDiscoveryError::AllSourcesFailed` and
that its rendered `Display` string contains all three required
substrings (`PATINA_REPO`, walk-up reference, persisted-default
reference). The substring contract is hard-coded in the
`AllSourcesFailed` `Display` template at
`patina-core/src/discovery/repo.rs` (the three-source rendering block).

Robustness coverage beyond the CHK set:
- `empty_env_var_is_treated_as_unset` — `PATINA_REPO=""` falls through
  to source 2 rather than erroring.
- `env_var_pointing_at_non_root_directory_errors` — a `PATINA_REPO`
  that points at a directory whose `patina.toml` is missing `root =
  true` surfaces a typed error rather than silently walking up.
- `persisted_default_is_consulted_when_other_sources_fail` — positive
  coverage that source 3 is actually exercised when sources 1 and 2
  fail (not in the CHK list but reinforces the triple-source contract
  named by CHK-007).

## REQ-004 — Flat depth-1 module enumeration

### CHK-008 — Discovers `{nvim, zsh}` alphabetically with absolute paths

`discovers_modules_alphabetically_with_absolute_paths` in
`patina-core/tests/module_discovery.rs` constructs the
SPEC-described layout (root manifest with `root = true`, plus
`zsh/patina.toml` and `nvim/patina.toml` declaring no `root` key),
runs `discover_modules(root)`, and asserts both the ordered name
vector (`["nvim", "zsh"]`) and that each module path is absolute and
rooted under the canonicalized tempdir.

### CHK-009 — Depth-2 manifest rejected with `maximum module depth` phrase naming the offending path

`rejects_manifest_at_depth_two` in `module_discovery.rs` constructs
`T/zsh/plugins/patina.toml`, runs `discover_modules(T)`, asserts the
error matches `ModuleDiscoveryError::MaximumModuleDepth`, and asserts
the rendered `Display` contains both the phrase `maximum module
depth` and the offending path. The path-substring assertion tolerates
either POSIX or Windows separator forms (Windows backslash vs POSIX
slash) so the test passes on every host.

Additional task-scenario coverage (named in the task body but not
enumerated as a numbered CHK):
- `rejects_non_root_manifest_declaring_root_true` — a depth-1 module
  manifest with `root = true` is rejected with a typed error.
- `rejects_root_manifest_missing_root_key` — the root manifest must
  declare `root = true`; absence is a typed error.
- `non_module_subdirectories_are_silently_skipped` — `.git/` and
  similar subdirectories without a `patina.toml` are not classified
  as modules and do not error (judgment call consistent with REQ-004's
  `<done-when>`, which only mandates rejecting depth-≥2 and root-key
  misuse).
