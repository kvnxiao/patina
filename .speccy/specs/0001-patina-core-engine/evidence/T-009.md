Red-then-green paper trail for T-009: `patina_core::paths` —

## Coverage rule note

REQ-010 carries exactly one `<scenario>`, CHK-021. Its full
end-to-end surface (`patina apply --yes --json` reporting a
`repo_root` JSON field) cannot run today: the `patina apply` CLI
subcommand, the `--json` envelope, and the `repo_root` output field
all land in T-016 (REQ-017). T-009 is the library task that
*produces* the canonical absolute path that surface will later
report. The scenario below demonstrates the load-bearing property
at the layer T-009 owns: a relative repository path resolves to a
canonical, absolute, dot-free root. CHK-021 is labelled `demonstrated`
for that property with the end-to-end CLI assertion deferred to T-016.

## Session 2026-05-28-T009-attempt-1

### Scenario 1 — Relative repository path resolves to canonical absolute root (CHK-021)

<red>
Pre-edit (`git show HEAD:patina-core/src/discovery/repo.rs`): repo
root resolution called `path.canonicalize_utf8()` /
`dir.canonicalize_utf8()` directly. On the filesystem branch this
already yields an absolute path, but there was no
lexical-fallback helper and no shared `paths` module — a
not-yet-existing path (the common target-path case the SPEC's
second `<behavior>` bullet calls out) had no canonicalization path
at all and any caller would have hit a filesystem
`NotFound` error from `canonicalize_utf8`. Demonstrated by the
absence of the module:

```
$ git show HEAD:patina-core/src/paths.rs
fatal: path 'patina-core/src/paths.rs' does not exist in 'HEAD'
$ grep -rn "canonicalize_utf8" patina-core/src/discovery/repo.rs
165:            .canonicalize_utf8()
186:                dir.canonicalize_utf8()
```

A `canonicalize` of `~/.config/foo/bar.conf` whose `~/.config/foo/`
parent does not exist would, pre-edit, have no helper to call and
the only available primitive (`canonicalize_utf8`) returns
`Err(NotFound)` for a non-existent path — i.e. the REQ-010
second-behavior case errors rather than falling back lexically.
</red>

<green>
Post-edit: `paths::canonicalize` exists with the two-branch
contract, and discovery routes through it. The relative-repo
resolution and the not-yet-existing-target lexical fallback are
asserted by the integration test
`patina-core/tests/path_canon.rs`:

```
$ cargo test -p patina-core --test path_canon
running 5 tests
test expand_tilde_then_canonicalize_yields_home_relative_absolute ... ok
test nonexistent_target_under_missing_grandparent_does_not_error ... ok
test nonexistent_leaf_under_existing_parent_resolves_through_parent_symlinks ... ok
test existing_path_resolves_absolute_with_no_dot_segments ... ok
test relative_repo_resolves_to_canonical_absolute_root ... ok

test result: ok. 5 passed; 0 failed; 0 ignored
```

`relative_repo_resolves_to_canonical_absolute_root` builds a repo at
`T/dot`, resolves it through `resolve_repository_root_with`, and
asserts the result equals the canonical absolute `T/dot` with no
`/./` or `/../` segments — then independently canonicalizes the
relative-style `T/./dot` form and asserts it folds to `T/dot`. This
is the value the T-016 `--json` surface will report as `repo_root`,
matching CHK-021's `repo_root == /tmp/work/dot` expectation at the
layer that computes it.

`nonexistent_target_under_missing_grandparent_does_not_error`
canonicalizes `T/cfg/foo/bar.conf` where the whole `cfg/foo/` chain
is absent and asserts an absolute path comes back rather than an
error — the REQ-010 second-behavior bullet (lexical fallback for a
not-yet-created parent).
</green>
