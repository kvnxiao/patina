# Evidence — SPEC-0001 / T-014

The five file-mode executors with multi-target fan-out (REQ-005). T-014
owns the **execution** side of each `FileMode` variant: given an
already-canonicalized absolute source and already-resolved absolute target
paths, materialize the source at every target per the mode and return one
`CompletionRecord` per materialized filesystem object. The parse-time
half of REQ-005 (the `target`/`targets` XOR rule, the mode allowlist, the
implicit-`.tmpl` rule) is owned by T-004's `config::file_entry` and proven
in `patina-core/tests/config_file_entry.rs`; T-014 consumes the resolved
`FileMode` rather than re-validating it.

The literal `patina apply --yes` surface the CHK/scenario prose names does
not exist yet (the `apply` subcommand, the plan loop that calls these
executors, the diff-and-prompt gate, and journal/backup wiring land with
T-015 / T-016 and the orchestrator). T-014's unit is the executor
functions; the tests drive the public `patina_core::materialize` entry
point directly against tempdir fixtures that stage the on-disk state each
scenario describes.

New files:

- `patina-core/src/apply/mod.rs` — module doc, `Materialization`,
  `CompletionRecord`, `ExecutorError`, the `materialize` dispatch entry
  point, and the shared `ensure_parent` helper.
- `patina-core/src/apply/symlink.rs` — per-file `Symlink` (with the
  directory-source per-file walk) and atomic `SymlinkDir`, plus the
  cross-platform `create_symlink` that maps a Windows privilege failure
  to the typed `WindowsSymlinkPermission`.
- `patina-core/src/apply/copy.rs` — single-file `Copy` and recursive
  `CopyTree`.
- `patina-core/src/apply/template.rs` — implicit `TemplateRender`,
  rendered once and written to each target with `.tmpl` stripped.
- `patina-core/tests/executor_modes.rs` — per-mode integration coverage.
- `patina-core/tests/multi_target.rs` — multi-target fan-out coverage.

Wired into `patina-core/src/lib.rs` (`pub mod apply;` plus re-exports of
`CompletionRecord`, `ExecutorError`, `Materialization`, `materialize`) and
`patina-core/src/error.rs` (`EngineError::Executor` via `#[from]`).

## Red/green pairs (integration)

### Scenario 1 — CHK-010 / CHK-041: file-source symlink to canonical source

Test: `symlink_mode_links_to_canonical_source` in
`patina-core/tests/executor_modes.rs`.

- Stages a `zshrc` source file, calls `materialize(FileMode::Symlink, ...)`
  with a single target.
- Green: the target's readlink resolves (canonicalized) to the canonical
  source path, and the returned `CompletionRecord` carries
  `Materialization::Symlink { link_target }` equal to the source. Default
  mode (`mode` omitted ⇒ `FileMode::Symlink`) shares this executor, so
  CHK-041 is the same code path.
- Red check: a writer that copied bytes instead of linking fails the
  readlink assertion; a missing record fails `records.len() == 1`.

### Scenario 2 — REQ-005: directory-source symlink walks per file

Test: `symlink_mode_directory_source_walks_per_file` in
`patina-core/tests/executor_modes.rs`.

- A `config/` source dir with `alias.sh` and `nvim/init.lua`.
- Green: two records, one symlink per source file at the mirrored target
  path, and the target directory itself is a real directory (not a single
  link) — proving the per-file walk rather than an atomic dir symlink.
- Red check: an atomic-dir-symlink implementation would yield one record
  and a symlinked target dir, failing both assertions.

### Scenario 3 — CHK (symlink-dir behavior): atomic single link, no walk

Test: `symlink_dir_mode_creates_single_atomic_link` in
`patina-core/tests/executor_modes.rs`.

- A `nvim/` source dir with a child file.
- Green: exactly one record; the target itself is a symlink whose
  canonicalized readlink equals the canonical source dir; no per-file walk
  occurred.

### Scenario 4 — CHK-043 (single-target): copy writes byte-identical file

Test: `copy_mode_writes_byte_identical_file` in
`patina-core/tests/executor_modes.rs`.

- Green: the target's bytes equal the source, the record is
  `Materialization::Copy`, and the target is a regular file (not a
  symlink).

### Scenario 5 — REQ-005: copy-tree mirrors a directory tree

Test: `copy_tree_mode_mirrors_directory` in
`patina-core/tests/executor_modes.rs`.

- A `src/` tree with `a.txt` and `deep/b.txt`.
- Green: two records, both files mirrored at the target with byte-identical
  content under the nested structure.

### Scenario 6 — CHK-011: template renders and strips the `.tmpl` suffix

Test: `template_render_mode_strips_suffix_and_renders` in
`patina-core/tests/executor_modes.rs`.

- A `gitconfig.tmpl` source `"[user]\n    email = {{ email }}"` with
  `email` resolved through the repo-shared layer.
- Green: the output lands at the target with `.tmpl` stripped, contains the
  rendered `email = kevin@example.com`, the record is
  `Materialization::Render`, and the `.tmpl` target path does not exist.
- Red check: a writer that left the `.tmpl` suffix, wrote the raw template,
  or produced a symlink fails the corresponding assertion.

### Scenario 7 — CHK-042: multi-target symlink fan-out

Test: `symlink_fans_out_to_every_target` in
`patina-core/tests/multi_target.rs`.

- One source, two targets (`~/.claude/...`, `~/.codex/...`).
- Green: two records in target order; both targets are symlinks pointing
  (canonicalized) at the same canonical source path.

### Scenario 8 — CHK-043: multi-target copy fan-out

Test: `copy_fans_out_to_every_target` in
`patina-core/tests/multi_target.rs`.

- Green: both targets are regular files with byte-identical content equal
  to the source.

### Scenario 9 — CHK-044: multi-target template renders once

Test: `template_renders_once_and_writes_each_target` in
`patina-core/tests/multi_target.rs`.

- A `.tmpl` source fanned out to two targets.
- Green: both outputs (with `.tmpl` stripped) carry byte-identical rendered
  content (`rendered1 == rendered2`), and neither `.tmpl` target path
  exists. The render-once guarantee is structural in `template::render`:
  `engine.render` is called once before the per-target write loop, and the
  same `rendered` bytes are written to each target; the byte-identity
  assertion is the observable witness.

## Windows symlink permission — judgment-only

The CHK/scenario for `EngineError::WindowsSymlinkPermission` (a Windows
host without Developer Mode hitting a `mode = "symlink"` op) cannot be
exercised deterministically in a test: the test host's privilege level is
fixed at run time and a CI runner with Developer Mode on (or elevation)
would create the link rather than fail. The typed surface is in place and
inspectable on the diff: `apply/symlink.rs::classify_symlink_error` maps a
`std::io::ErrorKind::PermissionDenied` from the Windows
`symlink_file`/`symlink_dir` call to `ExecutorError::WindowsSymlinkPermission`
(carrying the target path and the underlying error), and that variant
threads through `EngineError::Executor` via `#[from]`. The prompt/elevate
flow is explicitly deferred to SPEC-0002 per the task; T-014 only needs to
surface the typed error so the CLI can exit non-zero with the message. The
reviewer judges this surface on the diff.

## Parse-time CHKs owned by T-004 — covered elsewhere

CHK-012 (`merge-json` rejected), CHK-045 (`target` + `targets` both),
CHK-046 (neither `target` nor `targets`), and CHK-047 (`targets = []`) are
parse-time rules of REQ-005 enforced in T-004's `config::file_entry` and
demonstrated in `patina-core/tests/config_file_entry.rs`. T-014 consumes
the already-resolved `FileMode`/`targets` and does not re-validate the
schema, so these CHKs are not re-proven in T-014's diff.
