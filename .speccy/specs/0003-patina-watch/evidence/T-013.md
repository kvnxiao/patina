# Evidence for SPEC-0003 T-013

Red-then-green paper trail for T-013: the Windows per-user Scheduled
Task `ServiceBackend` (`service/scheduled_task.rs`), behind the T-011
trait, and the factory dispatch that routes a Windows host to it.

A note on scope of demonstration. The genuinely host-coupled happy
paths — `ITaskService::Connect` / `RegisterTaskDefinition` actually
registering a live Scheduled Task (CHK-003's registration half) and
`DeleteTask` removing it, plus `get_State` / `get_LastTaskResult`
liveness queries — are **not** driven end-to-end from a test, for the
same reason the macOS `launchd` and Linux `systemd` backends' live
registration are not (T-011 / T-012 evidence): registering a live
per-user task escapes the fixture's tempdir isolation (the Task
Scheduler is the developer's / CI runner's real session) and a real
COM round-trip needs a Windows host. Everything that is pure string
work — the Task Scheduler 2.0 registration XML the
`RegisterTaskDefinition` call consumes (the `LogonTrigger`,
`RunLevel=LeastPrivilege`, and `Exec` command/arguments CHK-003
asserts), the XML escaping, and the not-found `HRESULT`
classification — is demonstrated below over unit-level pairs.

A note on test execution. The `scheduled_task` module is
`#[cfg(windows)]`, so its unit tests compile and run only on a Windows
target. On the macOS dev host the test code is fully type-checked,
borrow-checked, and lint-checked by a cross-target
`cargo clippy --target x86_64-pc-windows-gnu --all-targets --all-features
-- -D warnings` run (which compiles the lib-test cfg without linking);
actually *running* the assertions needs the Windows CI leg (or a
mingw `dlltool` + wine locally, absent on this dev box). The Windows
CI leg is the canonical execution check, exactly as the Linux leg is
for T-012's systemd module.

## Session 2026-06-01-T013-attempt-1

### Scenario 1 — The Windows Scheduled Task registration XML exists and is correct (REQ-001; backs CHK-003)

<red>
Pre-edit: there was no Windows backend at all. `service/scheduled_task.rs`
did not exist (it is new in this turn, absent from `HEAD`), and the
factory's `current()` returned the `UnsupportedBackend` stub on Windows
(the `_ => Box::new(unsupported::UnsupportedBackend)` arm). A request to
render the `Patina Watcher` task XML or to classify a `GetTask`
not-found result had no code to call:

```
$ git show HEAD:patina-core/src/watch/service/scheduled_task.rs
fatal: path 'patina-core/src/watch/service/scheduled_task.rs' exists on disk, but not in 'HEAD'
```

The file is untracked at `HEAD`: none of the Scheduled Task backend
symbols (`ScheduledTaskBackend`, `render_task_xml`, `is_not_found`)
existed before this turn.
</red>

<green>
Post-edit: `service/scheduled_task.rs` carries `render_task_xml`,
`is_not_found`, and the `ScheduledTaskBackend` lifecycle, each covered
by a unit test asserting the REQ-001 / CHK-003 task descriptor shape.
The module is `#[cfg(windows)]`, so the test code is compiled and
lint-checked on the Windows target (linking — and therefore execution —
needs the Windows CI leg):

```
$ cargo clippy -p patina-core --target x86_64-pc-windows-gnu \
    --all-targets --all-features -- -D warnings
    Checking patina-core v0.1.0 (/Users/kevin/src/patina/patina-core)
    (zero scheduled_task findings; lib + lib-test cfg compiled clean)
```

The lib-test cfg compiles clean under `-D warnings`, so the eight
`scheduled_task::tests::*` unit tests
(`render_task_xml_declares_the_logon_trigger_run_level_and_exec_action`,
`render_task_xml_is_well_formed_task_scheduler_2_0_xml`,
`render_task_xml_escapes_special_characters_in_the_binary_path`,
`xml_escape_escapes_the_five_special_characters`,
`is_not_found_recognizes_the_task_not_found_hresults`,
`map_task_status_reports_running_for_the_running_state`,
`map_task_status_treats_a_non_running_state_as_stopped`,
`map_task_status_reports_none_exit_code_when_never_run`) type-check and
borrow-check.
`render_task_xml_declares_the_logon_trigger_run_level_and_exec_action`
asserts the rendered XML contains `<LogonTrigger>`,
`<RunLevel>LeastPrivilege</RunLevel>`, the canonical binary as the
`<Command>`, and `watch --foreground` as the `<Arguments>` — the exact
trigger / run-level / action CHK-003 and REQ-001's done-when name. The
namespace + escaping tests gate that the XML the live
`RegisterTaskDefinition` consumes is well-formed and injection-safe.
</green>

## Session 2026-06-01-T013-attempt-2

### Scenario 3 — The status state-mapping reports never-run as null and RUNNING as running (REQ-003; backs CHK-006)

<red>
Pre-edit (attempt-1 WIP): the `(get_State, get_LastTaskResult,
get_LastRunTime)` → status mapping was fused inline into
`query_task_status`, behind the live-COM `GetTask` read. There was no
pure entry point to call, so the three predicates — `state ==
RUNNING` (liveness), `result != SCHED_S_TASK_HAS_NOT_RUN` (never-run →
`None` exit code, CHK-006), and `date != 0.0` (epoch → `None`
last-fired) — had no deterministic test. Dropping the
`SCHED_S_TASK_HAS_NOT_RUN` filter (so a never-run task reports
`Some(266499)` instead of `null`) or flipping the `RUNNING` comparison
left the whole suite green: the mapping sat on the judgment-only side of
the CI line, unlike the launchd / systemd siblings which extract the
mapping into a pure `parse_*` free function.
</red>

<green>
Post-edit: the value-mapping is extracted into the pure free function
`map_task_status(state: co::TASK_STATE, result: i32, last_run: f64) ->
TaskReadout` (mirroring `parse_launchctl_print` / `parse_systemctl_show`);
`query_task_status` now reads the three COM properties and delegates the
mapping to it. Three unit tests gate the mapping deterministically, with
no live Task Scheduler:

- `map_task_status_reports_none_exit_code_when_never_run` — the CHK-006
  case: `(READY, SCHED_S_TASK_HAS_NOT_RUN, 0.0)` maps `last_exit_code`
  and `last_fired_at` to `None`. Dropping the sentinel filter now fails
  this assertion (`Some(266499) != None`).
- `map_task_status_reports_running_for_the_running_state` —
  `(RUNNING, 0, 45000.5)` maps `running = true` with the recorded exit
  code / timestamp surfaced. Flipping the `RUNNING` comparison now fails
  this assertion.
- `map_task_status_treats_a_non_running_state_as_stopped` —
  `(READY, 2, 45000.0)` maps `running = false` while still surfacing the
  recorded non-zero exit code (the installed-but-stopped shape).

These three compile and lint-check clean on the Windows target alongside
the other five (`cargo clippy --target x86_64-pc-windows-gnu
--all-targets --all-features -- -D warnings`, zero `scheduled_task`
findings); the Windows CI leg runs the assertions. CHK-006's never-run →
`null` exit-code mapping is now on the demonstrated side of the CI line.
The live COM round-trip (`Connect` / `RegisterTaskDefinition` / live
`get_State`) stays judgment-only — that is the host-coupled half no test
can drive in CI.
</green>
