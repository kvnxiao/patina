# Evidence for SPEC-0003 T-010

Red-then-green paper trail for the watcher drift-detection handler
(`patina-core/src/watch/drift.rs` + the `watch/mod.rs` select-loop
content-target arm). The "red" baseline is the T-009 loop, whose
content-target-only branch was an explicit no-op (the T-008/T-009
module doc stated "until the drift handler arrives, a batch touching
only content-target paths is a no-op"). The "green" state is the
T-010 drift handler: it re-hashes the touched content target, logs a
`drift` event, upserts the `<state>/patina/drift.cache`, and notifies
through an injectable sink gated by the per-target 60s rate limit.

The notification-count assertions (CHK-011 "exactly one") run against
the in-memory capture sink in `patina-core` unit tests (DEC-013: the
CLI binary always uses the real `notify-rust` sink, uncaptured on
headless CI). The on-disk + status side-effects run end-to-end against
a real watcher subprocess in `patina-cli/tests/watch_foreground_cli.rs`.

## Session 2026-05-31-T010-attempt-1

### Scenario 1 — Divergent copy target notifies exactly once and records H1/H2 (CHK-011)

<red>
Pre-edit (T-009 loop): a debounced batch touching only the content
target `~/.gitconfig` fell through to the no-op branch — no hash
compare, no `drift` event, no cache write, no notification. A capture
sink driven through the handler did not exist (the handler did not
exist), and `<state>/patina/drift.cache` was never written by the
watcher. A test asserting `sink.count() == 1` and a populated cache
would observe `0` notifications and a missing cache file.

```
// T-009 mod.rs select-loop, content-target-only batch:
//   } else if source_event(&batch, &watch_set.sources) {
//       let _outcome = reapply::run_reapply().await;
//   }
//   // no else arm — content-only batch silently ignored
```
</red>

<green>
Post-edit: the `patina-core` unit test
`watch::drift::tests::divergent_target_notifies_once_and_records_h1_h2`
records a content target with the recorded hash `H1 = blake3("H1")`,
writes the target's live bytes to `"H2"` (`H2 = blake3("H2") ≠ H1`),
and drives `handle_target_events` with a `CaptureSink`. It asserts:
`outcomes == [DriftOutcome::Notified]`; `sink.count() == 1`; the one
captured notification has title `"Patina: drift detected"` and a body
naming `.gitconfig`; and the decoded `drift.cache` holds one entry with
`expected_hash == H1`, `actual_hash == H2`, `expected != actual`.

```
test watch::drift::tests::divergent_target_notifies_once_and_records_h1_h2 ... ok
```

End-to-end, `foreground::an_external_target_edit_logs_drift_and_populates_the_cache`
spawns a real `patina watch --foreground` over an applied copy-mode
`~/.gitconfig`, overwrites the target out-of-band, and asserts the
watcher logs a `drift` event and the on-disk `drift.cache` (decoded via
`patina_core::load_drift_cache_file`) holds an entry for `.gitconfig`,
while no `re_apply` event fires (a content edit must not re-apply).

```
test foreground::an_external_target_edit_logs_drift_and_populates_the_cache ... ok
```
</green>

### Scenario 2 — Second detection within 60s is rate-limited (DEC-004)

<red>
Pre-edit: no rate limit existed because no notification path existed —
N edits within any window produced 0 notifications.
</red>

<green>
Post-edit: `watch::drift::tests::second_detection_within_the_window_is_rate_limited`
notifies on a first detection at `t = 1000`, rewrites the target to a
third distinct content, then detects again at `t = 1030` (30s, inside
the 60s window). It asserts `outcomes == [DriftOutcome::RateLimited]`,
`sink.count() == 1` (the second is suppressed), and that the cache entry
is still refreshed to the latest observation (`actual_hash` and
`detected_at_unix` updated). The complementary test
`detection_past_the_window_notifies_again` detects at `t = 1000` and
`t = 1061` (61s, past the window) and asserts `sink.count() == 2`,
proving the limit is a sliding window, not a one-shot mute.

```
test watch::drift::tests::second_detection_within_the_window_is_rate_limited ... ok
test watch::drift::tests::detection_past_the_window_notifies_again ... ok
```
</green>

### Scenario 3 — Status reports DRIFTED from its own live re-hash, not the cache (CHK-012)

<red>
Pre-edit: with no drift handler, `~/.gitconfig` overwritten out-of-band
was still classified DRIFTED by `patina status` — because status has
always derived DRIFTED from SPEC-0001 REQ-018's own live re-hash,
independent of the (then-nonexistent) cache. This scenario is owned by
SPEC-0001; T-010's contribution is to confirm the watcher's cache does
NOT feed it.
</red>

<green>
Post-edit: the end-to-end test parses `patina status --json` after the
out-of-band edit and asserts the `files` array carries an entry whose
`path` contains `.gitconfig` with `state == "drifted"`, and that the
aggregate `drifted` counter is `>= 1`. The DRIFTED verdict comes from
the live re-hash (`status/classify.rs`), proven by the existing
`patina-cli` integration test `edited_copy_target_reports_drifted`,
which reports DRIFTED with no watcher running at all.

```
test foreground::an_external_target_edit_logs_drift_and_populates_the_cache ... ok
test edited_copy_target_reports_drifted ... ok
```
</green>

### Scenario 4 — Symlink target produces no drift entry (DEC-008)

<red>
Pre-edit: symlink targets were already excluded from the subscription
set (`compute_subscriptions`, T-007) so no FS event for a symlink
target reached drift detection; but with no handler there was no
positive proof a content target *does* drift while a symlink does not.
</red>

<green>
Post-edit: `subscriptions::tests::watch_set_separates_sources_from_content_targets`
asserts `WatchSet.content_targets` holds exactly the one content
target with its recorded hash and that the symlink target contributes
no content target — so the drift handler is never handed a symlink
path. The handler only iterates `content_targets`, so a symlink target
cannot drift (DEC-008).

```
test watch::subscriptions::tests::watch_set_separates_sources_from_content_targets ... ok
```
</green>
