# Evidence for SPEC-0003 T-004

Red-then-green paper trail for the drift-cache format and module
(REQ-007 format slice): `patina-core/src/watch/drift_cache.rs` plus the
`watch/` module root, encode/decode over the T-001 version-envelope
helper with the cache's OWN `DRIFT_CACHE_MAJOR_VERSION`, an atomic
tempfile-then-rename `write_drift_cache`, a `load_drift_cache_file`
loader, and a `render_drift_cache` formatter.

This task lands the format and its read/write/decode surface only — no
watcher, no notifications, no `patina debug drift-cache` subcommand.
REQ-007's runtime CHKs (CHK-011 drift detection + notify, CHK-012
status independence, CHK-018 the debug subcommand) are out of T-004's
scope and land in T-005 / T-008 / T-010; see the journal roll call.
The proofs here track the four `<task-scenarios>` in TASKS.md.

## Session 2026-05-31-T004-attempt-1

`load_drift_cache_file`, including the bound `journal_ts` and both
32-byte hashes.

The compile failure is the red state: the round-trip behaviour has no

</red>

<green>

timestamp and both hashes.

`write_then_load_round_trips_through_the_public_api` asserts

</green>

### Scenario 2 — Newer drift-cache major is refused naming both versions (task-scenario 2)

Proves a cache whose envelope major is `DRIFT_CACHE_MAJOR_VERSION + 1`
is refused on load with a typed `DriftCacheError::VersionMismatch`
naming both the found and supported majors.

<red>
Pre-edit: no `DriftCacheError` type and no decode path exist, so the
assertion that load returns `VersionMismatch { found, supported }`
cannot be expressed — the test does not compile (same
`unresolved import` failure as Scenario 1).
</red>

<green>
Post-edit: a tampered envelope (major bumped by 1) is refused.

```
$ cargo test -p patina-core --test drift_cache newer_major_load_is_refused_naming_both_versions
running 1 test
test newer_major_load_is_refused_naming_both_versions ... ok
test result: ok. 1 passed; ...
```

`newer_major_load_is_refused_naming_both_versions` writes a cache,
overwrites its leading `u16` with `DRIFT_CACHE_MAJOR_VERSION + 1`, and
asserts `load_drift_cache_file` returns
`DriftCacheError::VersionMismatch { found, supported }` with
`found == DRIFT_CACHE_MAJOR_VERSION + 1` and
`supported == DRIFT_CACHE_MAJOR_VERSION`.
</green>

### Scenario 3 — Drift-cache major is independent of the journal major (task-scenario 3)

Proves the drift cache decodes using its own major and does not
reference the journal's `FILE_MAJOR_VERSION`, so the two formats version
separately.

<red>
Pre-edit: `DRIFT_CACHE_MAJOR_VERSION` does not exist; there is nothing
to compare against `FILE_MAJOR_VERSION` and no encode path to inspect
(same compile-failure red state).
</red>

<green>
Post-edit: the two constants are distinct, and an encoded cache carries
the drift-cache major in its envelope (not the journal's) and decodes.

```
$ cargo test -p patina-core --test drift_cache drift_cache_major_is_independent_of_the_journal_major
running 1 test
test drift_cache_major_is_independent_of_the_journal_major ... ok
test result: ok. 1 passed; ...
```

The integration test asserts `DRIFT_CACHE_MAJOR_VERSION != FILE_MAJOR_VERSION`
and that the encoded envelope's leading `u16` equals
`DRIFT_CACHE_MAJOR_VERSION`. The sibling unit test
`decode_uses_its_own_major_not_the_journals` reads the envelope version
back via `version_envelope::read_envelope_version` and confirms it is
the drift-cache major. Structurally, `DriftCache::decode` calls
`decode_envelope(bytes, DRIFT_CACHE_MAJOR_VERSION)` — it never names
`FILE_MAJOR_VERSION`.
</green>

### Scenario 4 — Write lands via rename, no in-place truncation (task-scenario 4)

Proves `write_drift_cache` stages into a sibling tempfile and renames it
over the destination, leaving no staging tempfile and never truncating
the destination in place.

<red>
Pre-edit: no `write_drift_cache` exists, so there is no write path to
inspect for the rename guarantee (same compile-failure red state).
</red>

<green>
Post-edit: writing a second cache over an existing one leaves no
`drift.cache.tmp` sibling and the destination holds the second cache in
full.

```
$ cargo test -p patina-core --test drift_cache atomic_write_replaces_via_rename_leaving_no_staging_tempfile
running 1 test
test atomic_write_replaces_via_rename_leaving_no_staging_tempfile ... ok
test result: ok. 1 passed; ...
```

`atomic_write_replaces_via_rename_leaving_no_staging_tempfile` writes a
first cache, overwrites it with a second, then asserts the sibling
`drift.cache.tmp` does not exist and `load_drift_cache_file` returns the
second cache in full. The sibling unit test
`write_lands_via_rename_leaving_no_tempfile_and_no_inplace_truncation`
exercises the same path and asserts the staging tempfile is renamed
away; `sibling_temp_path_stays_in_the_same_directory` pins the tempfile
to the destination's directory so the rename is atomic (same
filesystem) rather than a cross-device copy.
</green>
