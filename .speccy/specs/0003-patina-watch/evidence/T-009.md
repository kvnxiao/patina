# Evidence for SPEC-0003 T-009

Red-then-green paper trail for the watcher re-apply handler
(`patina-core/src/watch/reapply.rs` + the `watch/mod.rs` select-loop
dispatch). The "red" baseline is the T-008 loop, which logged
`watch_event` on every batch and took no mutating action; the "green"
state is the T-009 classifier-and-dispatch wiring. Both states are
exercised by `patina-cli/tests/watch_foreground_cli.rs` against an
isolated repo/state/home fixture, with the watcher spawned as a real
subprocess and its stderr drained on a reader thread.

## Session 2026-05-31-T009-attempt-1

### Scenario 1 — Five touches within the debounce window coalesce to one re_apply (CHK-010)

<red>
Pre-edit (T-008 loop): a batch of five rapid source writes coalesced
into a single `watch_event` line, but the loop took no action — there
was no `re_apply` event at all because the re-apply handler did not
yet exist. The T-008 module doc states the loop "logs receipt of each
debounced batch and leaves the mutating work to those tasks". So a
test asserting `count_event_lines("patina_core: re_apply re_apply_id")
== 1` would observe `0` against the T-008 binary.

```
INFO patina_core: watch_started subscriptions=3
INFO patina_core: watch_event paths=1
(no re_apply line — handler absent)
```
</red>

<green>
Post-edit: `five_touches_within_the_debounce_window_coalesce_to_one_reapply`
writes the watched source `git/gitconfig` five times at ~20ms spacing
(≈100ms total, well inside the 500ms window), waits, and asserts
exactly one `re_apply` event line and zero `re_apply_failed`. The
single coalesced batch drives one `NonBlocking` re-apply; the
re-apply's own journal write comes back as a journal event and routes
to rescan (not another re-apply), and its content-target rewrite
routes to the drift no-op — so the count stays at exactly one.

```
INFO patina_core: watch_started subscriptions=3
INFO patina_core: watch_event paths=1
INFO patina_core: re_apply re_apply_id="20260531T234131Z" \
  re_apply_duration_ms=14 re_apply_files_changed=1 re_apply_result="applied"
```

```
$ cargo test -p patina-cli --test watch_foreground_cli \
    five_touches_within_the_debounce_window_coalesce_to_one_reapply
test foreground::five_touches_within_the_debounce_window_coalesce_to_one_reapply ... ok
```
</green>

### Scenario 2 — A watcher re-apply commits exactly one new journal record and logs no contention skip (CHK-013 surface)

<red>
Pre-edit (T-008 loop): a watched-source edit produced a `watch_event`
but no re-apply, so the journal COMMIT count stayed at the fixture's
initial `1` forever and no `re_apply` / `lock_contention_skip` event
was ever emitted. A test asserting the COMMIT count reaches `2` after
an edit would time out at `1` against the T-008 binary.
</red>

<green>
Post-edit: `a_watcher_reapply_commits_exactly_one_new_journal_record`
edits the watched source, waits for the `re_apply` event, and polls the
journal directory until it holds exactly two `.COMMIT` sentinels (the
fixture's initial apply plus the watcher's one re-apply). It also
asserts `count_event_lines("lock_contention_skip") == 0`: the watcher
held the lock for its own uncontended re-apply, so no skip occurred.
This is the deterministic single-process slice of CHK-013's
"exactly two committed plans" contract; the concurrent-CLI lock race
(watcher wins, Blocking CLI waits behind it) is an engine lock-policy
property already proven by `patina-core`'s
`non_blocking_contention_*` unit tests in `apply/engine.rs`.

```
INFO patina_core: re_apply re_apply_id="…" re_apply_files_changed=1 \
  re_apply_result="applied"
(journal dir now holds two <ts>.COMMIT sentinels; zero lock_contention_skip lines)
```

```
$ cargo test -p patina-cli --test watch_foreground_cli \
    a_watcher_reapply_commits_exactly_one_new_journal_record
test foreground::a_watcher_reapply_commits_exactly_one_new_journal_record ... ok
```
</green>

### Scenario 3 — A parallel CLI apply triggers a journal rescan, not a re-apply loop (CHK-017 / REQ-005 loop guard)

<red>
Pre-edit (during T-009 development, before the source-vs-target
classifier landed): classifying *any* non-journal batch as a re-apply
trigger caused an unbounded loop — each re-apply rewrote the copy-mode
content target, whose FS event was misclassified as a re-apply trigger,
which re-applied again. The observed log showed `re_apply` firing every
~500ms without end:

```
INFO patina_core: watch_event paths=4
INFO patina_core: re_apply re_apply_id="20260531T233840Z" …
INFO patina_core: watch_event paths=4
INFO patina_core: re_apply re_apply_id="20260531T233841Z" …
INFO patina_core: watch_event paths=4
INFO patina_core: re_apply re_apply_id="20260531T233842Z" …
(repeats unbounded)
```
</red>

<green>
Post-edit: the loop now routes journal-sentinel events to rescan and
content-target-only events to the drift no-op, re-applying *only* on a
repository source-path event.
`a_parallel_cli_apply_triggers_a_journal_rescan` runs a parallel
`patina apply --yes` (writing a fresh `.plan`/`.COMMIT`) and asserts the
watcher logs a `journal_rescan` event and that the rescan count stays
bounded (`< 10`) — no runaway loop.

```
INFO patina_core: watch_started subscriptions=3
INFO patina_core: watch_event paths=4
INFO patina_core: journal_rescan commits="20260531T….COMMIT" subscriptions=3
(bounded; no re_apply storm)
```

```
$ cargo test -p patina-cli --test watch_foreground_cli \
    a_parallel_cli_apply_triggers_a_journal_rescan
test foreground::a_parallel_cli_apply_triggers_a_journal_rescan ... ok
```
</green>
