## Session 2026-05-31-T003-attempt-1

### Scenario 1 — Off Windows, an ordinary write failure surfaces with no retry event (CHK-016)

Integration test
`non_windows_write_failure_surfaces_without_retry_event` in
`patina-core/tests/fs_retry.rs`. A `FileMode::Copy` is driven through
the public `materialize` entry point into a directory stripped of
write permission (`chmod 0o500`), so the byte-copy write fails with the
OS's ordinary `EACCES`. A minimal in-test `tracing::Subscriber`
(`RecordingSubscriber`, `enabled` returns `true` for every event so a
stray retry event cannot be filtered out) records the `message` field
of every event for the duration of the call.

<red>
Pre-edit: before this task the three write sites called
`fs_err::copy` / `fs_err::write` / `symlink_*` directly with no retry
wrapper, so the concept "no `fs_write_retry` event is emitted" was
vacuously true but the wrapper that the assertion guards did not yet
exist — there was no `with_sharing_violation_retry`, no
`fs_write_retry` event name, and no module `patina_core::apply::retry`.
A test referencing the wrapper would fail to compile:

```
error[E0433]: failed to resolve: could not find `retry` in `apply`
error[E0425]: cannot find function `with_sharing_violation_retry`
```

The red state here is the compile failure: the production surface the
green assertion exercises (the wrapped write sites and the
debug-event-or-not branch) did not exist.
</red>

<green>
Post-edit: the three write sites route through
`with_sharing_violation_retry`, which off Windows runs the operation
exactly once and emits no event. The test drives a failing copy and
asserts (a) the result is `Err` — the EACCES surfaces immediately, on
the first attempt — and (b) no recorded event message contains
`fs_write_retry`.

```
$ cargo test -p patina-core --test fs_retry
running 1 test
test non_windows_write_failure_surfaces_without_retry_event ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

The directory permission is restored to `0o700` before the assertions
so the tempdir cleans up regardless of outcome.
</green>
