# Evidence for SPEC-0003 T-012

Red-then-green paper trail for T-012: the Linux `systemd --user`
`ServiceBackend` (`service/systemd.rs`), the factory dispatch that
routes a systemd Linux host to it and a non-systemd Linux host to the
`unsupported` foreground-escape-hatch stub (DEC-010), and the
not-systemd `unsupported` fallback the factory returns.

A note on scope of demonstration. The genuinely host-coupled happy
paths — `systemctl --user enable --now` actually enabling and starting
a live unit (CHK-002's registration half) and `systemctl --user
disable` + unit removal of a live service (CHK-005's live half) — are
**not** driven end-to-end from a test, for the same reason the macOS
`launchd` backend's live registration is not (T-011 evidence): enabling
a live per-user service escapes the fixture's tempdir isolation (the
`systemd --user` manager is the developer's / CI runner's real session)
and is unsafe in CI. Everything that is pure string or filesystem work
is demonstrated below over unit-level red/green pairs: the unit-file
render (the `Restart=on-failure` / `WantedBy=default.target` /
`ExecStart … watch --foreground` directives CHK-002 asserts the file
contains), the unit-file write, the `systemctl --user show` status
parse, and the unit-directory resolution. The supervisor-free CLI
surface (the not-installed no-ops shared with the launchd backend) is
demonstrated over the integration binary.

## Session 2026-06-01-T012-attempt-2 (round-1 review blockers)

Round-1 review raised two blockers, both addressed here by amending the

### Scenario 3 — The systemd-availability routing decision is deterministically gated (DEC-010; tests blocker)

The `SystemdBackend::is_available()` probe is the factory match guard
that selects the systemd backend vs the `UnsupportedBackend`
`--foreground` fallback. Round 1 inlined the stderr→bool decision into
the spawn, so no test could gate it.

<red>
Pre-fix: the decision lived inline inside `is_available()` as
`!stderr.contains("Failed to connect to bus")`, behind a live
`Command` spawn. There was no pure entry point to test, and no test
named the routing predicate:

```
$ rg 'fn bus_reachable' patina-core/src
$ echo $?
1
```

`rg` exits `1`: the pure helper did not exist; inverting the inlined
predicate left the whole suite green.
</red>

<green>
Post-fix: the decision is extracted into a pure
`fn bus_reachable(stderr: &str) -> bool`, and
`bus_reachable_is_false_only_when_the_bus_connect_fails` gates both
arms — the connect-failure stderr routes to the unsupported fallback,
and a degraded/running/empty stderr routes to the systemd backend.
Inverting `bus_reachable` now fails that test.

```
$ cargo test -p patina-core --lib watch::service::systemd   # gate widened on macOS dev host
test watch::service::systemd::tests::bus_reachable_is_false_only_when_the_bus_connect_fails ... ok
... (12 passed; 0 failed)
```
</green>

### Scenario 4 — The systemd `ExecStart=` path is quoted and escaped (REQ-001; security blocker)

Round 1 interpolated the canonical binary path raw into `ExecStart=`,
unlike the launchd sibling that XML-escapes the same path. systemd's
`ExecStart` word-splits on whitespace, expands `%` specifiers, and
treats a newline as the end of the directive.

<red>
Pre-fix: `render_unit` joined the tokens with a bare space
(`.join(" ")`) and interpolated the path directly, so a path with a
space was word-split, a `%` was specifier-expanded, and a `\n` would
terminate the directive and start a fresh one. There was no test
exercising a hostile path.
</red>

<green>
Post-fix: each `ExecStart` token is rendered through
`systemd_exec_quote` — wrapped in `"…"`, with `\`/`"` backslash-escaped,
`\n`/`\r`/`\t` rendered as C-style escapes, and `%` doubled to `%%`.
`render_unit_neutralizes_space_percent_and_newline_in_the_binary_path`
gates a path containing all three: it asserts no injected
`\nExecStartPost=` directive appears and that the path lands as a single
quoted-and-escaped word. `systemd_exec_quote_escapes_quotes_and_backslashes`
gates the quote/backslash escapes directly. The two pre-existing
`ExecStart` assertions (`render_unit_*`, `write_unit_*`) were updated to
the quoted form `ExecStart="/usr/local/bin/patina" "watch" "--foreground"`.

```
$ cargo test -p patina-core --lib watch::service::systemd   # gate widened on macOS dev host
test watch::service::systemd::tests::render_unit_neutralizes_space_percent_and_newline_in_the_binary_path ... ok
test watch::service::systemd::tests::systemd_exec_quote_escapes_quotes_and_backslashes ... ok
test watch::service::systemd::tests::render_unit_contains_the_required_directives_and_exec_start ... ok
test watch::service::systemd::tests::write_unit_writes_a_valid_unit_to_the_target_path ... ok
... (12 passed; 0 failed)
```
</green>

`target_os = "linux"` gate and clippy `-D warnings`) was verified

the module active, then reverting to the strict gate. The Linux CI leg
is the canonical check for the gated code.
