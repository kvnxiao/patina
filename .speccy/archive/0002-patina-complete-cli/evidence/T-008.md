# Evidence for SPEC-0002 T-008

Paper trail for T-008: the standalone Windows-only `patina-elevate` helper
crate (REQ-008). The crate is the workspace's third member; it parses one
subcommand (`enable-developer-mode`) and, on Windows, sets the Developer
Mode registry switch under one-time UAC elevation. It has no dependency on
`patina-core` / `patina-cli` (DEC-002) and is dropped from non-Windows
release artifacts (DEC-003 / CHK-015). The `ShellExecuteEx`/`runas` launch,
the engine gate, and the CLI prompt are out of scope here — they land in
T-009.

## Crate shape

- `patina-elevate/Cargo.toml`: third workspace member. `clap` (workspace)
  for arg parsing; `winsafe` (workspace, `advapi`+`kernel`, **not**
  `taskschd`) under `[target.'cfg(windows)'.dependencies]` only. No
  `patina-core`/`patina-cli` dependency (DEC-002).
- `[[bin]] required-features = ["windows"]` with a `windows = []` feature,
  OFF by default. A plain `cargo build --release` skips the bin (Cargo
  silently drops a bin whose required feature is off), so no
  `patina-elevate` artifact is emitted on macOS/Linux (CHK-015). Windows
  release packaging and the crate's own tests turn it on with
  `--features windows`. DEC-003 accepts this explicit build flag as the
  documented "build-config wart".
- `src/lib.rs`: the clap `Cli`/`Command` surface plus `run(&Command)`,
  factored into a library so the parsing contract is unit-testable on any
  host without the gated binary artifact.
- `src/main.rs`: thin entry point — `Cli::parse()` then
  `patina_elevate::run(&cli.command)`.
- `src/devmode.rs`: `enable_developer_mode()`. `#[cfg(windows)]` does the
  real `RegCreateKeyEx` + `RegSetValueEx` write; `#[cfg(not(windows))]`
  returns `DevModeError::NotWindows`. The Developer Mode key/value
  constants are duplicated verbatim from `patina-core::windows::registry`
  on purpose (DEC-002 forbids the cross-crate dependency).

## No-panic invariant

No `unwrap`/`expect`/`panic!` in production code. The workspace panic-family
clippy denials are inherited via `[lints] workspace = true`. The single
`eprintln!` (the exit-1 typed-error path) carries a scoped
`#[expect(clippy::disallowed_macros, reason = ...)]`: the helper has no
`output::Reporter` (DEC-002 forbids the `patina-core` dependency that would
bring it), and a raw stderr write is the documented primitive — exactly the
carve-out the workspace `clippy.toml` comment sanctions. `.expect()` appears
only in `#[cfg(test)]` / `tests/*.rs`.

## Windows-target verification (one-off)

The `#[cfg(windows)]` registry-write path cannot compile on the macOS CI
host. Following the T-007 precedent, the Windows path was type-checked and
clippy-linted by cross-compiling (CI runs the native target only; this
cross-check is recorded here, not added to CI):

```
$ cargo clippy -p patina-elevate --features windows \
    --target x86_64-pc-windows-gnu --all-targets -- -D warnings
    Checking patina-elevate v0.1.0 (/Users/kevin/src/patina/patina-elevate)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.43s
$ echo $?
0
```

This exercises `HKEY::LOCAL_MACHINE.RegCreateKeyEx` (NON_VOLATILE,
`co::KEY::SET_VALUE`), `.RegSetValueEx(Some(..), RegistryValue::Dword(1))`,
and `co::ERROR::ACCESS_DENIED` — all resolve against the real winsafe
0.0.27 API. (`cargo build` for that target needs mingw `dlltool` to link
`windows-sys`, which is absent on this host; `clippy`/`check` type-checks
the full crate without the final link, which is what proves the API calls
are real.)

## CHK roll call (REQ-008)

- CHK-014 (elevated `enable-developer-mode` sets the flag to 1, exit 0):
  judgment-only for CI / demonstrated-by-host-test. Needs a real elevated
  Windows host with Developer Mode OFF and a UAC accept; neither is
  available in macOS/Linux automation. Encoded as the `#[cfg(windows)]`
  `#[ignore]` test `enable_developer_mode_elevated_sets_flag_and_exits_0`
  in `tests/cli.rs`, runnable by hand with `--ignored` on an elevated
  Windows shell. The Windows registry-write path it drives is
  type-checked/clippy-clean via the cross-compile above.
- CHK-015 (Linux release build emits `patina` but no `patina-elevate`):
  demonstrated. `tests/artifact_absence.rs::release_build_emits_patina_but_not_patina_elevate`
  drives a real `cargo build --workspace --release --message-format=json`
  in a hermetic target dir and asserts the emitted-executable set contains
  `patina` and contains neither `patina-elevate` nor `patina-elevate.exe`.
  Green run captured in the suite output below.

## Task-scenario evidence

Three `<task-scenarios>`:

### Scenario 1 — non-Windows release build drops the artifact (= CHK-015)

`tests/artifact_absence.rs`. Authoritative because it reads Cargo's own
`compiler-artifact` JSON (an artifact Cargo did not build cannot appear),
rather than scanning a shared `target/` that could hold stale binaries.
Red/green: if the `[[bin]]` lost its `required-features` gate (or the
feature were defaulted on), `patina-elevate` would appear in the emitted
set and the `!any(== "patina-elevate")` assertion would fail. With the gate
in place the test is green.

### Scenario 2 — unknown subcommand exits 2 with usage listing the subcommand

Process level: `tests/cli.rs::unknown_subcommand_exits_2` asserts
`patina-elevate frobnicate` exits 2 (clap usage error via `Error::exit`).
`tests/cli.rs::help_lists_the_supported_subcommand` asserts the usage
surface (`--help`, exit 0) lists `enable-developer-mode` — clap enumerates
subcommands in the help, not in the bare unknown-subcommand error.
In-process: `lib.rs` unit tests `unknown_subcommand_is_a_usage_error`
(asserts `ErrorKind::InvalidSubcommand`),
`rendered_help_lists_the_supported_subcommand`, and
`missing_subcommand_does_not_run_an_action`
(`ErrorKind::DisplayHelpOnMissingArgumentOrSubcommand`) — all cross-platform.

### Scenario 3 — elevated Windows toggle sets the flag, exit 0 (= CHK-014)

The `#[cfg(windows)]` `#[ignore]` host test (see CHK-014 above).


`enable_developer_mode_off_windows_exits_1` (a `#[cfg(not(windows))]` test)
proves the exit-1 typed-error path: off Windows the action returns
`DevModeError::NotWindows`, the process exits 1, and the message reaches
stderr. On Windows the same exit-1 contract is reached when non-elevated
(the `ERROR_ACCESS_DENIED` branch of `registry_error`), which the
cross-compile type-checks.
