# Evidence for SPEC-0002 T-001

Paper trail for the format-preserving `toml_edit` manifest writer
(`patina-core::config::writer`). T-001 builds the writer primitive
that REQ-002 (`patina add`) and REQ-003 (`patina remove`) will call;
the end-to-end command CHKs (CHK-003..006) are demonstrated by the
later CLI-wiring tasks. This task's behaviour is proved by unit
tests run under `cargo test --workspace`, captured below.

## Coverage rule

T-001 introduces no new user-facing command, so it has no
red-then-green command-level scenarios. The writer's contract is
proved by the unit tests in
`patina-core/src/config/writer.rs` (`mod tests`). The journal roll
call labels CHK-003..006 `hygiene` against these tests, because the
behaviours the CHKs assert at the command level are exactly the
writer behaviours these tests gate at the unit level.

## Writer unit-test results

`cargo test -p patina-core config::writer` — 7 passed, 0 failed:

- `remove_file_entry_preserves_siblings_hook_variables_and_comments`
  — removing one `[[file]]` (target `~/.zshrc`) from a manifest with
  a sibling `[[file]]`, a `[[hook]]`, a `[variables]` table, and a
  hand-written comment leaves all of them intact and the removed
  target absent. This is the load-bearing DEC-007 / REQ-010
  determinism proof: a one-entry delete does not rewrite siblings or
  drop comments. Underpins CHK-005 (`patina remove` rewrites the
  owning module's `patina.toml` removing only the one entry).
- `append_file_entry_into_empty_text_round_trips_through_parser`
  — appending into empty text yields a `[[file]]` the read-side
  parser resolves to `source="zshrc"`, `target="~/.zshrc"`,
  `FileMode::Symlink`. Underpins CHK-003 (`patina add` writes a
  `[[file]]` entry with the expected keys).
- `scaffold_root_manifest_emits_root_marker_and_timestamp` — the
  scaffolded root manifest carries `[patina]` `root = true` and
  `created_at = "2026-05-30T12:00:00Z"`. Backs REQ-001's root
  manifest (consumed by `patina init`, a later task).
- `append_each_mode_uses_the_parser_accepted_spelling` — the four
  user-declarable modes serialize as `symlink` / `symlink-dir` /
  `copy` / `copy-tree`, the exact spellings `file_entry.rs` accepts.
- `append_template_render_omits_mode_key` —
  `FileMode::TemplateRender` emits no `mode` key (the parser rejects
  an explicit template mode).
- `remove_file_entry_errors_when_no_entry_matches` — a missing
  target yields `ConfigWriteError::EntryNotFound`.
- `append_to_malformed_text_surfaces_a_parse_error` — malformed
  input yields `ConfigWriteError::Parse` rather than a panic.
