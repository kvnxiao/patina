# Evidence for SPEC-0002 T-006

Red-then-green paper trail for `patina promote <target>` (REQ-004, REQ-009,
REQ-010). The red phase runs the new `patina-cli/tests/promote_cli.rs` crate
against a binary with the `Promote` subcommand wiring absent; the green phase
runs it against the implemented command. The scenarios back the
`demonstrated` CHK labels in `journal/T-006.md`.

## Red capture method

The tracked CLI/engine wiring edits (`patina-cli/src/cli.rs`,
`patina-cli/src/main.rs`, `patina-cli/src/cmd/mod.rs`) were `git stash`ed
while the new untracked `patina-cli/src/cmd/promote.rs` and
`patina-cli/src/cmd/managed.rs` (both unreferenced by the reverted
`cmd/mod.rs`, so not compiled in) and the new `patina-cli/tests/promote_cli.rs`
crate were kept present, then the crate was run. With the `Promote` arm gone
from the clap surface, `patina promote ...` is an unrecognized subcommand and
clap exits `2`, not the `0`/`1` the tests expect.

```

assertion `left == right` failed: promote must exit 0; stderr: error: unrecognized subcommand 'promote'
  left: 2
 right: 0
```

### Scenario 1 — `patina promote ~/.gitconfig --yes` rewrites the source and re-journals the new hash (CHK-007)

`promote_copy_target_rewrites_source_and_rejournals`.

<red>
Pre-edit: the `Promote` subcommand wiring is stashed, so `patina promote
~/.gitconfig --yes` is a clap usage error (exit 2). The exit-0 assertion
fails; the source-rewrite / hash / status-CLEAN assertions are never reached:

```
thread 'promote_copy_target_rewrites_source_and_rejournals' panicked at patina-cli/tests/promote_cli.rs:77:5:
assertion `left == right` failed: promote must exit 0; stderr: error: unrecognized subcommand 'promote'
  left: 2
 right: 0
```
</red>

<green>
Post-edit: the fixture first `patina apply --yes`s a `git` module whose
copy-mode `[[file]]` materializes `~/.gitconfig` from `<repo>/git/gitconfig`
(content `[user]\nemail = old@example.com`). The test then overwrites
`~/.gitconfig` with `[user]\nemail = new@example.com` (the external edit) and
runs `patina promote ~/.gitconfig --yes`:

- the repository source `<repo>/git/gitconfig` now holds the new bytes
  (`[user]\nemail = new@example.com`);
- the most recent journal record's expected hash for `~/.gitconfig` equals
  `content_hash` of the new bytes (decoded from the fresh `<ts>.COMMIT` the
  `Held`-policy re-apply wrote);
- a subsequent `patina status --json` classifies `~/.gitconfig` as `clean`.

All asserted; the test exits 0.

```
test promote_copy_target_rewrites_source_and_rejournals ... ok
```
</green>

### Scenario 2 — promoting a template-rendered target refuses, mutating nothing (CHK-008)

`promote_template_target_refuses`.

<red>
Pre-edit: `patina promote ~/.gitconfig --yes` is a clap usage error (exit 2),
not the expected exit 1:

```
thread 'promote_template_target_refuses' panicked at patina-cli/tests/promote_cli.rs:157:5:
assertion `left == right` failed: promoting a template target must exit 1; stderr: error: unrecognized subcommand 'promote'
  left: 2
 right: 1
```
</red>

<green>
Post-edit: the fixture applies a `git` module whose `[[file]]` declares
`source = "gitconfig.tmpl"` (implicit template mode). `patina promote
~/.gitconfig --yes` finds the journaled `ExpectedTarget::Content` whose source
ends in `.tmpl` and refuses:

- exits 1;
- writes a stderr message naming the `.tmpl` source `gitconfig.tmpl` and the
  word `template` (templating is non-invertible, DEC-006);
- leaves the template source `<repo>/git/gitconfig.tmpl` unchanged (the
  refusal is before any mutation).

All asserted; the test exits 0.

```
test promote_template_target_refuses ... ok
```
</green>
