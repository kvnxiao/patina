# Evidence for SPEC-0002 T-007

Paper trail for T-007: the cross-platform Developer Mode detection
module `patina-core::windows` (REQ-007 read side per DEC-008). This task
delivers detection (registry / token reads behind a `#[cfg(windows)]`
seam), the pure helpers (`is_unc_path`, `plan_has_symlink_op`), and the
testable gate-decision function (`decide_symlink_gate` over a
`DevModeProbe`). The elevation launch and the engine/CLI gate wiring land
in T-009.

## Coverage note

REQ-007's two scenarios, CHK-012 and CHK-013, are full end-to-end runs on
a Windows host with a TTY/UAC harness — both depend on machinery that
lands in T-009 (the engine gate + `ShellExecuteEx`/`runas` elevation
launch + CLI `Reporter` prompt). Neither is demonstrable from T-007's
read-only slice, and the macOS/Linux CI has no Windows host, so both are
`judgment-only` for this task and will be `demonstrated` against T-009.

What T-007 *does* deliver is verified two ways below: (1) the three
`<task-scenarios>` are proved by unit tests that run under the standard
`cargo test` hygiene gate on the CI host (`hygiene` label), and (2) the
otherwise-uncompilable `#[cfg(windows)]` registry/token path is
type-checked and clippy-clean against the real `winsafe` 0.0.27 API via a
cross-compile to `x86_64-pc-windows-gnu` — captured as a one-off
verification below since CI does not build that target.

## Task-scenario evidence

The three `<task-scenarios>` are unit tests in
`patina-core/src/windows/mod.rs`. They are deterministic and run on every
platform; the assertions encode the expected behaviour directly.

### Scenario A — non-Windows reads return NotWindows / false, no registry access

`windows::tests::non_windows_status_is_not_windows_and_unelevated`. On the
macOS/Linux CI host (`#[cfg(not(windows))]`), `dev_mode_status()` returns
`DevModeStatus::NotWindows` and `is_elevated()` returns `false`. No
registry access is possible because the `registry` submodule is compiled
out entirely under `#[cfg(windows)]` — the `#[cfg(not(windows))]` arms in
`dev_mode_status` / `is_elevated` are literal returns with no call into
`registry`, which the structure of the source enforces.

### Scenario B — `is_unc_path` distinguishes UNC from POSIX

`windows::tests::unc_prefix_check_distinguishes_unc_from_posix`:

```
is_unc_path(r"\\fileserver\share\dotfiles") == true
is_unc_path("/home/user/dot")               == false
is_unc_path(r"\Users\dot")                  == false  // single backslash is drive-relative
```

### Scenario C — gate decision: Disabled+not-elevated requires elevation; Enabled proceeds

`windows::tests::gate_disabled_not_elevated_requires_elevation` and
`gate_enabled_proceeds`. Against a fake `DevModeProbe` and a plan
containing one `FileMode::Symlink` op:

```
probe = Disabled + not-elevated  ⇒  decide_symlink_gate == RequireElevation
probe = Enabled  + not-elevated  ⇒  decide_symlink_gate == Proceed
```

Companion cases lock the remaining branches: `Disabled + elevated ⇒
ProceedElevatedWarning`, a copy-only plan ⇒ `Proceed` regardless of
status, and `Unsupported ⇒ Proceed`.
