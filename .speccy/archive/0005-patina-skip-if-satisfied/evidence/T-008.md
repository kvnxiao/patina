## Session 2026-06-03-T008-attempt-1

The "red" baseline is the pre-T-008 rollback replay. `replay_entry`
snapshotted and reverted *every* target of a `[[file]]` entry, with no
notion of per-target disposition: it took `&[&str]` target paths and
passed each to `revert_target`. `revert_target`
(`patina-core/src/rollback/replay.rs`) had only two arms â€” backup
present â†’ `clone_entry` (restore), no backup â†’ `remove_entry` (delete).
A target the apply had recorded as `Unchanged` carries no backup (apply
skipped both its write and its backup), so on rollback it fell into the
no-backup arm and was **deleted** â€” destroying a file the apply never
touched.

T-008 introduces `RevertTarget { target, disposition }` and filters
`Disposition::Unchanged` targets out of the snapshot/roll-forward set in
`replay_entry` before `revert_target` is ever called, so an Unchanged
target is left in place regardless of backup presence. `Create` (no
backup â†’ delete) and `Update` (backup â†’ restore) flow through
`revert_target` exactly as before.

### Scenario 1 â€” Mixed Create/Update/Unchanged rollback leaves Unchanged byte-identical (CHK-010)

End-to-end test
`rollback_leaves_unchanged_deletes_create_and_restores_update` in
`patina-cli/tests/skip_if_satisfied.rs`, reinforced by unit tests
`unchanged_target_is_left_in_place` and
`mixed_entry_reverts_create_and_update_but_leaves_unchanged` in
`patina-core/src/rollback/replay.rs`.

<red>
Pre-edit (`replay_entry` taking `&[&str]`; `revert_target` with only the
two backup-presence arms): an `Unchanged` target carries no backup, so
on rollback it reaches the no-backup `else` arm at
`patina-core/src/rollback/replay.rs:202-206`
(`crate::fsx::remove_entry(target)`) and is **deleted**, even though the
apply never wrote or backed it up.

Demonstrated against the new tests by reverting the production filter:
removing the `Disposition::Unchanged` filter in `replay_entry` (so every
target is snapshotted and reverted) makes
`unchanged_target_is_left_in_place` go red â€” its
`assert!(target.exists(), ...)` fails because the Unchanged target with
no backup is removed by the no-backupâ†’delete arm. The e2e
`rollback_leaves_unchanged_deletes_create_and_restores_update` likewise
fails its post-rollback Unchanged-byte-identical assertion, since the
Unchanged target is gone.

```
$ cargo test -p patina-core --lib rollback::replay::tests::unchanged_target_is_left_in_place
# (with the Disposition::Unchanged filter removed from replay_entry)
thread '...' panicked at patina-core/src/rollback/replay.rs:
  assertion failed: target.exists()
test result: FAILED. 0 passed; 1 failed
$ echo $?
101
```
</red>

<green>
Post-edit: `replay_entry` filters `Disposition::Unchanged` out of the
`to_revert` set before snapshotting (an all-Unchanged entry returns
`Ok(())` before any stage dir is created), so an Unchanged target is
never snapshotted nor passed to `revert_target`. The `Create` target
(no backup) is still deleted and the `Update` target (backup present)
is still restored from backup.

The e2e test drives a real `patina apply --yes` then
`patina rollback --yes` over a single commit that records one of each
disposition (the live filesystem is pre-staged so apply classifies one
target Create, one Update, one Unchanged), and asserts all three
post-rollback outcomes: the Unchanged target's bytes are byte-identical
to its pre-rollback content, the Create target is absent, and the
Update target is restored to its pre-apply bytes.


`mixed_entry_reverts_create_and_update_but_leaves_unchanged` guards the
over-broad-filter regression: a single entry carrying all three
dispositions reverts Create (deleted) and Update (restored) while
leaving Unchanged in place, so a filter that swallowed Create-with-no-
backup would fail its `assert!(!create_target.exists())`.
</green>
