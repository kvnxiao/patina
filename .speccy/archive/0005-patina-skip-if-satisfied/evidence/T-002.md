# Evidence for SPEC-0005 T-002

Red-then-green paper trail for T-002: a `disposition: Disposition`
field threaded onto every `PlannedOperation` and `ExpectedTarget`
variant, surviving the postcard version envelope at major 1.

T-002 is a field-only, behavior-neutral task: it adds a field to the
two wire types and threads a placeholder (`Disposition::Create`)
through the live call sites so the workspace compiles. The observable
change is purely on the serialized wire form, so the demonstration is
an encode/decode round-trip rather than a CLI behaviour shift.

## Session 2026-06-02-T002-attempt-1

### Scenario 1 — Plan per-op dispositions round-trip (CHK-004)

<red>
Before this task, `PlannedOperation` carried no `disposition` field,
so there was nothing to round-trip and no test could assert one. A
test that constructs a `Plan` with one Create, one Update, and one
Unchanged op does not compile against the pre-edit type:

```
$ cargo test -p patina-core --lib journal::plan
error[E0061]: this function takes 2 arguments but 3 arguments were supplied
   --> patina-core/src/journal/plan.rs
    |  PlannedOperation::symlink("a", "/x/a", Disposition::Create)
    |                                         ^^^^^^^^^^^^^^^^^^^^ unexpected argument
```

The constructor took only `(source, target)`; the disposition the
scenario needs to thread did not exist on the type.
</red>

<green>
After this task, the constructors take the disposition and the new
`per_op_dispositions_round_trip` test (CHK-004) passes: a plan with
one Create, one Update, and one Unchanged op encodes through the
version envelope and decodes with each op's disposition unchanged.

```
$ cargo test -p patina-core --lib journal::plan::tests::per_op_dispositions_round_trip
test journal::plan::tests::per_op_dispositions_round_trip ... ok
test result: ok. 1 passed; 0 failed
```

The test reads each decoded op's disposition via the new
`PlannedOperation::disposition()` accessor and asserts the vector
equals `[Create, Update, Unchanged]`, so a field dropped from the
postcard wire would fail the assertion.
</green>

### Scenario 2 — ApplyRecord per-leaf dispositions round-trip at major 1 (CHK-005, CHK-017)

<red>
Before this task, `ExpectedTarget` carried no `disposition` field. A
record fixture asserting per-target dispositions does not compile:

```
$ cargo test -p patina-core --lib journal::record
error[E0063]: missing field `disposition` in initializer of `ExpectedTarget`
```
</red>

<green>
After this task, the `record()` fixture carries one Create, one
Update, and one Unchanged target, and the new
`per_leaf_dispositions_round_trip_at_major_one` test (CHK-005,
CHK-017) passes: the encoded record's envelope major byte reads `1`
and the decoded record's per-target dispositions equal the originals.

```
$ cargo test -p patina-core --lib journal::record::tests::per_leaf_dispositions_round_trip_at_major_one
test journal::record::tests::per_leaf_dispositions_round_trip_at_major_one ... ok
test result: ok. 1 passed; 0 failed
```

The test asserts both `read_envelope_version(&bytes) == 1` and that
the decoded dispositions are `[Create, Update, Unchanged]` via the
new `ExpectedTarget::disposition()` accessor.
</green>
