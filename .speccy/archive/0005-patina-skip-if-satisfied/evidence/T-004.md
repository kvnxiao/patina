# Evidence for SPEC-0005 T-004

Red-then-green paper trail for T-004: wiring the T-003 leaf
classifier into the plan path so each resolved target carries its
real `Disposition` on the in-memory `ResolvedOperation` and on the
durable `PlannedOperation` (per-op aggregate for tree modes,
DEC-007).

The red baseline is the pre-T-004 tree at the parent commit: T-002
threaded a `disposition` field through the wire types but seeded it
with the behaviour-neutral `PLACEHOLDER_DISPOSITION =
Disposition::Create` at every site, and the T-003 `classify_leaf`
classifier had no caller in the engine (it carried a
`#[cfg_attr(not(test), expect(dead_code, ...))]` annotation). So
before this task every planned operation reported `Create`
regardless of live target state, and there was no `classify_entry`
/ `classify_target` seam at all. The red runs below were taken
against that pre-edit shape; the green runs are the post-edit tests
that drive the new seam.

## Session 2026-06-02-T004-attempt-1

### Scenario 1 — Satisfied symlink, copy, and template all classify Unchanged (CHK-001)

<red>
Pre-edit: there is no `classify_entry` / `classify_target` function
in `patina-core/src/apply/engine.rs`, and `planned_operation`
hard-codes `PLACEHOLDER_DISPOSITION` (`Disposition::Create`). A
satisfied symlink/copy/template would therefore have been recorded
as `Create`, not `Unchanged`. The test below references the
not-yet-existing `classify_entry`, so the pre-edit crate does not
compile it:

```
$ cargo test -p patina-core --lib apply::engine::tests::satisfied_symlink_copy_and_template_all_classify_unchanged
error[E0425]: cannot find function `classify_entry` in this scope
```

(There was no plan-time classification call site to exercise on the
pre-edit tree; the placeholder always yielded `Create`.)
</red>

<green>
Post-edit: `classify_entry` renders the template once and routes
each target through `classify_leaf`. A symlink already pointing at
its source, a copy whose target bytes match, and a template target
holding the rendered output all classify `Unchanged`.

```
$ cargo test -p patina-core --lib apply::engine::tests::satisfied_symlink_copy_and_template_all_classify_unchanged
test apply::engine::tests::satisfied_symlink_copy_and_template_all_classify_unchanged ... ok
test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured
```
</green>

### Scenario 2 — Mutated copy is Update, deleted symlink is Create, template stays Unchanged (CHK-002)

<red>
Pre-edit: as above, the placeholder reports `Create` for every
target irrespective of live state, so a drifted copy would have
been mis-reported as `Create` and an unchanged template would also
have read `Create`. No classifier call site existed to produce the
correct `Update` / `Create` / `Unchanged` split.
</red>

symlink target deleted, and the template target left matching, the
classifier returns `Update`, `Create`, and `Unchanged`

</green>

### Scenario 3 — Copy-tree with one drifted leaf: that leaf Update, others Unchanged, aggregate Update (CHK-003)

<red>
Pre-edit: tree modes had no per-leaf classification at plan time
and the durable op carried the `Create` placeholder. A
partially-drifted tree could not report per-leaf dispositions, nor
the DEC-007 aggregate (`Unchanged` iff every leaf is `Unchanged`).
</red>

<green>
Post-edit: a `copy-tree` materialized to three leaves with one leaf
altered out of band classifies exactly that leaf `Update` and the
other two `Unchanged`, and the durable op's aggregate disposition is
`Update`. Two adjacent edge cases are also proven green: an absent
tree target aggregates to `Create` with no per-leaf enumeration, and
a fully-clean tree aggregates to `Unchanged`.

```
$ cargo test -p patina-core --lib apply::engine::tests::copy_tree_with_one_drifted_leaf_aggregates_to_update
test apply::engine::tests::copy_tree_with_one_drifted_leaf_aggregates_to_update ... ok
test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured

$ cargo test -p patina-core --lib apply::engine::tests::copy_tree_absent_target_is_create_aggregate
test apply::engine::tests::copy_tree_absent_target_is_create_aggregate ... ok

$ cargo test -p patina-core --lib apply::engine::tests::copy_tree_all_leaves_match_aggregates_to_unchanged
test apply::engine::tests::copy_tree_all_leaves_match_aggregates_to_unchanged ... ok
```
</green>

### Scenario 4 — Durable plan carries the classified disposition (REQ-002)

<red>
Pre-edit: `assemble_plan_operations` called
`planned_operation(mode, source, target)` with no disposition
argument; the function hard-coded `PLACEHOLDER_DISPOSITION`. The
classified disposition therefore never reached the durable
`PlannedOperation`, so `PlannedOperation::disposition()` always
returned `Create`.
</red>

<green>
Post-edit: `assemble_plan_operations` zips each target with its
`TargetDisposition` and threads the aggregate onto the durable
`PlannedOperation`, and the full per-target/per-leaf classification
onto the in-memory `ResolvedOperation`. A resolved entry classified
`Unchanged` assembles a `PlannedOperation::Copy` whose
`disposition()` is `Unchanged`.

```
$ cargo test -p patina-core --lib apply::engine::tests::assemble_plan_threads_disposition_onto_durable_operation
test apply::engine::tests::assemble_plan_threads_disposition_onto_durable_operation ... ok
test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured
```
</green>
