# Evidence for SPEC-0005 T-005

Red-then-green paper trail for T-005: execute skips both the write and
the backup for `Unchanged` targets (single-target entries and tree
leaves), and `build_apply_record` records every managed target —
including the skipped `Unchanged` ones — with its real plan-time
`Disposition`, sourced from the resolved plan rather than from the
written `CompletionRecord`s.

Two red baselines establish non-vacuity:

1. **The pre-T-005 tree.** Before this task the execute materialize
   loop backed up and re-materialized every target unconditionally
   (`backup_before_overwrite` per target, then `materialize` over all
   targets), and `build_apply_record` recorded only the written objects
   with `PLACEHOLDER_DISPOSITION = Disposition::Create`. So an
   already-satisfied target was rewritten and backed up on every
   re-apply, and the commit carried `Create` for everything.
2. **Mutation checks** against the implemented code, shown inline per
   scenario: disabling the skip or dropping `Unchanged` from the record
   flips each test red, proving the assertion gates the real invariant.

The green runs are the post-edit tests.

## Session 2026-06-02-T005-attempt-1

### Scenario 1 — Unchanged entry is not rewritten or backed up while drift is (CHK-006)

Test: `patina-cli/tests/skip_if_satisfied.rs::unchanged_entry_is_not_rewritten_or_backed_up_while_drift_is`.
Two `copy` entries; after the first apply both match their source. One
target (`b_out`) is drifted out of band, then `apply` re-runs. The
Unchanged target (`a_out`) must keep its mtime and produce no backup
entry; the drifted target must be re-materialized and its prior bytes
backed up.

<red>
Mutation: disable the `Unchanged` write/backup skip in
`materialize_target` (`if false && disposition.aggregate == Unchanged`).
The Unchanged target is then backed up like any other pre-existing
target:

```

the Unchanged target must produce no backup entry; found ["a_out", "b_out"]
test result: FAILED. 2 passed; 1 failed
```
</red>

<green>
Post-edit: an `Unchanged` aggregate returns from `materialize_target`
before any backup or write, so `a_out` keeps its mtime and bytes and
no `a_out` entry appears in the run's backup cycle, while `b_out` is
re-materialized and its prior (drifted) bytes land at the mirrored
backup path.

```

### Scenario 2 — copy-tree re-apply restores drift and backs up the tree as a unit (DEC-007 tree path)

Test: `patina-cli/tests/skip_if_satisfied.rs::copy_tree_re_apply_restores_drift_and_backs_up_the_tree_as_a_unit`,
plus the exact per-leaf write-skip executor unit tests
`patina-core/src/apply/copy.rs::copy_tree_only_writes_selected_leaves`
and `patina-core/src/apply/symlink.rs::tree_symlink_only_links_selected_leaves`.

The integration test drives a real `copy-tree` apply with one leaf
drifted: the drifted leaf is restored and the whole target directory is
backed up as a unit (the drifted leaf's prior bytes are recoverable).
The executor unit tests prove the per-leaf write-skip exactly: with a
`LeafWrite::Only` set naming one of two leaves, only that leaf is
written and the unselected leaf is never touched.

<red>
Mutation: make `LeafWrite::includes` always return `true` (write every
leaf regardless of the `Only` set):

```

</red>

<green>
Post-edit: `LeafWrite::Only(set)` writes only the leaves in the set; the
tree executors skip the rest and emit no record for them.

### Scenario 3 — Unchanged entry is recorded Clean and survives reap (CHK-007)
