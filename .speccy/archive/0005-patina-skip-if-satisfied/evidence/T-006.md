### Scenario 1 — Fully-satisfied re-apply writes no new journal or backup (CHK-011)

Integration test
`fully_satisfied_reapply_writes_no_new_journal_or_backup` in
`patina-cli/tests/skip_if_satisfied.rs`.

<red>
Pre-edit (`execute` with no short-circuit): a second `patina apply
--yes` over an unchanged repo flushes a fresh `<ts>.plan` and commits a
new `<ts>.COMMIT` (and, for any pre-existing target, a backup cycle).
The test snapshots `entry_names(journal_dir)` after the first apply,
then re-applies; the journal directory gains new `.plan` / `.COMMIT`
entries, so the `assert_eq!(entry_names(&journal_dir), journal_before)`
fails — `left` carries the new timestamped entries, `right` does not.
</red>

<green>
Post-edit: `is_full_noop` fires (every target `Unchanged`, prior commit
present, no orphan), so `execute` returns `ApplyResult::Applied {
up_to_date: true }` before `flush_plan_and_fsync`. The journal and
backups directories are byte-for-byte the same set of entries before
and after the second apply.

```
running 1 test
test fully_satisfied_reapply_writes_no_new_journal_or_backup ... ok
```
</green>

<round n="2">
Round-1 review (tests persona) showed the basename-set compare was
collision-blind: `current_timestamp()` is second-resolution, so two
back-to-back applies share a `<ts>` and a full plan→commit write cycle
overwrites `<ts>.COMMIT` in place — the basename set is identical either
way, and disabling `is_full_noop` left the test GREEN. Round 2 makes the
assertion collision-proof: it captures the sole `<ts>.COMMIT` file's
path, bytes, and mtime before the second apply and asserts all three are
unchanged afterward. Re-confirmed by mutation: forcing
`is_full_noop → Ok(false)` (a full write cycle) now turns the test RED
on the bytes assertion — the rewritten commit record's per-entry
`skipped` flag byte differs even within the same wall-clock second
(`left` byte `..., 1` vs `right` `..., 0` at the trailing disposition
flag). Reverted the mutation; the test is GREEN under the real engine.
</round>

### Scenario 3 — Fully-satisfied apply skips the prompt and reads no stdin (CHK-013)

Round-1 shipped only the subprocess test
`fully_satisfied_apply_without_yes_skips_prompt_and_reports_up_to_date`
(`patina-cli/tests/skip_if_satisfied.rs`), which runs on the non-TTY
path where the `Apply?` prompt is suppressed regardless — so its
`!stderr.contains("Apply?")` assertion was vacuous for the interactive
contract CHK-013 / REQ-009 actually names. Round 2 keeps that test for
the up-to-date stdout signal and adds an in-process unit test driving the
interactive prompt-skip directly.

Interactive unit test
`cmd::apply::tests::full_noop_interactive_skips_prompt_and_reads_no_stdin`
in `patina-cli/src/cmd/apply.rs`, with the red guard
`cmd::apply::tests::non_noop_interactive_does_prompt_and_reads_the_answer`.

The round-1 short-circuit was refactored to route the prompt decision
through a single testable helper `confirm_apply(is_full_noop, yes, tty,
reader, reporter) -> Confirmation`. The no-op branch returns
`Confirmation::Proceed` without touching `reporter.prompt` or
`reader.read_line`; `run` still skips the diff render and execute-as-no-op
exactly as before. (An in-process test of the full `run` is not possible:
`run` resolves the repo/state from process env, and the workspace forbids
the `unsafe` `set_var` in tests — see `watch/service/systemd.rs:248`. The
helper is the faithful injectable seam the review asked for.)

<red>
A `confirm_apply` that ignored `is_full_noop` (always prompting on an
interactive TTY) would call `PanickingReader::read_line` — which panics —
and push `Apply? [y/N] ` into the recording `BufferReporter.err`. The
no-op test asserts the reader is never read (no panic) and
`reporter.err.is_empty()` (zero prompt invocations); the red-guard test
proves the *non*-no-op interactive path genuinely prompts (asserts
`Apply?` present and a Declined verdict), so the skip cannot be a blanket
"never prompt".
</red>

<green>
Post-edit: `confirm_apply(true, false, Tty::Interactive, panicking_reader,
recording_reporter)` returns `Confirmation::Proceed`, emits no prompt
text, and reads no stdin. The non-no-op interactive counterpart prompts
and reads the answer. The subprocess test still proves the up-to-date
stdout line and exit 0 on the no-op path.

```

test cmd::apply::tests::non_noop_interactive_does_prompt_and_reads_the_answer ... ok
test cmd::apply::tests::yes_proceeds_without_prompting_on_any_tty ... ok
test cmd::apply::tests::non_tty_without_yes_previews_only ... ok
...
test fully_satisfied_apply_without_yes_skips_prompt_and_reports_up_to_date ... ok
```
</green>
