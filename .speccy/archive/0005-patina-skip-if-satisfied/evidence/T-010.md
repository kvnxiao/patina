# Evidence for SPEC-0005 T-010

Red-then-green paper trail for T-010: each `--json` plan entry carries a
`state` field equal to the target's classified disposition label, and a
fully-satisfied `apply --json` still emits the standard envelope shape with
every entry `unchanged`.

The red counterfactual for both scenarios is the same one-line production
edit: dropping the `"state": disposition.aggregate.label()` field from the
single-target row in `json_envelope`'s `plan_rows` helper
(`patina-cli/src/cmd/apply.rs`). With the field gone, the plan rows carry no
`state` key, so both new tests fail on the `state` lookup.

## Session 2026-06-03-T010-attempt-1

### Scenario 1 — Mixed plan stamps create / update / unchanged per entry (CHK-015)

Fixture: three `copy` `[[file]]` entries (`unchanged_out`, `create_out`,
`update_out`). The live filesystem is pre-staged so a single apply classifies
one of each — `unchanged_out` already holds the source bytes, `update_out`
exists with different bytes, `create_out` is absent.

<red>
With the `"state"` field removed from the single-target plan row in
`json_envelope`, the test's `state_for(&rows, "unchanged_out")` (and the
`create_out` / `update_out` lookups) find a row whose `state` key is absent,
so the `.expect(...)` in `state_for` fails:

```
thread 'json_plan_entries_carry_their_disposition_state' panicked at
patina-cli\tests\skip_if_satisfied.rs:...:
a plan row whose target basename matches, carrying a string `state`
...
test result: FAILED. 0 passed; 2 failed
```
</red>

<green>
With the `state` field present, each row carries the classified label:

```
running 2 tests
test fully_satisfied_json_emits_standard_envelope_all_unchanged ... ok
test json_plan_entries_carry_their_disposition_state ... ok

test result: ok. 2 passed; 0 failed; 0 ignored; 0 measured; 6 filtered out
```

The test asserts `state_for(rows, "unchanged_out") == "unchanged"`,
`"create_out" == "create"`, `"update_out" == "update"`, then re-applies twice
over the now-satisfied state and asserts the two `--json` stdouts are
byte-identical (the REQ-012 determinism tie).
</green>

Backing test: `json_plan_entries_carry_their_disposition_state`
(`patina-cli/tests/skip_if_satisfied.rs`). Scoped run:
`cargo test -p patina-cli --test skip_if_satisfied -- json_plan`.

### Scenario 2 — Fully-satisfied no-op emits the standard all-unchanged envelope (CHK-016)

Fixture: one `copy` entry and one template entry, converged by a priming
`--json --yes` apply so the measured apply is a full no-op.

<red>
With the `"state"` field removed, the all-unchanged assertion observes
`row.get("state") == None` for the first plan row and fails:

```
thread 'fully_satisfied_json_emits_standard_envelope_all_unchanged' panicked at
patina-cli\tests\skip_if_satisfied.rs:...:
assertion `left == right` failed: every entry of a fully-satisfied plan must be
`unchanged` (zero change counts); row: {"mode":"copy","source":...,"target":...}
  left: None
 right: Some("unchanged")
```
</red>

<green>
With the field present, the no-op `--json` apply emits the standard envelope
(the `plan_rows` test helper asserts all four top-level keys `repo_root`,
`profile`, `plan`, `result` are present — not a reduced shape), the plan array
is non-empty (every managed entry is still listed on a no-op), and every entry's
`state` is `"unchanged"` — the realization of "zero change counts" against an
envelope that reports state per entry:

```
test fully_satisfied_json_emits_standard_envelope_all_unchanged ... ok
```
</green>

Backing test: `fully_satisfied_json_emits_standard_envelope_all_unchanged`
(`patina-cli/tests/skip_if_satisfied.rs`). Scoped run:
`cargo test -p patina-cli --test skip_if_satisfied -- fully_satisfied_json`.
