# Evidence for SPEC-0005 T-007

Red-then-green paper trail for T-007: crash recovery leaves
plan-recorded `Unchanged` targets untouched (REQ-005).

## Coverage rule

Every `demonstrated` label in the journal `<implementer>` block's
`Evidence:` roll call is backed by exactly one `### Scenario N` block
here. CHKs labelled `hygiene` are proved by the standard test suite
(`just test` / `cargo test --workspace`); they have no scenario block.

## Session 2026-06-02-T007-attempt-1

The "red" baseline is the pre-T-007 `reverse_operation`: it had only
two arms — backup present → restore, no backup → delete. An orphan plan
whose target was marked `Unchanged` with no backup fell into the
no-backup arm and was **deleted**, even though apply never wrote or
backed it up (its live state already is the pre-apply state). T-007
adds a third arm ahead of the backup-presence branch: an op whose
plan-recorded disposition is `Unchanged` returns `Ok(())` and is left
in place regardless of backup presence.

### Scenario 1 — Unchanged-marked orphan target is preserved (CHK-008)

Unit test
`journal::recovery::tests::unchanged_marked_orphan_target_is_left_in_place`
in `patina-core/src/journal/recovery.rs`.

<red>
Pre-edit (`reverse_operation` with only the two backup-presence arms):
the crafted orphan plan carries a single `PlannedOperation::copy` with
`Disposition::Unchanged` and the target file exists with bytes
`matches-the-source`; no backup is stashed. Recovery probes the backup
path, finds nothing present, and takes the no-backup `else` arm —
`crate::fsx::remove_entry(target)` — deleting the target. The test's
`assert!(target.exists(), ...)` then fails because the Unchanged target
was removed.
</red>

<green>
Post-edit: `reverse_operation` checks `op.disposition() ==
Disposition::Unchanged` first and returns `Ok(())` before computing the
backup path, so the target is never touched. The orphan plan is still
consumed and cleaned up (`report.recovered_any()` is true), the target
still exists, and its bytes are byte-for-byte `matches-the-source`.
</green>

### Scenario 2 — Create-marked orphan with no backup is still deleted (CHK-009)

Unit test
`journal::recovery::tests::create_marked_orphan_target_with_no_backup_is_deleted`
in `patina-core/src/journal/recovery.rs`.

<red>
This guards that the new `Unchanged` arm does not swallow the
`Create`/no-backup case. The crafted orphan plan carries a single
`PlannedOperation::copy` with `Disposition::Create` and the target file
exists (the fresh creation the crashed apply made); no backup is
stashed. A naive "leave everything in place" implementation would fail
the `assert!(!target.exists(), ...)` because the fresh creation would
survive.
</red>

<green>
Post-edit: the disposition is `Create`, not `Unchanged`, so the new arm
does not fire; the no-backup `else` arm removes the fresh creation.
`report.recovered_any()` is true and the target no longer exists —
the unchanged Create-with-no-backup behavior is preserved.
</green>
