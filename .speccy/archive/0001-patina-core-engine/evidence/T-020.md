# Evidence for SPEC-0001 T-020

Red-then-green paper trail for the formalized CLI exit-code contract
(REQ-022): every terminal CLI state maps to exactly one of `0` / `1` /
`2` / `3` / `4` / `5` through a single `cli::resolve_exit_code` funnel and
an `ExitCode::from_error_chain` mapping. The runs use the built
`target/debug/patina` binary in a tempdir fully isolated via
`PATINA_REPO` / `HOME` / `USERPROFILE` / `XDG_STATE_HOME` / `LOCALAPPDATA`
so the apply never touches the developer's real `$HOME`.

## Coverage rule

The journal `<implementer>` block's `Evidence:` roll call labels each CHK
under REQ-022. REQ-022 attaches exactly one `<scenario>` (CHK-036:
`pre_apply` `command = "false"` → exit 2), demonstrated in Scenario 1
below. The remaining `<done-when>` / `<behavior>` bullets (exit 3, exit 1,
exit 4, exit 0) are additional coverage demonstrated in Scenarios 2-5.

## Session 2026-05-29-T020-attempt-1

### Scenario 1 — pre_apply hook `false` aborts before any file op, exit 2 (CHK-036)

A `[[hook]]` with `event = "pre_apply" command = "false"` and the default
`must_succeed = true`. `patina apply --yes` aborts before any file
operation runs and exits 2.

<red>
Pre-edit: a failed `must_succeed` `pre_apply` hook already returned
`ApplyResult::Aborted`, which `cmd::apply::exit_code_for` mapped to the
literal `2` — so this specific code happened to be correct before T-020.
What was *not* in place is the formalization: there was no `ExitCode`
enum, no single `cli::resolve_exit_code` funnel, and engine errors flowed
to `main`'s `anyhow::Result<()>` default handler, which prints `Error: …`
and exits a flat `1` for *every* engine failure — including the
exclusive-lock timeout, which REQ-022 requires to exit 4 (see Scenario 4
red). The exit-code contract was scattered across literal integers in
four `cmd` modules with no `EngineError -> ExitCode` mapping.
</red>

<green>
Post-edit, exercised by
`patina-cli/tests/exit_codes.rs::pre_apply_hook_failure_exits_2`: after a
`pre_apply` `false` hook, `patina apply --yes` exits 2 and no `~/.rc`
target is created.

```
$ cargo test -p patina-cli --test exit_codes pre_apply_hook_failure_exits_2
running 1 test
test pre_apply_hook_failure_exits_2 ... ok
test result: ok. 1 passed; 0 failed
```

The code now flows through `ApplyResult::Aborted -> ExitCode::PreApplyAbort
(2)` in `cmd::apply::exit_code_for`, returned as `Ok(2)` and passed
verbatim through `cli::resolve_exit_code`.
</green>

### Scenario 2 — post_apply hook failure rolls back, exit 3

A `post_apply` `exit 1` hook (default `must_succeed = true`). Apply runs
the file ops, the hook fails, the ops are rolled back, exit 3.

<red>
Pre-edit: the literal-`3` mapping existed in `exit_code_for`; T-020
re-expresses it as `ExitCode::PostApplyRollback`.
</red>

<green>
`patina-cli/tests/exit_codes.rs::post_apply_hook_failure_exits_3`: exit 3
and the copied `~/.rc` is reversed on rollback.

```
test post_apply_hook_failure_exits_3 ... ok
```
</green>

### Scenario 3 — TOML syntax error is a generic failure, exit 1

A module `patina.toml` with `source =` and no value is a TOML syntax
error. `patina apply --yes` exits 1 with the parse failure on stderr.

<red>
Pre-edit: this exited 1 only because `anyhow`'s default `main` handler
exits 1 on any `Err`. There was no typed mapping; the same flat 1 also
swallowed the lock-timeout case that REQ-022 assigns to 4.
</red>

<green>
`patina-cli/tests/exit_codes.rs::toml_syntax_error_exits_1_and_names_the_failure`:
exit 1 with a non-empty stderr. The chain now resolves through
`ExitCode::from_error_chain`: the wrapped `EngineError::ConfigParse` is not
a lock timeout, so it falls through to `ExitCode::Generic (1)`, and the
full `anyhow` context chain is rendered to stderr by `resolve_exit_code`.

```
test toml_syntax_error_exits_1_and_names_the_failure ... ok
```
</green>

### Scenario 4 — exclusive-lock timeout exits 4 (the load-bearing T-020 fix)

A second `apply` that cannot acquire a held exclusive lock within the
timeout cap exits 4. The test holds `<state>/patina/lock` in-process and
shrinks the subprocess cap via `PATINA_LOCK_TIMEOUT_MS=200`.

<red>
Pre-edit: a lock timeout surfaced as `EngineError::Lock(LockError::Timeout)`
wrapped by `anyhow` context and propagated to `main`'s default handler,
which exits a flat **1** — not the 4 REQ-022 requires. This is the
contract violation T-020 exists to fix: there was no `EngineError ->
ExitCode` mapping inspecting the error chain for the timeout variant.
</red>

<green>
`patina-cli/tests/exit_codes.rs::exclusive_lock_timeout_exits_4`: with the
lock held, the subprocess exits 4. `ExitCode::from_error_chain` walks the
`anyhow` chain, finds the wrapped
`EngineError::Lock(LockError::Timeout { .. })`, and maps it to
`ExitCode::LockTimeout (4)`.

```
test exclusive_lock_timeout_exits_4 ... ok
```

The `PATINA_LOCK_TIMEOUT_MS` knob (read by `patina_core::lock::exclusive_timeout`)
is a test-only affordance that shrinks the exclusive-lock cap so the
timeout is provoked in ~200 ms instead of the production minute; unset, it
falls back to `EXCLUSIVE_TIMEOUT`.
</green>

### Scenario 5 — successful apply exits 0

A clean apply that lands its single copy file exits 0.

<red>
Pre-edit: `ApplyResult::Applied` mapped to the literal `0`; T-020
re-expresses it as `ExitCode::Success`.
</red>

<green>
`patina-cli/tests/exit_codes.rs::successful_apply_exits_0`: exit 0 and
`~/.rc` contains the applied content.

```
test successful_apply_exits_0 ... ok
```
</green>

Note on exit 5 (declined prompt): REQ-022's exit-5 case needs an
interactive TTY, which a subprocess pipe cannot simulate. It is covered by
the in-process unit tests on the injected `PromptReader`
(`cmd::apply` declined path returns 5; `cmd::rollback::declined_prompt_exits_five`),
which assert the `ExitCode::UserDeclined (5)` return without a real tty.
