# Evidence — SPEC-0001 / T-012

Backups before overwrite, mirrored target paths, retention by count of ten
(REQ-014, REQ-015). The end-to-end `patina apply --yes` surface CHK-025 /
CHK-026 name does not exist yet (the `apply` subcommand, the executor loop
that calls the backup writer, and the commit→GC sequencing land in T-014 /
T-016). T-012 owns `patina_core::backups`; the tests drive
`backup_before_overwrite` and `gc_retain` directly and stage the on-disk
state each scenario describes.

## Red/green pairs

### Scenario 1 — CHK-025: backup-on-overwrite stashes the original bytes

Test: `overwriting_a_pre_existing_target_stashes_the_original_bytes` in
`patina-core/tests/backups.rs`.

- Stages a pre-existing `~/.zshrc` containing "original", then calls
  `backup_before_overwrite(backups, TS, zshrc)`.
- Green: the function returns `true`, and the backup at
  `mirror_backup_path(backups, TS, zshrc)` is a regular file holding the
  bytes "original", nested under `<backups>/<TS>/` and keeping the
  `.zshrc` file name. Asserting against `mirror_backup_path` (rather than a
  path the test recomputes) proves the writer and recovery's reader agree
  on the layout.
- Red check: if the writer skipped the copy or wrote the post-overwrite
  bytes, the `assert_eq!(read(backup), b"original")` fails; if it reported
  `false` for an existing target, the `assert!(made)` fails.

### Scenario 2 — REQ-014: a fresh target produces no backup entry

Test: `a_freshly_created_target_produces_no_backup_entry` in
`patina-core/tests/backups.rs`.

- A `~/.gitconfig` that does not pre-exist (the template-render target).
- Green: `backup_before_overwrite` returns `false` and no file exists at
  the mirrored backup path.

### Scenario 3 — REQ-014: the dotfiles repository is never written

Test: `backups_never_write_into_the_dotfiles_repository` in
`patina-core/tests/backups.rs`.

- Seeds a sibling `repo/` tree (source files + a nested dir), snapshots it
  byte-for-byte, backs up two pre-existing user targets under `home/`,
  then re-snapshots the repository.
- Green: the before/after repository snapshots are equal (no engine write
  reached the repo), while both backups did land under the state tree.

### Scenario 4 — CHK-026: retention prunes to exactly ten, keeping newest

Test: `a_successful_apply_prunes_to_exactly_ten_keeping_the_newest` in
`patina-core/tests/backup_retention.rs`.

- Seeds 15 timestamped cycle directories (each holding a backed-up file so
  removal must recurse), then calls `gc_retain(backups, RETENTION_COUNT)`.
- Green: the five oldest names are the removed set; exactly ten cycles
  survive and they are the ten most recent.
- Red check: a writer that kept the oldest, or pruned the wrong count,
  fails the `assert_eq!(removed, oldest_five)` / `surviving == newest_ten`
  lines.

### Scenario 5 — REQ-015: a failed apply triggers no GC

Test: `a_failed_apply_leaves_historical_backups_untouched` in
`patina-core/tests/backup_retention.rs`.

- Seeds 3 historical cycles plus a partial attempt directory, and models a
  failed apply by *not* calling `gc_retain` (the failure path never
  commits, so it never invokes retention).
- Green: all 3 historical cycles and the partial attempt directory survive
  untouched.

## Unit coverage (lib)

`patina-core/src/backups/mirror.rs` `#[cfg(test)]`: existing-target copy to
mirrored path, absent-target no-op, and a round-trip proving the backup is
readable from the exact path recovery looks at (`mirror_backup_path`).

`patina-core/src/backups/retention.rs` `#[cfg(test)]`: prune-oldest-keep-
newest (15→10), fewer-than-keep no-op, exactly-keep no-op, stray files are
not retention candidates (and survive), missing-backups-dir no-op.

## CLI surface — no `gc` subcommand

REQ-015 / the TASKS.md scenario require that no `gc` subcommand or `--gc`
flag exists. This is judgment-only at T-012: `patina_core::backups` exposes
retention purely as the library function `gc_retain`, with no CLI surface,
and the clap subcommand surface does not land until T-016. There is nothing
in this task's diff that could introduce a `gc` command; the guarantee is
that retention is implicit library housekeeping with no public command.
