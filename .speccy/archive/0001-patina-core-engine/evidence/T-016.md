# Evidence — SPEC-0001 / T-016

`patina apply` CLI surface with the TTY/non-TTY prompt decision, `--yes`,
`--force-deploy`, `--json`, `--pager`, and `-v` overrides (REQ-017). This
task lands the end-to-end apply pipeline: a new `patina_core::apply::engine`
orchestrator (`plan` resolves repo → modules → manifests → profile →
variables → canonical paths → durable `Plan`; `execute` recovers orphans,
locks, journals, runs `pre_apply` hooks, materializes with pre-overwrite
backups, runs `post_apply` hooks, then commits or rolls back), plus the
`patina-cli` surface — clap parser (`cli.rs`), the `output::Reporter`
abstraction (`output/reporter.rs`), the embedded `similar` diff renderer
(`output/diff.rs`), and the `apply` command (`cmd/apply.rs`) that owns the
TTY decision tree and the exit-code mapping (0 / 2 / 3 / 5).

All three runs below use the release-equivalent debug binary at
`target/debug/patina.exe`, each against an isolated tempdir repo + `$HOME`
+ state directory (`PATINA_REPO`, `HOME`/`USERPROFILE`, `XDG_STATE_HOME`,
and `LOCALAPPDATA` all redirected). Pre-edit, `patina apply` did not exist
as a subcommand at all — the binary's `main` awaited
`patina_core::status(..)` and the engine returned
`EngineError::NotImplemented("apply")` — so the pre-edit baseline for every
scenario is "the `apply` subcommand is unparseable / unimplemented".

## Scenario 1 — Non-TTY apply previews without mutating (CHK-028)

<red>
Pre-edit there is no `apply` subcommand: the binary parsed no clap surface
and `main` drove `patina_core::status`. `patina apply` would fail at
argument parsing (no such subcommand), and the engine's `apply` entry
point returned `EngineError::NotImplemented("apply")`. No diff was ever
rendered and the non-TTY preview path did not exist.
</red>

<green>
Post-edit: a module declaring one `symlink` `[[file]]` (`source = "rc"`,
`target = "~/.rc"`), invoked with stdin redirected from `/dev/null` (a
non-TTY) and no `--yes`:

```
$ PATINA_REPO=$REPO HOME=$HOME ... patina apply </dev/null
symlink .../home/.rc
  - (absent)
  + .../repo/shell/rc
$ echo $?
0
$ ls .../home/.rc
ls: cannot access '.../home/.rc': No such file or directory
```

Exit code 0, the rendered diff is on stdout naming the target `.rc`, and
no symlink was created — the non-TTY-without-`--yes` preview path of
REQ-017 / CHK-028. Reproduced as the integration test
`non_tty_apply_previews_without_mutating` in
`patina-cli/tests/apply_cli.rs`, which additionally asserts
`symlink_metadata` reports the target absent.
</green>

## Scenario 2 — post_apply must_succeed hook failure rolls back, exit 3 (CHK-029)

<red>
Pre-edit: no `apply` subcommand, no plan loop, no journal/backup wiring, no
rollback machinery. The rollback-on-`post_apply`-failure behaviour the
scenario names had no code path.
</red>

<green>
Post-edit: a module with a `copy` `[[file]]` plus a `[[hook]]`
(`event = "post_apply"`, `command = "exit 1"`, default
`must_succeed = true`), invoked with `--yes`:

```
$ PATINA_REPO=$REPO ... patina apply --yes </dev/null
copy .../home/.rc
  + payload
... (pwsh runs `exit 1`) ...
post_apply hook `exit 1` failed; rolled back all file operations
$ echo $?
3
$ test -e .../home/.rc && echo EXISTS || echo reversed
reversed
```

The file op executed first (the `copy` materialized `.rc`), the hook
returned non-zero, the engine reversed the copy using the pre-overwrite
backup machinery, the process exited 3, and `.rc` no longer exists.
Reproduced as `post_apply_hook_failure_rolls_back_and_exits_3`.
</green>

## Scenario 3 — `--force-deploy` downgrades the failure, exit 0 (CHK-030)

<red>
Pre-edit: no `apply` subcommand and no `--force-deploy` flag; the override
that flips every hook to `must_succeed = false` had no CLI surface.
</red>

<green>
Post-edit: the same module/hook as Scenario 2, invoked with
`--yes --force-deploy`:

```
$ PATINA_REPO=$REPO ... patina apply --yes --force-deploy </dev/null
copy .../home/.rc
  + payload
... (pwsh runs `exit 1`, stderr names the hook) ...
$ echo $?
0
$ cat .../home/.rc
payload
```

The file op executed, the hook failed but was degraded to a warning, the
file op was NOT rolled back (`.rc` still holds `payload`), and the process
exited 0. Reproduced as `force_deploy_downgrades_hook_failure_and_exits_0`,
which also asserts stderr names the failed hook (`exit 1`).
</green>

## Hygiene gates

root):

The CLI integration suite (`patina-cli/tests/apply_cli.rs`, 7 tests) also
covers the `--json` preview (`result: "previewed"`, no mutation), the
`--json --yes` apply (`result: "applied"`), the `-v email=…` override
flowing into a `.tmpl` render, and the `--pager=delta` PATH-miss fallback
warning. Each test isolates its state directory via `LOCALAPPDATA` /
`XDG_STATE_HOME` so the engine's startup `recover_orphans` pass cannot
race a sibling test's just-applied files (a real flake that surfaced and
was fixed during this task).
