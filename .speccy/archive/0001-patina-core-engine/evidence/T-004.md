## REQ-005 — `[[file]]` table-array schema

### CHK-010 — `$HOME/.zshrc` symlink target equals canonical `T/zsh/zshrc`

Judgment-only at this phase. CHK-010's scenario runs `patina apply
--yes` end-to-end and inspects `$HOME/.zshrc`'s readlink target. The
parser landed here validates that the `mode = "symlink"` declaration
parses to `FileMode::Symlink` and that the single `target = "~/.zshrc"`
becomes a one-element `targets` vec (covered by
`parses_single_target_explicit_symlink_mode` in
`patina-core/tests/config_file_entry.rs`). Actually performing the
symlink syscall belongs to the file-mode executor in T-014; the apply
pipeline that drives variable resolution + canonicalization + executor
dispatch is wired in T-006 → T-009 → T-014.

### CHK-011 — `gitconfig.tmpl` renders to `$HOME/.gitconfig` with variables resolved

Judgment-only at this phase. CHK-011 exercises the full template
pipeline — MiniJinja rendering (T-008) plus variable resolution (T-006)
plus the apply executor (T-014). T-004 covers the parser-side
prerequisite: a `.tmpl` source with no declared mode resolves to
`FileMode::TemplateRender`
(`tmpl_source_resolves_to_template_render_mode`), and a `.tmpl` source
that declares any explicit `mode` is rejected with a typed error
naming `.tmpl` and the `implicit-template` rule
(`rejects_tmpl_source_with_explicit_mode`).

### CHK-012 — `mode = "merge-json"` rejected, listing the four accepted values

Demonstrated by `rejects_merge_json_mode_listing_accepted_values` in
`patina-core/tests/config_file_entry.rs`. The test parses
`[[file]] mode = "merge-json" source = "x" target = "y"` via
`parse_module_config_str`, asserts the rendered error string contains
`merge-json`, `symlink`, `symlink-dir`, `copy`, and `copy-tree`, and
matches the error against
`ConfigParseError::FileEntry(FileEntryError::UnsupportedMode { .. })`.
The four accepted values are hard-coded in the `UnsupportedMode`
`Display` template at
`patina-core/src/config/file_entry.rs:73-79`.

### CHK-041 — Default mode `symlink` when `mode` field omitted

Demonstrated at the parser level by
`defaults_to_symlink_when_mode_omitted` in
`patina-core/tests/config_file_entry.rs`: a `[[file]]` with
`source = "zshrc"` and `target = "~/.zshrc"` (no `mode` key) parses
to `FileEntry { mode: FileMode::Symlink, .. }`. CHK-041's
end-to-end scenario (the resulting `$HOME/.zshrc` is a symlink whose
readlink target equals the canonical `zsh/zshrc`) lives in the apply
executor layer T-014; the parser correctness it depends on is
established here.

### CHK-042 — Multi-target `targets = [...]` with `mode = "symlink"`

Judgment-only at this phase for the end-to-end readlink check.
Parser-side: the table-array variant
`source = "agent.toml" targets = ["~/.claude/agent.toml", "~/.codex/agent.toml"]`
is exercised by `parses_targets_array_with_copy_mode` in
`patina-core/tests/config_file_entry.rs` (same shape with `mode =
"copy"`); the `targets` vector preserves declaration order
(`["~/a", "~/b"]`), the resulting `FileEntry.targets.len() == 2`, and
each entry is a `Utf8PathBuf` carrying the verbatim string. The
symlink-mode end-to-end materialization is T-014's responsibility.

### CHK-043 — Multi-target `targets = [...]` with `mode = "copy"`

Demonstrated at the parser level by
`parses_targets_array_with_copy_mode` in
`patina-core/tests/config_file_entry.rs`: the mode resolves to
`FileMode::Copy` and the targets vector equals `["~/a", "~/b"]`.
End-to-end byte-content equality belongs to T-014.

### CHK-044 — Multi-target with `.tmpl` source renders MiniJinja output to each target

Judgment-only at this phase. The end-to-end render belongs to T-008
(MiniJinja env) + T-014 (apply executor). T-004 covers the parser
prerequisite that a `.tmpl` source with multi-target declaration
resolves to `FileMode::TemplateRender` and is not rejected by the
implicit-template rule when no explicit `mode` is set
(implicit-template positive path covered by
`tmpl_source_resolves_to_template_render_mode`).

### CHK-045 — Declaring both `target` and `targets` is rejected

Demonstrated by `rejects_target_and_targets_both_set` in
`patina-core/tests/config_file_entry.rs`. The test declares both

`targets = ["~/.codex/agent.toml"]`, asserts the rendered error
contains `target`, `targets`, and `exactly one`, and matches the

`ConfigParseError::FileEntry(FileEntryError::TargetAndTargets)`. The
substring contract is hard-coded in the variant's `Display` template
at `patina-core/src/config/file_entry.rs:58`.

### CHK-046 — Declaring neither `target` nor `targets` is rejected

### CHK-047 — `targets = []` is rejected with `non-empty`
