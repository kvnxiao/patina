# Evidence for SPEC-0001 T-027

Red-then-green paper trail for the REQ-030 lock-acquisition policy:
a `LockPolicy` enum (`Blocking` default / `NonBlocking` / `Held`)
threaded into the engine apply path (`patina_core::execute_plan`),
plus a `try_acquire` primitive and a `LockError::Contended` variant
in `patina-core/src/lock.rs`.

The two `demonstrated` scenarios drive the engine `execute` entry
point in-process (the only layer that can pass a test-controlled
`LockGuard` for `Held` or observe the `NonBlocking` early return);
they live in the `lock_policy_tests` module in
`patina-core/src/apply/engine.rs`.

## Session 2026-05-30-T027-attempt-1

### Scenario 1 — NonBlocking apply on a contended lock errors and writes no journal (CHK-065)

<red>
Before this task, `execute` had no policy parameter and
unconditionally called `acquire_lock(.., Exclusive,
exclusive_timeout())`; there was no way to make a single
non-blocking attempt and no `LockError::Contended` variant to
match. A `NonBlocking` apply could not be expressed, so the test
`non_blocking_apply_on_contended_lock_errors_and_writes_no_journal`
did not compile against the pre-task API
(`LockPolicy` / `LockError::Contended` are undefined symbols).
</red>

<green>
Post-edit, with a test-controlled exclusive guard held at the
fixture's `<state>/patina/lock`, the test drives
`execute(&resolved, &ApplyRequest::default(), LockPolicy::NonBlocking)`
and asserts:

```
running 1 test
test apply::engine::lock_policy_tests::non_blocking_apply_on_contended_lock_errors_and_writes_no_journal ... ok
```

The call returns
`Err(EngineError::Lock(LockError::Contended { kind: LockKind::Exclusive, .. }))`,
and neither `<ts>.plan` nor `<ts>.COMMIT` exists in the journal
directory afterward — the early return on contention happens before
`Journal::flush_plan_and_fsync`, so the zero-mutation guarantee
holds.
</green>

### Scenario 2 — Held policy applies with the caller's guard and commits (CHK-066)

<red>
Before this task there was no `Held(LockGuard)` policy: an apply
always re-acquired the exclusive lock internally, so a caller that
already held it would deadlock against its own lock (or, with a
short cap, time out). The test
`held_policy_applies_with_callers_guard_and_commits` did not compile
against the pre-task API.
</red>

<green>
Post-edit, the test acquires the exclusive lock up front, then drives
`execute(&resolved, &ApplyRequest::default(), LockPolicy::Held(guard))`:

```
running 1 test
test apply::engine::lock_policy_tests::held_policy_applies_with_callers_guard_and_commits ... ok
```

The apply does not time out against its own held lock; it returns
`ApplyResult::Applied { .. }`, a `<ts>.COMMIT` record is present in
the journal directory, and the `<ts>.plan` file is gone (deleted
after the COMMIT sentinel is durable).
</green>
