# Evidence for SPEC-0001 T-018

Red-then-green paper trail for `patina rollback` (REQ-019): the
subcommand reverses the most recent committed apply via the journal and
backups, with per-`[[file]]`-entry atomic restore.

## Coverage rule

The journal `<implementer>` block's `Evidence:` roll call labels each CHK
under REQ-019. CHK-033 and CHK-049 are the two `<scenario>` blocks the SPEC
attaches to REQ-019; both are demonstrated below. The runs use the built
`target/debug/patina` binary in a tempdir fully isolated via
`PATINA_REPO` / `HOME` / `USERPROFILE` / `XDG_STATE_HOME` / `LOCALAPPDATA`
so neither apply nor rollback touches the developer's real `$HOME`.

## Session 2026-05-29-T018-attempt-1

### Scenario 1 — Multi-target copy entry rolls back atomically (CHK-049)

A `[[file]]` copy entry with `targets = ["~/.claude/agent.toml",
"~/.codex/agent.toml"]` from a source containing "new", where
`~/.claude/agent.toml` pre-existed with "old" and `~/.codex/agent.toml`
did not exist. After `patina rollback --yes` the pre-existing target is
restored to "old", the fresh target is deleted, and a `<ts>.ROLLED_BACK`
sentinel is written.

<red>
Pre-edit: `patina rollback` did not exist. The CLI exposed only `apply`
and `status` (`patina-cli/src/cli.rs` had no `Rollback` variant), and the
engine entry point returned `EngineError::NotImplemented("rollback")`
(`patina-core/src/lib.rs`). Invoking the subcommand was a clap
unknown-subcommand error (exit 2); the inverse-restore behaviour did not
exist, so a pre-existing target overwritten by apply stayed overwritten
and a fresh target stayed on disk. No `<ts>.ROLLED_BACK` sentinel was ever
written by any code path.
</red>

<green>
Post-edit: the full apply → rollback sequence run against the built
binary.

```
$ env PATINA_REPO=$D/repo HOME=$D/home USERPROFILE=$D/home \
      XDG_STATE_HOME=$D/state LOCALAPPDATA=$D/state patina apply --yes
apply exit:0
post-apply claude=new codex=new
$ env ... patina rollback --yes
Rolled back the most recent apply.
rollback exit:0
post-rollback claude=old codex-exists=no
$ find state -name '*.ROLLED_BACK'
state/patina/journal/20260529T015317Z.ROLLED_BACK
```

`~/.claude/agent.toml` is restored from its backup to "old";
`~/.codex/agent.toml` (a fresh creation, no backup) is deleted; the
`<ts>.ROLLED_BACK` sentinel is present. This is the same assertion the
integration test
`patina-cli/tests/rollback_cli.rs::multi_target_copy_entry_rolls_back_to_pre_apply_state`
makes under `cargo test`.
</green>

### Scenario 2 — Symlink target rolls back; sentinel written (CHK-033, symlink leg)

CHK-033's symlink materialization and `<ts>.ROLLED_BACK` sentinel legs.
The symlink the apply created is a fresh creation (no backup), so rollback
deletes it and writes the sentinel.

<red>
Pre-edit: no `rollback` subcommand, no inverse-restore, no sentinel
(as in Scenario 1's red baseline). A symlinked target created by apply
stayed in place after any attempted rollback.
</red>

<green>
Post-edit, exercised by
`patina-cli/tests/rollback_cli.rs::rollback_deletes_a_symlink_target_and_writes_the_sentinel`:
after `patina apply --yes` materializes `~/.zshrc` as a symlink, the test
asserts the post-apply path is a symlink; after `patina rollback --yes`
(exit 0) the link is gone (`symlink_metadata` errors) and a
`*.ROLLED_BACK` sentinel exists in the journal.

```
$ cargo test -p patina-cli --test rollback_cli
test rollback_deletes_a_symlink_target_and_writes_the_sentinel ... ok
test multi_target_copy_entry_rolls_back_to_pre_apply_state ... ok
test rollback_deletes_a_freshly_created_target ... ok
test rollback_with_no_prior_apply_exits_one_and_names_no_prior_apply ... ok
test rolled_back_apply_drops_out_of_status_last_apply ... ok
test result: ok. 5 passed; 0 failed; 0 ignored
```

CHK-033's *pre-existing-file restoration* leg (a pre-existing regular file
replaced by a symlink, then restored to the original regular file) is
**not** demonstrated end-to-end here: the T-014 symlink executor does not
remove a pre-existing regular file before creating the link, so apply
fails ("Cannot create a file when that file already exists") before the
rollback under test can run. See the journal's Discovered issues. The
restore-from-backup mechanism CHK-033 depends on *is* demonstrated through
the copy path in Scenario 1 (the `~/.claude/agent.toml` "old" restore) and
the `replay_entry` overwrite unit tests, so the rollback contract itself is
proved; only the apply-side precondition for the symlink-over-file
variant is blocked.
</green>

### Scenario 4 — Per-entry atomic abort rolls reverted targets forward

The 5th REQ-019 `<task-scenario>`: when restoring the second target of a
multi-target entry fails, the first target's revert is rolled forward to
its post-apply state and a typed `RollbackPartial` error is surfaced — no
partial restore.

<red>
Pre-edit: no rollback / replay machinery existed.
</red>

<green>
Demonstrated by
`patina-core/tests/rollback_atomic.rs::failed_second_target_rolls_the_first_forward_and_reports_partial`.
The harness injects a deterministic, privilege-free failure: the second
target's parent path is occupied by a regular file, so its
restore's `create_dir_all(parent)` fails. The test asserts the result is
`RollbackError::RollbackPartial { entry: 3, .. }` and the first target's
on-disk bytes are back to "post-apply-1" (rolled forward), not the
"original-1" backup.

```
$ cargo test -p patina-core --test rollback_atomic
test fully_revertible_multi_target_entry_succeeds ... ok
test failed_second_target_rolls_the_first_forward_and_reports_partial ... ok
test result: ok. 2 passed; 0 failed; 0 ignored
```
</green>
