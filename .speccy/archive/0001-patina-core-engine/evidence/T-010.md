# Evidence for SPEC-0001 T-010

Red-then-green paper trail for T-010: `patina_core::journal` — the
binary postcard plan file with a single up-front `fsync` plus a
non-fsync'd per-operation progress cursor (REQ-011, REQ-012).

## Coverage rule note

REQ-011 carries CHK-022; REQ-012 carries CHK-023. Both scenarios
phrase their setup as `patina apply --yes` killed/run end-to-end. That
full surface cannot run today: the `apply` subcommand, the executor
mutation loop, and the SIGKILL/`TerminateProcess` harness all land in
later tasks (T-014 the executor, T-016 the CLI, T-011 the crash-probe
recovery suite that owns the actual kill-and-reconcile harness). T-010
is the library task that *owns* the journal mechanism those scenarios
exercise. The scenarios below demonstrate the load-bearing journal
properties at the layer T-010 produces them, driving the
`patina_core::journal` module directly through a recording `Syncer`
fake (the very test hook the task body asks T-010 to provide for
T-011). Both CHKs are labelled `demonstrated` for the journal-level
property with the end-to-end `patina apply --yes` framing deferred to
the tasks that build that surface.

## Session 2026-05-28-T010-attempt-1

### Scenario 1 — Plan exists, durable, no COMMIT sentinel in the crash window (CHK-022)

<red>
Pre-edit (`git show HEAD:patina-core/src/lib.rs`): there is no
`journal` module at all. The plan file, the `.plan`/`.progress`/
`.COMMIT` filename scheme, the version envelope, and the single-fsync
flush point do not exist. Demonstrated by the absence of the module
and the entry point:

```
$ git show HEAD:patina-core/src/journal/mod.rs
fatal: path 'patina-core/src/journal/mod.rs' does not exist in 'HEAD'
$ git show HEAD:patina-core/src/lib.rs | grep -n "journal\|flush_plan"
(no matches)
```

With no journal, there is no `<ts>.plan` written before mutations and
no notion of a COMMIT sentinel — the CHK-022 post-flush filesystem
state cannot be produced or asserted against the pre-edit tree.
</red>

<green>
Post-edit: `Journal::flush_plan_and_fsync` writes and fsyncs the plan
before returning, and no COMMIT is written until `commit` is called.
Asserted by `after_flush_plan_exists_with_no_commit_sentinel` in
`patina-core/tests/journal_fsync_shape.rs`:

```
$ cargo test -p patina-core --test journal_fsync_shape
running 5 tests
test same_plan_encodes_to_identical_bytes ... ok
test newer_major_version_is_refused_with_both_versions_named ... ok
test after_flush_plan_exists_with_no_commit_sentinel ... ok
test three_op_apply_fsyncs_plan_dir_commit_but_never_progress ... ok
test commit_deletes_plan_and_progress_leaving_only_commit_sentinel ... ok

test result: ok. 5 passed; 0 failed; 0 ignored
```

`after_flush_plan_exists_with_no_commit_sentinel` flushes a one-op
plan, drops the handle (the crash window — flush returned but no
mutation, no progress, no commit), then asserts the journal dir holds
exactly one `<ts>.plan` file, no `<ts>.COMMIT`, and that the on-disk
plan bytes decode back to the identical `Plan` — proving the flush was
durable rather than a partial write. This is the CHK-022 filesystem
state ("exactly one `<ts>.plan` file and no `<ts>.COMMIT` sentinel")
at the moment after `flush_plan_and_fsync` returns.
</green>

### Scenario 2 — fsync shape: plan + dir + COMMIT, never per-op progress (CHK-023)

<red>
Pre-edit: no journal, no `Syncer` trait, no fsync call sites — there is
nothing to count. The REQ-012 invariant ("zero per-operation fsyncs on
the progress file") has no code that could satisfy or violate it
because no progress file is written.

```
$ git show HEAD:patina-core/src/journal/sync.rs
fatal: path 'patina-core/src/journal/sync.rs' does not exist in 'HEAD'
```
</red>

<green>
Post-edit: the `Syncer` trait abstracts the durability syscalls and a
recording fake counts them per path.
`three_op_apply_fsyncs_plan_dir_commit_but_never_progress` flushes a
three-operation plan, records three progress entries, commits, and
asserts the exact fsync shape:

- exactly one file-fsync on a path ending `.plan`,
- exactly one file-fsync on a path ending `.COMMIT`,
- zero file-fsyncs on any `.progress` path,
- exactly two directory fsyncs (after the plan flush, after COMMIT).

```
$ cargo test -p patina-core --test journal_fsync_shape three_op_apply_fsyncs_plan_dir_commit_but_never_progress
running 1 test
test three_op_apply_fsyncs_plan_dir_commit_but_never_progress ... ok
test result: ok. 1 passed; 0 failed
```

This is exactly CHK-023's claim ("one on the plan file, one on its
parent directory, one on the `COMMIT` sentinel, but no per-operation
fsyncs on the progress file") measured through the test hook the task
body specifies for T-011's reuse.
</green>

### Scenario 3 — Newer major version refused with a typed error naming both versions (task-body version-envelope scenario)

<red>
Pre-edit: no version envelope exists, so a poisoned `u16::MAX` prefix
has no meaning and no `JournalVersionMismatch`-style error type exists
to return.
</red>

<green>
Post-edit: `Plan::decode` reads the 2-byte little-endian envelope
before invoking postcard and refuses any major exceeding the compiled
`FILE_MAJOR_VERSION = 1`.
`newer_major_version_is_refused_with_both_versions_named` poisons the
envelope to `u16::MAX` and asserts a
`JournalError::VersionMismatch { found: u16::MAX, supported: 1 }` whose
`Display` names both numbers:

```
$ cargo test -p patina-core --test journal_fsync_shape newer_major_version_is_refused_with_both_versions_named
running 1 test
test newer_major_version_is_refused_with_both_versions_named ... ok
test result: ok. 1 passed; 0 failed
```

The rendered Display string is
`journal plan major version 65535 is newer than supported version 1; upgrade patina to read this plan`,
which contains both `65535` and `1`.
</green>

### Scenario 4 — COMMIT deletes plan and progress, leaving only the sentinel (task-body cleanup scenario)

<green>
`commit_deletes_plan_and_progress_leaving_only_commit_sentinel`
flushes, records progress, commits, then asserts the `.plan` and
`.progress` files are gone and only the `.COMMIT` sentinel remains; a
second flush writes a fresh `.plan` beside the surviving sentinel
without disturbing it. This is the REQ-011 `<done-when>` "deleted only
after a successful `COMMIT` sentinel is written and fsync'd" plus the
"no recovery for a committed timestamp" `<behavior>` at the journal
layer.

```
$ cargo test -p patina-core --test journal_fsync_shape commit_deletes_plan_and_progress_leaving_only_commit_sentinel
running 1 test
test commit_deletes_plan_and_progress_leaving_only_commit_sentinel ... ok
test result: ok. 1 passed; 0 failed
```

A `<red>` baseline is omitted for this scenario: the cleanup ordering
is a new mechanism with no pre-edit analogue (the journal module did
not exist), so the green assertion stands on its own as the proof that
the delete-after-COMMIT ordering holds.
</green>

## Cross-platform note (Windows correctness)

`OsSyncer::sync_file` opens the target with `OpenOptions::write(true)`
(no truncation) rather than read-only, because Windows
`FlushFileBuffers` requires a handle with write access and rejects a
read-only handle with `ERROR_ACCESS_DENIED` (code 5). This was caught
by the integration suite failing on the dev Windows host before the
fix; the fix makes the single-fsync durability point work on all three
v1 platforms. The directory fsync is best-effort on Windows (a
directory handle cannot be flushed there), which is sound because the
preceding file write is already durable via the file fsync.
