## Session 2026-05-28-T013-attempt-1

### Scenario 1 — Two exclusive holders serialize; windows are disjoint (CHK-037)

The contract: two processes that both request the **exclusive**
lock against the same state directory must run sequentially — the
second's acquire/release window opens only after the first's
closes. The red baseline shows the assertion has teeth by running
the same harness in **shared** mode (where overlap is permitted and
correct), proving the windows genuinely interleave when the lock
does not exclude; the green run flips the mode to exclusive and the
windows become disjoint.

<red>
Pre-condition probe: run the harness in `shared` mode, which is the
mode whose contract *permits* concurrent holders. Two shared
holders each hold for 300ms with a 5s acquire cap, started
back-to-back. Their windows overlap — this is what "interleaved"
looks like, and it is exactly the failure the CHK-037 assertion
(`first.released <= second.acquired`) must reject for the exclusive
case.

```
$ lock_helper <tmp> shared 300 5000 false   # process A
ACQUIRED 1779994766648718000
RELEASED 1779994766949408900
$ lock_helper <tmp> shared 300 5000 false   # process B
ACQUIRED 1779994766655210700
RELEASED 1779994766955756600
```

A acquired at ...648718000 and did not release until ...949408900;
B acquired at ...655210700 — *inside* A's window. The disjointness
predicate `A.released (…949408900) <= B.acquired (…655210700)` is
false, so against shared holders the CHK-037 assertion fails. This
is the red: nothing serialized the two processes, and the windows
interleave.
</red>

<green>
Post-edit: the integration test
`two_exclusive_applies_do_not_interleave` spawns the same two
processes in `exclusive` mode (300ms hold, 5s cap, started
back-to-back well within the 100ms window CHK-037 names). The
exclusive lock forces the second process to block on the first.

```
$ cargo test -p patina-core --test lock_concurrency \
    two_exclusive_applies_do_not_interleave
running 1 test
test two_exclusive_applies_do_not_interleave ... ok
test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured
```

The test asserts (a) exactly two `<nanos>.plan` markers exist in
the journal directory — the union of the two applies' plan files
numbers exactly two, as CHK-037 requires — and (b) the disjointness
predicate `first.released <= second.acquired` holds, i.e. whichever
process acquired first released before the other acquired. Under
the exclusive lock this predicate is satisfied; under the red
(shared) counterfactual above it is not. Same harness, same timing,
only the lock mode changed — so the green is attributable to the
lock's exclusion, not to scheduling luck.
</green>
