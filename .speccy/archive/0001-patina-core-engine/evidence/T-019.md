# Evidence ŌĆö SPEC-0001 / T-019

`patina debug journal <path>` decodes a `postcard`-encoded `<ts>.plan`
file into a human-readable view (REQ-020). This task lands:

- `patina_core::journal::render` ŌĆö `load_plan_file(path)` reads the file,
  runs the existing version-envelope check (`Plan::decode`, reused from
  T-010), and returns the decoded `Plan` plus the `<ts>` recovered from the
  filename; `render_plan(plan, ts)` formats a header (compact + RFC 3339
  timestamp, operation count) and one `mode / source / target` block per
  operation. A `PlanRenderError` distinguishes a read failure (names the
  path) from a decode failure (carries `JournalError`, including
  `VersionMismatch`). (`patina-core/src/journal/render.rs`, `journal/mod.rs`
  re-exports, `lib.rs` re-exports.)
- The `patina debug` clap namespace with a hidden-but-documented
  `journal <path>` member, routed through the `Reporter`; a decode failure
  maps to exit code 1 with the cause on stderr. (`patina-cli/src/cli.rs`,
  `cmd/debug.rs`, `cmd/mod.rs`, `main.rs`.)

The plan body records only the resolved file operations (symlink / render
/ copy, each `source` + `target`); hooks and the resolved variable context
are not serialized into the `Plan`, so they do not appear in the render ŌĆö
the module doc states this explicitly.

Pre-edit baseline for every scenario: `debug` was not a parseable
subcommand (the top-level `Command` enum had only `Apply` / `Status` /
`Rollback`), so `patina debug journal <path>` failed at argument parsing
and there was no plan-render path at all.

## Scenario 1 ŌĆö decode a plan, print its modes and a target (CHK-034)

<red>
Pre-edit: `patina debug journal <path>` was unparseable (no `debug`
subcommand), so no plan was ever decoded or rendered. There was no
`render_plan` / `load_plan_file` surface.
</red>

<green>
Post-edit: a plan declaring a `symlink` + `copy` operation, written to
`<ts>.plan` via the public `Plan::encode`, renders with the mode words
(`symlink` / `copy`) and an absolute target on stdout, exit 0. Reproduced
by `patina-cli/tests/debug_journal_cli.rs::decodes_a_plan_and_prints_its_modes_and_targets`
(drives the built `CARGO_BIN_EXE_patina` binary against a real tempdir
plan file) and the lib unit `journal::render::tests::render_names_each_mode_source_and_target`.
</green>

## Scenario 2 ŌĆö missing path exits 1 and names the path (REQ-020 done-when)

<red>
Pre-edit: no `debug journal` path existed to surface a missing-file error.
</red>

<green>
Post-edit, against the built binary:

```
$ patina debug journal <tmp>/missing.plan
cannot read plan file `<tmp>/missing.plan`: failed to open file `<tmp>/missing.plan`: The system cannot find the file specified. (os error 2)
exit=1
```

The path is named on stderr and the process exits 1. Reproduced by
`debug_journal_cli.rs::missing_path_exits_one_and_names_the_path_on_stderr`
and the lib unit `render::tests::load_missing_path_is_a_read_error_naming_the_path`.
</green>

## Scenario 3 ŌĆö newer version envelope exits 1, names both versions (REQ-020 done-when)

<red>
Pre-edit: no decode path; the version-envelope refusal was never surfaced
to a user through a CLI.
</red>

<green>
Post-edit: a plan whose envelope `u16` is overwritten with `u16::MAX`
(65535), decoded by a binary whose supported major is `1`, exits 1 with a
stderr line naming both `65535` and `1` plus the word `version` (the
`JournalError::VersionMismatch` Display). Reproduced by
`debug_journal_cli.rs::newer_version_envelope_exits_one_and_names_both_versions`,
the CLI unit `cmd::debug::tests::version_mismatch_exits_one_and_names_both_versions`,
and the lib unit `render::tests::load_newer_major_is_a_decode_version_mismatch`.
</green>

## Scenario 4 ŌĆö `patina debug --help` names `journal`, exits 0 (CHK-050)

<red>
Pre-edit: `patina debug --help` was unparseable (no `debug` group).
</red>

<green>
Post-edit, against the built binary:

```
$ patina debug --help
...
Commands:
  journal  Decode a `<ts>.plan` journal file into a human-readable view
  help     Print this message or the help of the given subcommand(s)
exit=0
```

`journal` is named as a subcommand and the process exits 0. The `debug`
group is hidden from the top-level `patina --help` Commands list (verified:
top-level help lists only `apply` / `status` / `rollback` / `help`) yet is
documented and parseable. Reproduced by
`debug_journal_cli.rs::debug_help_names_journal_subcommand`.
</green>
