# Evidence — SPEC-0001 / T-011

Crash recovery converges backward to pre-apply state via filesystem probe
(REQ-013). The end-to-end `patina apply --yes` surface CHK-024 names does
not exist yet (the `apply` subcommand, executor loop, and backup writer
land in T-012 / T-014 / T-016). T-011 owns `patina_core::journal::recovery`
and `::probe`; the tests drive `recover_orphans` directly and stage the
on-disk crash state the scenario describes.

## Red/green pairs

### Scenario 1 — CHK-024: N-of-M completed ops reversed from backups

Test: `restores_overwritten_targets_and_clears_orphan_files` in
`patina-core/tests/recovery_crash.rs`.

- Stages an orphan plan (no `<ts>.COMMIT`) of 5 copy ops: 3 overwrote
  pre-existing targets (backup of each original stashed under the
  per-apply backup root via the REQ-014 mirror layout, target left with
  the overwriting bytes), 1 fresh creation that completed (no backup,
  target present), 1 fresh op that never started (no backup, no target).
  Progress cursor claims `[0,1,2]` (3 of 5).
- Green: after `recover_orphans`, targets a/b/c read back their pre-apply
  bytes (`orig-a/b/c`), the completed fresh target `d` is deleted
  (converges to pre-apply = absent), and the orphan `.plan` + `.progress`
  files are gone. `report.recovered_timestamps()` is `[TS]`.
- Red check: before the fix the reversal logic must restore-from-backup
  (not delete) overwritten targets; deleting `a/b/c` or leaving the new
  bytes would fail the `assert_eq!(got, original)` lines.

### Scenario 2 — interrupted before any op executed

Test: `interrupted_before_any_op_touches_nothing_and_clears_orphan`.

- Orphan plan of 2 fresh-unstarted ops, empty progress.
- Green: no target is created, both orphan files removed, filesystem
  unchanged.

### Scenario 3 — idempotence

Test: `recovery_is_idempotent`.

- Runs `recover_orphans` twice. First pass reverses (restores `a`, deletes
  `b`); second pass finds no orphan (plan deleted) and is a no-op. Asserts
  the restored content and the deleted target are byte-stable across both
  passes.

### Scenario 4 — backward-only, never forward

Test: `recovery_rolls_back_and_never_completes_an_unstarted_op`.

- An un-started fresh op's target stays absent after recovery — recovery
  rolls back, it does not finish the half-done apply by creating the
  target.

### Behavior — probe over a lying cursor

Test: `lying_progress_cursor_is_ignored_in_favour_of_the_filesystem`.

- Cursor claims `[0,1]` completed but only op 0 actually overwrote a
  target. Recovery restores op 0 from its backup and does not invent a
  phantom restore/delete for op 1, because it probes the filesystem and
  consults the backup directory rather than trusting the cursor (REQ-012).
