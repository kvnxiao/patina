### Blocker 1 — infallible step expressed as fallible (`rust-let-types-carry-invariants.md`)

Round-1 `decode_envelope` re-read the body with a fallible
`bytes.get(ENVELOPE_LEN..).ok_or(Truncated…)` after
`read_envelope_version` had already proven the buffer was long enough — a
dead failure arm. The first remediation (`&bytes[ENVELOPE_LEN..]`) traded
the dead arm for a `clippy::indexing_slicing` violation (the project lints
panic-prone indexing). Final shape: a private `split_envelope` helper that
calls `bytes.split_first_chunk::<ENVELOPE_LEN>()` once, destructuring the
fixed-size version array and the body tail together. Both public functions
read through it, so the body slice now falls out of the same bounds check —
no second lookup, no panic path, no fallible-looking arm.

<green>
```
$ just lint
LINT EXIT: 0          # fmt, clippy -D warnings (indexing_slicing clean), doc -D warnings, deny
advisories ok, bans ok, licenses ok, sources ok
```
The same `truncated_buffer_is_rejected_before_decode` unit test still
asserts the one genuine failure arm (a buffer shorter than the envelope),
so the real Truncated path remains covered.
</green>

### Blocker 2 — vacuous constant-copy test dropped

Round-1 `tests/version_envelope.rs::envelope_sits_at_offset_zero_as_little_endian_u16`
copied the production constant `FILE_MAJOR_VERSION` into both the frame and
the assertion, so it could only ever prove `u16::to_le_bytes` round-trips
with itself. It is deleted. The offset-zero / little-endian contract is
still gated structurally and with literals: the unit test
`envelope_is_the_literal_leading_le_u16` frames major `7` and asserts the
leading bytes equal `7u16.to_le_bytes()` (a literal, not a production
constant), and Scenario 1's round-trip plus Scenario 2's version-mismatch
both depend on the offset-zero layout.

`tests/version_envelope.rs` now runs 2 integration tests (Scenario 1

</green>
