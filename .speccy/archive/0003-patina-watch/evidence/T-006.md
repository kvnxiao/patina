# Evidence for SPEC-0003 T-006

Red-then-green paper trail for `<state>/patina/logs/` and the watcher's
`tracing-appender` rotating-log stack (REQ-009 / DEC-009). This task
lands the logging substrate: a new `tracing-appender` dependency
(cleared through `cargo deny`), `patina-core/src/watch/logging.rs`
exposing `build_file_appender`, and a `state_dir.rs` module-doc note
that `logs/` is created by the watcher, not by `resolve`. The watcher
event loop that *emits* the `re_apply` events CHK-014 inspects lands in
T-008; this task proves only that the sink exists, is created lazily,
and that `resolve` does not create it.

The three `<task-scenarios>` in TASKS.md are the contract.

## Session 2026-05-31-T006-attempt-1

### Scenario 1 — `resolve()` creates `journal/` and `backups/` but NOT `logs/` (task-scenario 1)

Proves `state_dir::resolve_with_env` materializes `journal/` and
`backups/` while leaving `logs/` absent (the watcher owns it).

<red>
Pre-edit: `patina-core/src/watch/` contained only `drift_cache.rs` and
`mod.rs`; there was no `logging` module and no notion of a `logs/`
directory anywhere in the crate. A test asserting "`logs/` does not
exist after `resolve`" could not even be written against a public
surface, because nothing in the crate referenced `logs/`:

```
$ rg -n 'logs' patina-core/src
$ echo $?
1   # no matches — the logs/ contract did not exist
```
</red>

<green>
Post-edit: `resolve_creates_journal_and_backups_but_not_logs`
(`patina-core/tests/watch_logging.rs`) runs `resolve_with_env` against a
tempdir and asserts `journal/` and `backups/` are dirs while
`!root.join("logs").exists()`. `resolve` is unchanged (it never created
`logs/`); the test pins that property as a regression guard now that the
watcher owns the directory.

```
$ cargo test -p patina-core --test watch_logging
test resolve_creates_journal_and_backups_but_not_logs ... ok
```
</green>

### Scenario 2 — first init creates `logs/` and a daily-rotating appender writes a file under it (task-scenario 2)

Proves `build_file_appender` lazily creates `<state>/logs/` on first
call and that a line written through the returned non-blocking writer
lands in a rotating file under it.

<red>
Pre-edit: `build_file_appender` did not exist; the symbol failed to
resolve, so a test exercising it would not compile:

```
$ cargo test -p patina-core --test watch_logging 2>&1 | head
error[E0425]: cannot find function `build_file_appender` in crate `patina_core`
```

That unresolved-symbol compile failure is the red state.
</red>

<green>
Post-edit: `build_file_appender_creates_logs_dir_and_writes_a_rotating_file`
(integration) asserts `logs/` is absent before the call, present as a
dir after, then writes `"re_apply id=1"` through `appender.writer`,
drops the appender to flush the non-blocking worker, and asserts exactly
one file exists whose name starts with the `watch.log` prefix and whose
contents contain the written line. The lib unit test
`watch::logging::tests::creates_logs_dir_and_writes_a_line_to_a_file_under_it`
asserts the same against the private constants.

```
$ cargo test -p patina-core --test watch_logging
test build_file_appender_creates_logs_dir_and_writes_a_rotating_file ... ok
$ cargo test -p patina-core --lib watch::logging
test watch::logging::tests::creates_logs_dir_and_writes_a_line_to_a_file_under_it ... ok
test watch::logging::tests::build_is_idempotent_when_logs_dir_already_exists ... ok
```
</green>

### CHK-014 status (deferred to T-008)

CHK-014 — three watched source files modified in succession produce
three info-level `re_apply` events on the foreground watcher's stderr,
each with a distinct `re_apply.id` — requires the watcher event loop,
re-apply dispatch, and the stderr layer, none of which exist until

events will be written into. CHK-014 is therefore not demonstrable by
this task; it is carried in the journal roll call as deferred, to be
demonstrated by T-008's integration test.
