# Evidence for SPEC-0003 T-011

Red-then-green paper trail for T-011: the per-OS `ServiceBackend`
abstraction, the cross-platform `patina watch` lifecycle CLI, and the
macOS `launchd` backend.

A note on scope of demonstration. Only the genuinely host-coupled
happy paths — `launchctl bootstrap` actually loading a LaunchAgent
(CHK-001's registration half) and `launchctl print` of a
live-but-stopped service (CHK-006's live-query half) — are **not**
driven end-to-end from a test: registering a live per-user service
escapes the fixture's tempdir isolation (the launchd GUI domain is the
developer's real session) and is unsafe in CI. Everything that is pure
filesystem or string work is demonstrated below: the plist bytes, the
plist's `0644` file mode (a tempdir-scoped write, no launchctl needed),
and the `launchctl print` parse, all over unit-level red/green pairs;
and the supervisor-free CLI surface (not-installed no-ops, the `status`
object shape) over the integration binary.

### Scenario 2 — The macOS LaunchAgent plist carries the REQ-001 keys (CHK-001 internals)

<red>
Pre-edit: there was no `render_plist` function and no
`patina-core/src/watch/service/` module — the watcher had no service
descriptor surface. A test referencing `render_plist` failed to
compile (`cannot find function render_plist in this scope`).
</red>

<green>
Post-edit: `render_plist` emits a well-formed plist declaring the
service label, `RunAtLoad = true`, the `KeepAlive` on-failure restart
(`SuccessfulExit = false`), and `ProgramArguments` of the canonical
binary plus `watch --foreground`. Proven by the `launchd` unit tests:

```
$ cargo test -p patina-core --lib watch::service::launchd
test ...render_plist_contains_the_required_keys_and_program_arguments ... ok
test ...render_plist_is_well_formed_xml_with_a_single_program_arguments_array ... ok
test ...xml_escape_escapes_the_five_special_characters ... ok
test ...write_plist_sets_the_file_mode_to_0644 ... ok
```

`render_plist_contains_the_required_keys_and_program_arguments` asserts
each REQ-001 key is present; the well-formed-XML test asserts exactly
four `<string>` elements (Label + binary + the two foreground tokens).
`write_plist_sets_the_file_mode_to_0644` writes the plist to a tempdir
path through `write_plist` (which applies `set_mode_0644`) and asserts
the resulting file's permission bits are exactly `0o644` — CHK-001's
mode clause, demonstrated. Mutating `Permissions::from_mode(0o644)` →
`from_mode(0o600)` flips this test red (verified), so a silent
permission regression is now caught. Only the real `launchctl bootstrap`
registration is not exercised against a live supervisor here — see the
scope note above.
</green>
