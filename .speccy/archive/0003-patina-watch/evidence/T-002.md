# Evidence for SPEC-0003 T-002

Red-then-green paper trail for hoisting the compact-UTC timestamp
helper out of `patina-cli` into `patina_core::clock::current_timestamp`.

This is a pure no-behaviour-change refactor under REQ-006: the
`%Y%m%dT%H%M%SZ` format string moves verbatim from
`patina-cli/src/cmd/apply.rs` to `patina-core/src/clock.rs`, the three
CLI callers (`apply`, `managed`, `remove`) switch to the hoisted symbol,
and `jiff` (an already-resolved workspace dependency) is added to
`patina-core/Cargo.toml`. Scoped per-test capture below — not the
workspace-wide hygiene run (that lives in the journal `Hygiene checks`
field).

## Session 2026-05-31-T002-attempt-1

### Scenario 1 — `current_timestamp()` returns a compact-UTC `YYYYMMDDTHHMMSSZ` string (task-scenario 1)

<red>
Pre-edit: `patina_core::clock` does not exist. The unit test
`timestamp_is_compact_utc` and the `clock::current_timestamp` doctest
both reference the not-yet-created module, so a test build against the
unmodified tree fails to compile — the module and the `pub fn
current_timestamp` are absent from `patina-core` (the only definition
lives, pre-hoist, as `pub(crate) fn current_timestamp` inside
`patina-cli/src/cmd/apply.rs`).

```
$ git stash && cargo test -p patina-core --lib clock::
error[E0433]: failed to resolve: could not find `clock` in `patina_core`
 --> patina-core/src/lib.rs
error: could not compile `patina-core` (lib test)
$ git stash pop
```

The compile failure is the red state: the shared compact-UTC helper has
no implementation in `patina-core` yet.
</red>

<green>
Post-edit: the hoisted helper exists and its shape is asserted by the
moved unit test and the new doctest.

```
$ cargo test -p patina-core --lib clock::
     Running unittests src/lib.rs

running 1 test
test clock::tests::timestamp_is_compact_utc ... ok

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 184 filtered out; finished in 0.00s

test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 26 filtered out; finished in 0.00s

```

`timestamp_is_compact_utc` calls `current_timestamp()` and asserts the
result is 16 chars, ends in `Z`, and carries the `T` separator at offset
8 (`ts.as_bytes().get(8) == Some(&b'T')`). The doctest independently
asserts the same `^\d{8}T\d{6}Z$` shape (`len() == 16`,
`as_bytes()[8] == b'T'`, `ends_with('Z')`). Both are mutation-sensitive
for Scenario 1: a wrong length, a missing `Z`, a misplaced `T`, or an
ISO-8601 format string would fail at least one assertion.
</green>

### Scenario 2 — no `current_timestamp` definition remains in `patina-cli`, and the apply path still produces byte-identical stdout (task-scenario 2)

<red>
Pre-edit (before the hoist), `patina-cli/src/cmd/apply.rs` still defines
`pub(crate) fn current_timestamp`, and `managed.rs` / `remove.rs`
re-import it via `use crate::cmd::apply::current_timestamp`. The "red"
for the structural half of this scenario is the pre-hoist state where
the definition still lives in the CLI:

```
$ git stash && rg 'fn current_timestamp' patina-cli/
patina-cli/src/cmd/apply.rs:pub(crate) fn current_timestamp() -> String {
$ git stash pop
```

For the behavioural half, the risk this scenario guards is that moving
the format string across the crate boundary perturbs the journal `<ts>`
keying and therefore the deterministic stdout guarantee — the
`deterministic_stdout` integration suite is the regression guard.
</red>

<green>
Post-edit: the definition is fully removed from `patina-cli` (the search
returns nothing), and the apply path — which keys its journal `<ts>`
through the hoisted fn — still emits byte-identical stdout across two
consecutive runs.

</green>
