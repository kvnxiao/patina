# Evidence for SPEC-0003 T-005

Red-then-green paper trail for the `patina debug drift-cache <path>`
CLI subcommand (REQ-007, CHK-018). The subcommand decodes a binary
`drift.cache` via `patina_core::load_drift_cache_file` and renders it
via `patina_core::render_drift_cache`, routed through the `Reporter`,
parallel to the shipped `patina debug journal <path>`
(`DebugCommand::Journal` / `run_journal`).

This task lands only the CLI surface over T-004's format/decode/render
functions: a `DebugCommand::DriftCache(DebugDriftCacheArgs)` variant
(`patina-cli/src/cli.rs`), a `DriftCache` arm + `run_drift_cache`
dispatch (`patina-cli/src/cmd/debug.rs`), with exit-code and error
mapping parallel to `run_journal`. The three `<task-scenarios>` in
TASKS.md are the contract.

## Session 2026-05-31-T005-attempt-1

### Scenario 1 — Decode a populated cache, exit 0 (task-scenario 1, CHK-018)

Proves `patina debug drift-cache <path>` over a populated cache prints
`version:`, the bound journal timestamp, the target path, and both
hex hash values, and exits 0.

<red>
Pre-edit: `DebugCommand` has only a `Journal` variant; there is no
`drift-cache` subcommand. Pointing the CLI at a cache file is rejected
by the argument parser before any decode runs:

```
$ cargo build -p patina-cli && \
    ./target/debug/patina debug drift-cache /tmp/drift.cache
error: unrecognized subcommand 'drift-cache'
$ echo $?   # clap usage error
2
```

The unknown-subcommand exit (2, not the contracted 0) is the red
state: the rendering path does not exist yet.
</red>

<green>
Post-edit: `decodes_a_drift_cache_and_prints_version_timestamp_path_and_hashes`
(`patina-cli/tests/debug_drift_cache_cli.rs`) spawns the real binary
against a cache written through `DriftCache::encode` and asserts exit 0
plus the `version:`, `20260528T120000Z`, `.gitconfig`, and both
32-byte hex-repeat substrings on stdout. The unit test
`renders_a_valid_drift_cache_to_stdout_and_exits_zero`
(`cmd/debug.rs`) drives `run_drift_cache` against a `BufferReporter`
and asserts the same substrings + empty stderr.

```
$ cargo test -p patina-cli --test debug_drift_cache_cli
test decodes_a_drift_cache_and_prints_version_timestamp_path_and_hashes ... ok
$ cargo test -p patina-cli drift_cache   # unit
test cmd::debug::tests::renders_a_valid_drift_cache_to_stdout_and_exits_zero ... ok
```
</green>

### Scenario 2 — Missing path, exit 1, names the path (task-scenario 2)

Proves a non-existent path exits 1 with a typed error naming the path.

<red>
Pre-edit: no subcommand exists, so the same `error: unrecognized
subcommand 'drift-cache'` / exit 2 blocks the scenario. Separately, a
probe confirmed the underlying `DriftCacheError::Filesystem`
(`#[from] std::io::Error`) Display does **not** carry the path —
`load_drift_cache_file("/no/such/drift.cache")` renders as
`drift-cache filesystem operation failed` with no path — so a naive
`reporter.warn(&err.to_string())` (the `run_journal` shape) would fail
the "names the path" assertion:

```
$ # transient probe test in patina-core
DISPLAY=[drift-cache filesystem operation failed]
thread '...' panicked: got: drift-cache filesystem operation failed
```

That panic is the red state for the path-naming contract.
</red>

<green>
Post-edit: `run_drift_cache` warns `"{path}: {err}"`, prefixing the
caller-held `args.path` at the control-flow layer (the layer that
knows the path) so the contract holds without touching the locked
T-004 format module. `missing_path_exits_one_and_names_the_path_on_stderr`
(integration) and `missing_drift_cache_path_exits_one_and_names_the_path`
(unit) assert exit 1, the path on stderr, and nothing on stdout.

```
$ cargo test -p patina-cli --test debug_drift_cache_cli
test missing_path_exits_one_and_names_the_path_on_stderr ... ok
$ cargo test -p patina-cli drift_cache
test cmd::debug::tests::missing_drift_cache_path_exits_one_and_names_the_path ... ok
```
</green>

### Scenario 3 — Newer-major cache refused, exit 1, names both versions (task-scenario 3)

Proves a cache whose envelope major exceeds the running binary's is
refused with exit 1 and a typed error naming both the found and
supported versions.

<red>
Pre-edit: the absent subcommand again yields `error: unrecognized
subcommand 'drift-cache'` / exit 2, never reaching the version check.
</red>

<green>
Post-edit: the test encodes an empty cache, overwrites the leading
2-byte LE envelope with `u16::MAX`, writes the bytes, and runs the
subcommand. `DriftCache::decode` refuses via
`DriftCacheError::VersionMismatch { found: 65535, supported: 1 }`,
which `run_drift_cache` warns verbatim (its Display names both majors
and the word "version"). `newer_version_envelope_exits_one_and_names_both_versions`
(integration) and `drift_cache_version_mismatch_exits_one_and_names_both_versions`
(unit) assert exit 1, `65535`, `DRIFT_CACHE_MAJOR_VERSION` (= 1), and
the lower-cased substring `version` on stderr.

```
$ cargo test -p patina-cli --test debug_drift_cache_cli
test newer_version_envelope_exits_one_and_names_both_versions ... ok
$ cargo test -p patina-cli drift_cache
test cmd::debug::tests::drift_cache_version_mismatch_exits_one_and_names_both_versions ... ok
```
</green>

### Supporting — `debug --help` names the subcommand

`debug_help_names_drift_cache_subcommand` asserts `patina debug --help`
exits 0 and lists `drift-cache`, mirroring the journal test's
`debug_help_names_journal_subcommand`. This pins the variant is wired
into the (hidden) `debug` group's help, not just dispatch.
