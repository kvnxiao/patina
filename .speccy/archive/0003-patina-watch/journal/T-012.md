<review persona="security" verdict="blocking" model="claude-opus-4-8[1m]/high" date="2026-06-01T03:43:04Z" round="1">
Unit-file injection: `render_unit` interpolates the binary path raw into the systemd `ExecStart=` line with no escaping or quoting, while the directly-analogous launchd sibling (T-011, already shipped) deliberately runs `xml_escape(token)` over the same path before writing it into the descriptor. (`patina-core/src/watch/service/systemd.rs` ExecStart interpolation vs `launchd.rs` xml_escape.)

systemd `ExecStart` is line-oriented and applies its own word-splitting and `%`-specifier expansion, so an unescaped path is strictly more dangerous than the launchd XML case:
1. Newline injection — a `\n` anywhere in the canonicalized path terminates `ExecStart=` and the following line is parsed as a fresh unit directive (e.g. an attacker-chosen `ExecStartPost=`). The path comes from `current_exe().canonicalize()`, so it is not externally-supplied argument data and the realistic exploit presupposes control of the binary location — recorded as a descriptor-hardening blocker rather than a privilege-escalation one — but "interpolate a path into a config format without neutralizing the format's metacharacters" is exactly the defect the sibling task escaped, and the bar it set should hold here.
2. `%` specifier expansion — an unescaped `%` in a legitimate install path is expanded by systemd (`%h`, `%t`, …) instead of taken literally, silently corrupting the ExecStart. Mitigation: double `%` to `%%` (or run `systemd-escape`).
3. Whitespace — a path containing a space is word-split by systemd into binary + spurious args, breaking a legitimate space-containing install prefix. Mitigation: quote the ExecStart value per systemd's quoting rules.

Required change: render the `ExecStart` value through systemd's escaping/quoting (quote the binary token and double `%`), mirroring how `launchd.rs` escapes the path for its descriptor format, and add a `render_unit` test exercising a path containing a space and a `%` (and ideally a newline) so the neutralization is gated.

Non-blocking observations, all clean: no shell anywhere — every `systemctl` call is `Command::new("systemctl")` with static-`&str` argv arrays, no `sh -c`, no user data in argv; no admin/sudo path (per-user `--user` unit); `status` is a read-only `systemctl --user show` query; `uninstall` is `--yes`-gated at the CLI; `HOME`/`XDG_CONFIG_HOME` reads only steer the write location under the user's own tree; no new dependencies, no secrets, no sensitive data logged.
