<review persona="correctness" verdict="blocking" model="claude-opus-4-8[1m]/high" date="2026-05-31T23:20:24Z" round="1">
Lost-log race from a thread-local subscriber on a multi-threaded runtime ‚Äî Critical, breaks CHK-008 (and the future CHK-007/CHK-010 dispatch point) intermittently.

`patina-core/src/watch/mod.rs:179` installs the watcher's `tracing` subscriber with `tracing::subscriber::set_default(subscriber)`, which is **thread-local**: it applies only to the thread that called it, for the life of the returned guard. But `main.rs:37` uses bare `#[tokio::main]`, i.e. the multi-threaded (work-stealing) runtime, and `cmd::watch::run` -> `run_foreground` -> `run_foreground_in` runs as the main task directly on that runtime.

The `watch_started` event at `mod.rs:190` is emitted synchronously before the first `.await`, so it lands on the install thread (which is why the startup-substring tests pass). But the two events the SPEC turns into acceptance criteria are emitted *after* an await inside the `tokio::select!` loop: `tracing::info!(... "shutdown")` at `mod.rs:200` and `tracing::info!(... "watch_event")` at `mod.rs:209`. The reviewer's claim is that after either await, the work-stealing scheduler may resume the task on a different worker thread where no thread-local default subscriber is installed; the event then falls through to the global no-op subscriber and is silently dropped. CHK-008 requires the `shutdown` substring on stderr within 1s of SIGINT/SIGTERM, and CHK-007/CHK-010 hang the re-apply log-event count (T-009) off this same post-await dispatch point. Fix options: pin the foreground loop to a current-thread context (`current_thread` runtime / `LocalSet`), or install process-globally (`set_global_default` / `subscriber.init()`) with a guard so tests don't double-install.
</review>
