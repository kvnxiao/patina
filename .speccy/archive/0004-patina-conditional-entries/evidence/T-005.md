# Evidence for SPEC-0004 T-005

Paper trail for T-005: gating a managed entry's plan presence on its `when`
predicate, evaluated before the source is canonicalized (REQ-003 / REQ-009).
T-001 already parses and carries `ManagedEntry.when`; T-002 established the
files-then-directories order and the single monotonic `entry_index` space.
This task inserts the `when` gate at the top of each entry's per-entry body
so the fixed REQ-009 order holds: (1) evaluate `when`, (2) canonicalize, and
(3) [kind/existence check — T-006]. A `when`-false entry is dropped before
canonicalization and emits no operation and no diff line (DEC-004), while
still occupying its `entry_index` (REQ-009).

## Production change

`patina-core/src/apply/engine.rs`:

- `plan()` constructs one `Engine` (`crate::template::Engine::new()`) — the
  same evaluator the hook `when` and template renders use (REQ-004 /
  DEC-006) — and threads it plus the planning `Resolver` into the new
  `gate_and_resolve_entry`.
- `gate_and_resolve_entry(entry, module_path, home, engine, resolver)`
  evaluates `entry.when` via `engine.eval_when(expr, resolver)?` **first**;
  on `false` it returns `Ok(None)` without ever calling `resolve_entry`
  (so the source is never canonicalized), and on true / no-`when` it returns
  `Ok(Some(resolve_entry(...)?))`. `eval_when`'s `TemplateError` threads up
  through the existing `EngineError::Template(#[from])` variant, so a
  `when` that accesses an undefined variable already fails the plan (the
  T-009 REQ-004 uniformity claim rides on this same path).
- The two declared-order buckets become `Vec<Option<ResolvedEntry>>`. A
  `None` slot keeps the entry's position in the declared sequence.
- `assemble_plan_operations` advances the `u32` `entry_index` for **every**
  declared slot (including `None`), but emits a `PlannedOperation` /
  `ResolvedOperation` only for `Some`. So a `when`-false entry occupies its
  index but contributes nothing — the survivors keep the indices they would
  have had if the gated-off entry were present-but-empty (REQ-009), rather
  than being renumbered.

The gate is per-entry, above the target loop in `resolve_entry`, so for a
multi-target entry the `when` gates all targets together (REQ-003).

## Session 2026-06-01-T005-attempt-1

### Unit test — index space preserved across a `when`-false slot

`patina-core/src/apply/engine.rs::tests::when_false_entry_occupies_its_index_but_emits_no_operation`
drives the real `assemble_plan_operations` with a declared file sequence
`[Some(f0), None, Some(f2)]` and one directory `[Some(d0)]`. It asserts the
emitted operations are exactly `f0, f2, d0` (the `None` emits nothing) and
that the resolved-op indices are `[0, 2, 3]` — the gated-off slot consumed
index 1, leaving a gap rather than renumbering the survivors.

Mutation check: compacting the indices (incrementing only on `Some`) yields
`[0, 1, 2]` and fails the assertion; emitting an op for the `None` slot
fails the source-order assertion.

### CHK-006 — A `when`-false entry creates no target and plans zero operations

`patina-cli/tests/conditional_entries.rs::when_false_entry_creates_no_target_and_plans_zero_operations`
drives the real `patina apply --json --yes` over a fixture whose `[[file]]`
entry carries `when = "patina.os == 'definitely-not-this-os'"`. It asserts
the apply exits 0, the `~/.zshrc` target is **not** created, and the `--json`
plan array contains no row whose `target` mentions `.zshrc`.

Mutation check: without the `when` gate the entry would plan and materialize
its target unconditionally, so both the `!exists` assertion and the
plan-array assertion fail red.

### CHK-007 — A `when`-true entry materializes; the second run is byte-identical

`patina-cli/tests/conditional_entries.rs::when_true_entry_materializes_and_second_run_is_byte_identical`
uses a copy-mode `[[file]]` entry whose `when` equals
`patina.os == '<current OS family>'` (built from `std::env::consts::OS`,
which equals the `patina.os` built-in's value on the three supported
platforms). After a priming apply converges the repo, it asserts the target
is materialized and that two further consecutive `patina apply --yes` runs
emit byte-identical stdout (REQ-021 parity with a `when` present).

Copy mode is used deliberately: a symlink's plan diff renders its link
target differently on a fresh-vs-converged run — an orthogonal determinism
quirk the existing `deterministic_stdout.rs` suite also sidesteps by using
copy/template modes — so copy mode isolates the `when`-parity property under
test. The priming-then-measure shape mirrors `deterministic_stdout.rs`:
REQ-021 guards that stdout is a stable function of identical inputs.

Mutation check: without the gate evaluating true the entry would vanish and
the materialize assertion fails; a `when` evaluation that leaked a timestamp
or run-id into stdout would fail the byte-identical assertion.

### Multi-target `when` gates all targets together (REQ-003)

`multi_target_false_when_plans_none_of_its_targets` and
`multi_target_true_when_plans_all_of_its_targets` drive a copy-mode
multi-target `[[file]]` (`targets = ["~/.codex/agent.toml",
"~/.claude/agent.toml"]`). With a false `when` neither target is created;
with a true `when` both are. This pins the gate as per-entry (above the
target loop), not per-target.
