# Evidence for SPEC-0004 T-010

Red-then-green paper trail for T-010: `patina add` detects the source
kind from the tilde-expanded target's on-disk metadata and writes the
matching table-array — `[[file]]` for a file source,
`[[directory]]` for a directory source — with kind-checked mode flags
(`--symlink-tree` directory-only, `--template` file-only). Backed by a
minimal `[[directory]]` manifest-writer surface (`append_directory_entry`)
landed here because the full T-011 writer has not yet shipped.

branch in `cmd/add.rs` to the file path (`if false` in place of
`if mode.is_directory()`) so a directory source wrongly emitted a
`[[file]]` entry, then running the new test. With the real dispatch
restored, it passes.

## Session 2026-06-01-T010-attempt-1

### CHK-016 — file source writes `[[file]]` and no `[[directory]]`

Test: `add_file_writes_file_table_and_no_directory_table` in
`patina-cli/tests/add_directory.rs`.

<red>
Before this task `add` always called `append_file_entry`, so a file
source did produce a `[[file]]` entry — but the test also asserts the
*absence* of a `[[directory]]` key and a `mode = "symlink"` value on the
emitted entry, and the kind-aware routing that guarantees a file source
never falls into the directory branch is new. Run against the
file-forced experiment (see top), the assertion `parsed.get("directory")
.is_none()` would not exercise the routing; the genuine red is the
companion CHK-017 below, which proves the two branches are distinct.
</red>

<green>
`patina add ~/gitconfig --module m --symlink --yes` on a regular file
detects `SourceKind::File`, routes to `append_file_entry`, and the
manifest carries exactly one `[[file]]` entry with `mode = "symlink"`
and no `[[directory]]` key: `ok`.
</green>

### CHK-017 — directory source `--symlink-tree` writes a `[[directory]]` entry with `mode = "symlink-tree"`

Test: `add_directory_symlink_tree_writes_directory_table_with_mode` in
`patina-cli/tests/add_directory.rs`.

<red>
With the writer dispatch forced to the `[[file]]` branch (the prior
behaviour — `add` had no directory path at all), a directory source
emitted a `[[file]]` entry. The test's assertion that no `[[file]]` key
is present panics: `no [[file]] entry may be written for a directory
source`. `FAILED`.
</red>

<green>
With source-kind detection (`detect_source_kind` →
`SourceKind::Directory`), `AddMode::resolve(SymlinkTree, Directory)` →
`DirectorySymlinkTree`, and dispatch to `append_directory_entry`, the
manifest carries one `[[directory]]` entry with `source = "nvim"`,
`target = "~/nvim"`, `mode = "symlink-tree"`, and no `[[file]]` key. The
directory tree is also staged recursively into the repo. `ok`.
</green>
