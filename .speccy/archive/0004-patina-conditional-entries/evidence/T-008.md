## Session 2026-06-01-T008-attempt-1

### Scenario 1 — Deleted `symlink-tree` source leaf is reported orphaned (CHK-014)

Test: `deleted_symlink_tree_source_leaf_is_reported_orphaned` in
`patina-cli/tests/symlink_tree_orphans.rs`.

<red>
Against the pre-edit code, `status::current_plan_targets` iterated
only `config.files` and never expanded a `symlink-tree` `[[directory]]`
entry into its per-leaf target paths. So `~/d/sub/b.conf` (a recorded
leaf) was never in the managed set on *any* run — the test's
assertion that the *surviving* leaf `~/d/a.conf` classifies `clean`
failed, because every symlink-tree leaf was treated as unmanaged
(it would classify orphaned even while its source leaf was live). The
test panics at the `state_for(&doc, "/d/a.conf") == "clean"` assertion
(and `/d/sub/b.conf` classification is not distinguishable from the
surviving leaf): `FAILED`.
</red>

<green>
With `current_managed_targets` walking the live source for leaves, the
surviving `~/d/a.conf` is in the managed set (classifies `clean`) and
the deleted `~/d/sub/b.conf` is absent (classifies `orphaned`). The
test asserts `state_for(&doc, "/d/sub/b.conf") == "orphaned"` and
`state_for(&doc, "/d/a.conf") == "clean"`: `ok`.
</green>

### Scenario 2 — Next apply reaps the orphan leaf, keeps sibling and directory (CHK-015)

Test: `next_apply_reaps_orphan_leaf_and_keeps_sibling_and_directory`
in `patina-cli/tests/symlink_tree_orphans.rs`.

<red>
Against the pre-edit code, `execute` had no reap step: a target a
prior commit materialized but the current plan no longer managed was
left on disk forever (ORPHANED was a status-only classification, never
acted on). After deleting the `sub/b.conf` source leaf and re-applying,
`~/d/sub/b.conf` still existed, so the test's
`symlink_metadata(leaf_b).is_err()` assertion failed: `FAILED`.
</red>

<green>
With the apply-time `reap_orphans` step, the re-apply removes
`~/d/sub/b.conf` (absent from the current managed set), while leaving
`~/d/sub` a real directory (the `is_dir()` guard never removes a
directory, DEC-005) and `~/d/a.conf` a symbolic link (still managed).
All three assertions hold: `ok`.
</green>

### Scenario 3 — `when` flipped to false orphans then reaps the target with a backup (CHK-019)

Test: `when_flipped_to_false_orphans_then_reaps_target_with_backup`
in `patina-cli/tests/symlink_tree_orphans.rs`.

<red>
Against the pre-edit code, `current_plan_targets` did not evaluate an
entry's `when`, so a `[[file]]` entry whose `when` was edited to a
false predicate still counted its target as managed — `status`
classified `~/.gitconfig` `clean`, not `orphaned`. And with no reap,
the subsequent apply never removed it. The test's
`state_for(&doc, "/.gitconfig") == "orphaned"` assertion failed:
`FAILED`.
</red>

<green>
With `current_managed_targets` evaluating `when` through
`Engine::eval_when` and dropping `when`-false entries, the flipped
target falls out of the managed set: `status` classifies it
`orphaned`. The next apply reaps it (target removed) after stashing its
prior bytes into the run's backup tree — proven by finding the
original `[user]\n  name = me\n` bytes in the backup tree via
`find_backup_with_bytes`. All assertions hold: `ok`.
</green>
